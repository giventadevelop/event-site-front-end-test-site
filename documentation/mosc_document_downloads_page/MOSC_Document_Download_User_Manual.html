Overview

One-off tooling migrates official documents from the legacy site (mosc.in) into this app’s official document library (categories, year bundles, bulk upload). Scripts run outside the Next.js browser.

Bulk load in the admin UI is the upload feature this ties to — not a separate “World Cup” feature. Flow: download with script → local folder → bulk upload at /admin/official-documents.

Legal, copyright & ethics

  1. You must have the right to copy and republish each file.
  2. Comply with mosc.in terms of service.
  3. robots.txt: The downloader respects Disallow for User-agent: *. Use --ignore-robots only for local debugging.
  4. Rate limits: Default delayMs 1500 ms between requests; increase if needed.
  5. No crawling: Only explicit URLs in your manifest — no site spidering.
📁 Download folder (default)
Setting Value
Default root C:\E_Drive\code_backup\mosc_downloads
Override Environment variable MOSC_DOWNLOAD_ROOT

Layout on disk:

<MOSC_DOWNLOAD_ROOT>/<categorySlug>/<year>/<filename>

Example: C:\E_Drive\code_backup\mosc_downloads\synod-minutes\2024\minutes-jan.pdf — matches how you select files for one category + year in bulk upload.

⚙️ Manifest (url-list.json) fields

Per asset (in items array):

  • url — full HTTPS URL to the file
  • categorySlug — must match /admin/official-document-categories
  • year — calendar year
  • filename — optional rename

Optional top-level: delayMs, userAgent, siteOrigin (for robots.txt), fetchRobotsTxt.

🗂 Title-folder mirror mode

Use mirror-downloads-tree.mjs when you want the local download folders to follow the site’s title structure (main title -> sub-title -> files), instead of only <categorySlug>/<year>.

Example source pages:

  • https://mosc.in/downloads/
  • https://mosc.in/downloads/form-for-preparing-financial-statements-by-institutions-for-the-year-ended/

Output root:

C:\E_Drive\code_backup\mosc_downloads\mosc-in-downloads-by-title\<top-title>\<sub-title>\<file-label>\<filename>
This mode is best when operators want a human-readable “website-like” folder mirror for manual review and selective upload.
🛠 Step-by-step workflow
  1. Copy url-list.example.json → url-list.json and list permitted URLs.
  2. Run a dry run (command catalog above); confirm output paths.
  3. Run the fetch command; check _robots_snapshot.txt and _fetch-report-*.json under the download root.
  4. If you need title/sub-title folder hierarchy, run mirror mode dry run then mirror download.
  5. Proceed to bulk upload in the app, or run auto-upload script for mirrored folders.
📤 Bulk load — admin UI

Backend route: POST /api/proxy/event-medias/upload/bulk-tenant-official

  1. Open /admin/official-documents as tenant admin.
  2. Choose category (slug = folder under downloads).
  3. Set year to match the subfolder.
  4. Select files from ...\mosc_downloads\<categorySlug>\<year>\
  5. Submit bulk upload.
Tenant: NEXT_PUBLIC_TENANT_ID must match your environment. Create categories first at /admin/official-document-categories if needed.
🔐 Upload authorization prerequisite (backend)

If real mirror upload fails with Admin access required, this is a backend authorization issue (not a frontend/script issue).

Required backend behavior:
  • Service JWT used by migration scripts must resolve to admin authorities (ROLE_ADMIN at minimum).
  • jwtadmin (or service subject) must map to admin authorities in backend JWT-to-authority conversion.

Recommended backend fix order

  1. Preferred: issue service JWT with role claim containing ROLE_ADMIN (or authority expected by requireAdmin()).
  2. Fallback workaround: in ClerkJwtAuthenticationFilter (or equivalent), if subject is jwtadmin, inject ROLE_ADMIN authority.

Backend ticket prompt (copy/paste)

We need a backend auth fix for MOSC migration uploads.

Context:
- Upload endpoint: POST /api/event-medias/upload/tenant-official-document
- Failure: 400 "Admin access required" from EventMediaResource.requireAdmin()
- Service auth user: jwtadmin
- Current authentication succeeds, but Granted Authorities is empty.

Please implement:
1) Preferred fix:
   - Ensure service JWT for jwtadmin contains admin role/authority claim
     (e.g., authorities or roles includes ROLE_ADMIN), and
   - Backend JWT-to-GrantedAuthority mapping consumes this claim.

2) Safe fallback workaround:
   - In ClerkJwtAuthenticationFilter (or equivalent JWT auth converter),
     if subject == "jwtadmin", add SimpleGrantedAuthority("ROLE_ADMIN")
     before creating Authentication.

Acceptance criteria:
- jwtadmin can upload via /api/event-medias/upload/tenant-official-document
  without "Admin access required".
- Security logs show ROLE_ADMIN in Granted Authorities for jwtadmin.
- Non-jwtadmin users are unchanged.
- Add unit/integration test for authority mapping.

Troubleshooting

Issue What to do
SKIP (robots) Path blocked by robots — obtain file legally; do not use --ignore-robots in production.
HTTP 403 / 401 Server may block bots — manual download or permission from owner.
Wrong category in app Align categorySlug with tenant categories.
Upload fails Check JWT env (API_JWT_USER / API_JWT_PASS) and proxy logs.
