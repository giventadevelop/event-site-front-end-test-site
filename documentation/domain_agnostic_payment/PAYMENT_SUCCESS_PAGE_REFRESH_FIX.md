# Payment Success Page Refresh and QR Code Display Fix

## Problem Statement

The payment success page (`/event/success`) is experiencing critical issues:

1. **Infinite Refresh Loop**: The page refreshes continuously (500+ times) after payment succeeds, preventing users from viewing their tickets and QR codes.

2. **QR Code Not Displaying**: Even though the backend successfully generates QR codes and sends emails, the frontend fails to display the QR code automatically, requiring manual page refreshes.

3. **Ticket Transaction Lookup Failure**: The frontend cannot reliably find the `EventTicketTransaction` record after payment succeeds, causing the QR code fetch to never trigger.

## Root Causes

### 1. Incorrect Ticket Query Method
- **Problem**: Frontend queries for ticket transactions using `transactionReference.equals={paymentTransactionId}` (e.g., `transactionReference.equals=5711`)
- **Reality**: The ticket transaction's `transaction_reference` field is auto-generated by database triggers in the format `TKTN{id}` (e.g., `TKTN5754`), which doesn't match the payment transaction ID
- **Result**: Query returns empty array, ticket transaction is never found, QR code fetch never triggers

### 2. Refresh Logic Using `useRef`
- **Problem**: Refresh state tracked using `useRef`, which resets to `false` on every page reload
- **Reality**: After refresh, component remounts, `refreshTriggeredRef.current` resets to `false`, refresh condition is met again → infinite loop
- **Result**: Page refreshes infinitely, never stabilizing

### 3. Missing Ticket Transaction Polling
- **Problem**: If ticket transaction isn't found immediately, frontend gives up
- **Reality**: Backend processes ticket generation asynchronously, ticket may not be available immediately after payment succeeds
- **Result**: QR code never appears without manual refresh

## Solution Approach

### 1. Fix Ticket Query Method

**Change**: Query by `stripePaymentIntentId` instead of `transactionReference`

**Why**:
- The `EventTicketTransaction` has a `stripe_payment_intent_id` field that matches the payment transaction's Stripe payment intent ID
- This is a reliable, direct relationship between payment and ticket
- Backend logs confirm tickets are created with `stripe_payment_intent_id` populated

**Implementation**:
```typescript
// Extract Stripe payment intent ID from payment transaction
const stripePaymentIntentId = paymentData.metadata?.stripePaymentIntentId ||
  (paymentData.paymentReference?.startsWith('pi_') ? paymentData.paymentReference : null);

// Query ticket by stripePaymentIntentId
const searchParams = new URLSearchParams();
searchParams.append('stripePaymentIntentId.equals', stripePaymentIntentId);
if (eventId) {
  searchParams.append('eventId.equals', eventId.toString());
}

const ticketRes = await fetch(
  `${baseUrl}/api/proxy/event-ticket-transactions?${searchParams.toString()}`,
  { cache: 'no-store' }
);
```

### 2. Use `sessionStorage` for Refresh State

**Change**: Replace `useRef` with `sessionStorage` to persist refresh state across page reloads

**Why**:
- `sessionStorage` persists across page reloads within the same browser session
- Prevents infinite refresh loops by remembering that refresh has already occurred
- Automatically clears when browser tab/window is closed

**Implementation**:
```typescript
// Use sessionStorage to prevent infinite refresh loops
const getRefreshKey = () => `payment-refresh-${transactionId}`;
const hasRefreshed = () => {
  if (typeof window === 'undefined') return false;
  return sessionStorage.getItem(getRefreshKey()) === 'true';
};
const markAsRefreshed = () => {
  if (typeof window !== 'undefined') {
    sessionStorage.setItem(getRefreshKey(), 'true');
  }
};

// In refresh useEffect:
if (hasRefreshed()) {
  console.log('Already refreshed for this transaction, skipping refresh logic');
  return;
}

// When ready to refresh:
markAsRefreshed(); // Mark BEFORE reloading
setTimeout(() => {
  window.location.reload();
}, 500);
```

### 3. Add Ticket Transaction Polling

**Change**: Poll for ticket transaction if not found immediately after payment succeeds

**Why**:
- Backend processes ticket generation asynchronously
- Ticket may not be available immediately after payment succeeds
- Polling ensures we find the ticket even if backend processing is delayed

**Implementation**:
```typescript
// Poll for ticket transaction if payment succeeded but ticket not found
useEffect(() => {
  const normalizedStatus = paymentTransaction?.status?.toUpperCase();

  if (
    normalizedStatus === 'SUCCEEDED' &&
    paymentTransaction &&
    !ticketTransaction &&
    !hasRefreshed()
  ) {
    const stripePaymentIntentId = paymentTransaction.metadata?.stripePaymentIntentId ||
      (paymentTransaction.paymentReference?.startsWith('pi_') ? paymentTransaction.paymentReference : null);

    if (!stripePaymentIntentId) return;

    // Poll every 2 seconds for up to 30 seconds
    async function pollForTicket() {
      // Query by stripePaymentIntentId
      const ticketRes = await fetch(
        `${baseUrl}/api/proxy/event-ticket-transactions?stripePaymentIntentId.equals=${stripePaymentIntentId}`,
        { cache: 'no-store' }
      );

      if (ticketRes.ok) {
        const ticketData = await ticketRes.json();
        if (Array.isArray(ticketData) && ticketData.length > 0) {
          setTicketTransaction(ticketData[0]);
          // Stop polling once found
        }
      }
    }

    pollForTicket(); // Immediate check
    const interval = setInterval(pollForTicket, 2000); // Then poll every 2 seconds
  }
}, [paymentTransaction?.status, ticketTransaction]);
```

### 4. Refresh Only After All Data is Ready

**Change**: Only refresh when payment succeeded, ticket transaction found, AND QR code fetched

**Why**:
- Ensures all data is available before refresh
- Prevents premature refreshes that would require another refresh cycle
- Single refresh after everything is ready

**Implementation**:
```typescript
// Refresh page ONCE after ticket transaction AND QR code are both ready
useEffect(() => {
  if (hasRefreshed()) return; // Skip if already refreshed

  const normalizedStatus = paymentTransaction?.status?.toUpperCase();
  if (
    normalizedStatus === 'SUCCEEDED' &&
    paymentTransaction &&
    ticketTransaction &&
    qrCodeData &&
    qrCodeData.qrCodeImageUrl // Ensure QR code URL is actually present
  ) {
    console.log('All data ready, refreshing page ONCE...');
    markAsRefreshed(); // Mark BEFORE reloading
    setTimeout(() => {
      window.location.reload();
    }, 500);
  }
}, [paymentTransaction?.status, ticketTransaction, qrCodeData, transactionId]);
```

## Legacy Workflow Reference

### How Legacy Workflow Achieved This

The legacy workflow (before recent changes) handled ticket and QR code display differently:

#### 1. **Separate API Calls**
- Frontend made separate API calls for:
  - Ticket transaction fetch
  - QR code generation (`/api/proxy/events/{eventId}/transactions/{transactionId}/emailHostUrlPrefix/{prefix}/qrcode`)
  - Email sending (`sendTicketEmailAsync`)

#### 2. **No Automatic Refresh**
- Legacy workflow did NOT automatically refresh the page
- Instead, it used React state updates and `useEffect` hooks to trigger re-renders when data became available
- Component would re-render when:
  - `ticketTransaction` state changed
  - `qrCodeData` state changed
  - This caused automatic UI updates without page refresh

#### 3. **Direct Ticket ID Lookup**
- Legacy workflow relied on:
  - Payment status response including `ticketTransactionId` directly
  - Or querying by known identifiers (payment reference, transaction reference)
  - No polling mechanism - if ticket wasn't found, it stayed in loading state

#### 4. **State-Based Rendering**
```typescript
// Legacy approach - no refresh, just state updates
useEffect(() => {
  if (ticketTransaction?.id && eventDetails?.id && !qrCodeData) {
    // Fetch QR code
    fetchQrCode();
  }
}, [ticketTransaction?.id, eventDetails?.id, qrCodeData]);

// Component automatically re-renders when qrCodeData state changes
{qrCodeData && (
  <img src={qrCodeData.qrCodeImageUrl} alt="QR Code" />
)}
```

### Why Current Approach Uses Refresh

The current approach uses a single refresh after all data is ready because:

1. **Backend Processing Delay**: Backend processes ticket generation asynchronously, ticket may not be immediately available
2. **Data Consistency**: Refresh ensures all data is fetched fresh from backend in a single request cycle
3. **Simpler State Management**: Avoids complex state synchronization between multiple async operations
4. **User Experience**: Single refresh after everything is ready is less jarring than multiple state updates

### Hybrid Approach (Current Implementation)

The current implementation combines both approaches:

1. **State-Based Updates**: Uses React state and `useEffect` to fetch ticket and QR code automatically
2. **Polling**: Polls for ticket transaction if not found immediately
3. **Single Refresh**: Refreshes once after all data is ready to ensure consistency
4. **SessionStorage Guard**: Prevents infinite refresh loops using `sessionStorage`

## Backend Requirements

For this solution to work correctly, the backend must:

1. **Include `stripePaymentIntentId` in Payment Status Response**
   - The `PaymentStatusResponse` should include `stripePaymentIntentId` field
   - Or include it in the `metadata` object
   - This allows frontend to query ticket transactions reliably

2. **Populate `stripe_payment_intent_id` in EventTicketTransaction**
   - When creating `EventTicketTransaction`, ensure `stripe_payment_intent_id` is set
   - This field should match the `UserPaymentTransaction.stripe_payment_intent_id`
   - This is already implemented based on backend logs

3. **Support Query by `stripePaymentIntentId`**
   - The `/api/event-ticket-transactions` endpoint should support filtering by `stripePaymentIntentId.equals`
   - This is likely already supported via Spring Data REST criteria queries

## Expected Flow

1. User completes payment → redirected to `/event/success?transactionId={id}&eventId={id}`
2. Frontend fetches payment status → status is `SUCCEEDED`
3. Frontend queries ticket transaction by `stripePaymentIntentId` → ticket found (or polling finds it)
4. Frontend fetches QR code using ticket transaction ID → QR code URL received
5. Frontend refreshes page ONCE → all data displays correctly
6. No further refreshes occur → user can view ticket and QR code
