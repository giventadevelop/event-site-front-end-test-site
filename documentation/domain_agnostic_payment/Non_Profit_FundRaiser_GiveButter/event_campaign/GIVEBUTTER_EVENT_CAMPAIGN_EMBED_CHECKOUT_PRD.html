GiveButter Event Campaign Embed Checkout PRD
Event Hero + Embedded Widget

1. Overview

This PRD defines the event campaign embed checkout: a new page that shows the event hero at the top and an embedded GiveButter widget in the center. It is distinct from:

  • Generic donate button: Site-wide or non-event GiveButter campaigns.
  • Donation-checkout: Existing /events/[id]/donation-checkout page that opens GiveButter in a popup or new tab (no embed).

The new page uses the same event-scoped URL pattern and eligibility rules as donation-checkout for ticketed fundraiser/charity events, but renders the GiveButter widget inline (no Stripe, no mixed flow).

✅ X-Frame-Options: GiveButter sets X-Frame-Options: sameorigin, so the full GiveButter page cannot be iframe'd. Only the widget (script + custom element <givebutter-widget> or <givebutter-form>) can be embedded on our site. This page is our layout (hero + center area) with the GiveButter widget embedded.

2. User Flow

Events listing or event detail (hero) → user clicks “Buy ticket” / fundraiser CTA → /events/[id]/givebutter-checkout → event hero at top, GiveButter widget in center. User completes donation/ticket purchase in the embedded widget.

3. Checkout URL Format

Pattern: /events/[id]/givebutter-checkout

Example: http://localhost:3000/events/4202/givebutter-checkout

No query parameters required; the event id from the path is sufficient. This keeps parity with /events/[id]/donation-checkout and makes the page clearly event-specific.

4. Data Model

Use existing event_details.donation_metadata (TEXT, JSON). No new database columns for the initial implementation.

  • givebutterCampaignId (already present): e.g. FwSx70 from the event ticketing URL.
  • givebutterWidgetId (optional, new): e.g. j1ek6j for <givebutter-widget id="j1ek6j">.

When rendering the embed page: if givebutterWidgetId is set, render <GivebutterWidget widgetId="..." />; else if givebutterCampaignId is set, render <GivebutterWidget campaignId="..." />. Backend must accept and return givebutterWidgetId inside the same JSON.

5. Page Layout and Behavior

  • Route: src/app/events/[id]/givebutter-checkout/ (page.tsx + server data + client component).
  • Top: Event hero (reuse same hero resolution as donation-checkout, e.g. from DonationCheckoutServerData / event media).
  • Center: GiveButter widget only (no Stripe, no donation-checkout form).

Data loading: Server fetches event by id, resolves hero image, reads donation_metadata via getDonationMetadata. Pass to client: heroImageUrl, eventId, givebutterCampaignId, givebutterWidgetId.

Widget rendering (client): If givebutterWidgetId present → <GivebutterWidget widgetId={givebutterWidgetId} />. Else if givebutterCampaignId present → <GivebutterWidget campaignId={givebutterCampaignId} />. Else show message that GiveButter is not configured and link back to event or donation-checkout.

Access control: Only allow for events that are ticketed + fundraiser/charity with GiveButter. Reuse isTicketedFundraiserEvent and donation metadata checks; 404 or friendly error otherwise.

6. Frontend Implementation

  • New route: src/app/events/[id]/givebutter-checkout/ with server data layer, page component, and client component (hero + GivebutterWidget).
  • Use existing GivebutterWidget (src/components/GivebutterWidget.tsx) with either widgetId or campaignId from donation_metadata.
  • EventForm: add optional “GiveButter event widget ID” (and optionally “GiveButter event campaign slug” to override). Persist in the same donation_metadata payload sent to the backend.

7. Backend

Ensure event_details API returns donationMetadata (or equivalent) with the full JSON. When saving event (admin), accept and persist givebutterWidgetId inside that JSON. No new endpoints required for the embed page; no change to event_details table for Option A (JSON-only).

8. Links to Update

For ticketed GiveButter events, point “Buy ticket” / fundraiser CTAs to /events/[id]/givebutter-checkout. Places to update:

  • src/components/HeroSection.tsx (e.g. href for ticketed fundraiser).
  • src/app/events/page.tsx (links for fundraiser cards).
  • src/app/events/[id]/page.tsx (link for ticketed fundraiser).
  • src/components/UpcomingEventsSection.tsx.

Decision: either replace donation-checkout links with givebutter-checkout for these events, or add a small config (e.g. in donation_metadata) to choose “embed” vs “popup” and set links accordingly.
