Official Document Upload Refactor - Backend

This document defines backend changes for explicit hierarchy storage and priority-first listing.

1) Database schema changes

Apply SQL migration:

documentation/mosc_document_downloads_page/upload_refactor/migration_official_document_hierarchy_fields.sql
Column Type Purpose
hierarchy_path TEXT Canonical tree path (for hierarchy traversal)
hierarchy_category_label TEXT Top-level visual category/folder label
display_priority INTEGER Explicit display priority for public downloads pagination

2) Upload endpoints - request contract updates

Extend upload and patch flows to accept and persist:

  • hierarchyPath -> event_media.hierarchy_path
  • hierarchyCategoryLabel -> event_media.hierarchy_category_label
  • displayPriority -> event_media.display_priority
Keep backward compatibility: continue accepting old description markers while new fields are gradually populated.

3) Listing endpoints - ordering and filters

For public downloads listing API, use this order:

ORDER BY COALESCE(display_priority, priority_ranking, 999999) ASC, created_at DESC

Recommended criteria for public listing:

  • tenant_id = :tenantId
  • is_event_management_official_document = true
  • is_public = true
  • Optional: official_document_category_id, official_document_year

4) Backfill strategy

The SQL migration includes optional backfill from legacy markers:

  • [[MOSC_TREE_PATH]]
  • [[MOSC_CATEGORY_LABEL]]
  • [[MOSC_PRIORITY]]

Result: existing rows continue to work, new uploads write explicit structured fields.

5) API examples (patch)

{
  "id": 12345,

  "hierarchyCategoryLabel": "local body Election winners award ceremony – Photos",

6) Validation rules

  • hierarchyPath must not contain traversal sequences (..)
  • displayPriority must be non-negative integer
  • Prefer storing path separator as \ for consistency with migration scripts

7) Upload authorization prerequisite (jwtadmin)

Real migration upload uses server-side service authentication. If upload fails with Admin access required, fix backend authority mapping for the service subject.

This is a backend fix. Frontend and migration scripts are already calling the correct upload endpoint.

Required behavior

  • Service JWT for jwtadmin must resolve to ROLE_ADMIN authority.
  • requireAdmin() checks must pass for official-document upload endpoint.

Implementation options

  1. Preferred: include admin role claim in service JWT (roles/authorities contains ROLE_ADMIN) and map claim to GrantedAuthority.
  2. Workaround: in ClerkJwtAuthenticationFilter (or equivalent), if sub == "jwtadmin", inject ROLE_ADMIN authority before building Authentication.
