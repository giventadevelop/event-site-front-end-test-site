GiveButter Donation Flow - Frontend Implementation PRD
Separate Flow for Non-Profit Fundraiser Events

Critical Requirements

  • Separate Flow: Create completely separate components and routes for GiveButter donation flow. DO NOT modify existing checkout flow.
  • Reference Only: Use /events/[id]/checkout as reference for transaction creation, QR code generation, and email sending patterns only.
  • Event Detection: Use existing event edit page fields (isFundraiserEvent, isCharityEvent, donationMetadata) to determine if event is donation-based.
  • Predefined Buttons: Support predefined donation amount buttons ($5, $10, $25, $50, $100, custom) for church offerings and Mass collections.
  • Data Persistence: All donations must be persisted to database for admin dashboard statistics.

1. Executive Summary

This PRD outlines the frontend implementation of a GiveButter-based donation flow for non-profit fundraiser events. The implementation creates a completely separate flow from the existing ticket checkout, while maintaining similar patterns for transaction creation, QR code generation, and email sending.

Key Objectives

  • Enable zero-fee donations for non-profit fundraiser events using GiveButter API
  • Support ticketed fundraiser events with donation-based payment routing
  • Support Mass offerings with predefined donation buttons and prayer intentions
  • Maintain separate flow from existing Stripe checkout (reference only)
  • Persist all donation data for admin dashboard analytics
  • Generate QR codes and send confirmation emails similar to ticket purchase flow

2. Event Configuration Detection

2.1 Event Edit Page Fields

The following fields in the event edit page (src/components/EventForm.tsx) determine if an event is donation-based:

Field Location Purpose Values
isFundraiserEvent donationMetadata.isFundraiserEvent Marks event as fundraiser true / false
isCharityEvent donationMetadata.isCharityEvent Marks event as charity true / false
zeroFeeProvider donationMetadata.zeroFeeProvider Payment provider for zero-fee processing "GIVEBUTTER"
givebutterCampaignId donationMetadata.givebutterCampaignId GiveButter campaign ID (optional) String or null
admissionType event_details.admission_type Event admission type "DONATION_BASED" or "TICKETED"

2.2 Detection Logic

// Event is donation-based if: // 1. admissionType === 'DONATION_BASED' OR // 2. (isFundraiserEvent === true OR isCharityEvent === true) AND zeroFeeProvider === 'GIVEBUTTER' function isDonationBasedEvent(event: EventDetailsDTO): boolean { // Check admission type if (event.admissionType === 'DONATION_BASED') { return true; } // Check fundraiser/charity configuration if (event.donationMetadata) { const donationMeta = JSON.parse(event.donationMetadata); const isFundraiser = donationMeta.isFundraiserEvent === true; const isCharity = donationMeta.isCharityEvent === true; const provider = donationMeta.zeroFeeProvider; if ((isFundraiser || isCharity) && provider === 'GIVEBUTTER') { return true; } } return false; }

4. Donation Checkout Flow

4.1 User Flow

  1. User visits event page (/events/[id])
  2. If event is donation-based, show "Make a Donation" button instead of "Buy Tickets"
  3. User clicks "Make a Donation" → redirects to /events/[id]/donation
  4. User selects donation amount (preset buttons or custom input)
  5. User enters email, name, phone (optional)
  6. User clicks "Donate" → calls POST /api/proxy/payments/initialize with GiveButter payment type
  7. Backend returns GiveButter checkout URL → redirect to GiveButter
  8. User completes payment on GiveButter
  9. GiveButter redirects to /events/[id]/donation/success?transactionId=xxx
  10. Success page displays QR code (if applicable) and sends confirmation email

6. Donation Success Page (Frontend Polling Pattern)

CRITICAL: Following the existing checkout flow pattern, the frontend handles all webhook-like behavior through polling. No backend webhook listener is needed.

6.1 Success Page Flow

  1. User completes payment on GiveButter
  2. GiveButter redirects to /events/[id]/donation/success?transactionId=xxx or ?donationId=xxx
  3. Success page polls donation status (similar to Stripe polling pattern)
  4. Frontend calls GiveButter API directly (or via backend status endpoint) to check donation status
  5. When status becomes "COMPLETED", frontend calls backend APIs to persist data
  6. Display donation confirmation with QR code (if applicable)
  7. Send confirmation email via backend API call
  8. Show donation receipt details

7. Ticketed Fundraiser Events

7.1 Flow for Ticketed Fundraisers

For events that are both ticketed AND fundraiser/charity events:

  1. User selects tickets on event page (same as regular ticket flow)
  2. User clicks "Checkout" → backend detects isFundraiserEvent: true and zeroFeeProvider: "GIVEBUTTER"
  3. Backend routes payment to GiveButter instead of Stripe
  4. User completes payment on GiveButter
  5. Success flow is same as regular ticket purchase (QR code, email, etc.)
Important: For ticketed fundraiser events, the existing checkout flow (/events/[id]/checkout) should be used, but the backend will automatically route to GiveButter based on event configuration. No frontend changes needed for ticketed fundraisers.

8. Frontend Polling Pattern (No Backend Webhooks)

Architecture Decision: Frontend Polling Instead of Backend Webhooks

Following the existing checkout flow pattern (/events/[id]/checkout), the frontend handles all webhook-like behavior through polling. This approach:

  • Eliminates the need for backend webhook listeners
  • Provides immediate feedback to users
  • Matches the existing Stripe payment flow pattern
  • Allows frontend to control transaction creation timing

8.1 Polling Flow Pattern

// Pattern: Similar to Stripe payment polling in PaymentSuccessClient.tsx 1. User completes payment on GiveButter 2. GiveButter redirects to success page with transactionId or donationId 3. Frontend starts polling donation status: - Option A: Poll backend status endpoint: GET /api/proxy/payments/{transactionId} - Option B: Poll GiveButter status: GET /api/proxy/givebutter/donations/{donationId}/status 4. When status becomes "COMPLETED" or "SUCCEEDED": - Frontend calls backend to create/update donation transaction - Frontend calls backend to generate QR code - Frontend calls backend to send confirmation email 5. Display success page with QR code and confirmation

16. Key Differences from Webhook-Based Approach

Frontend Polling vs Backend Webhooks

Aspect Webhook-Based (NOT Used) Frontend Polling (Used)
Status Detection Backend receives webhook events from GiveButter Frontend polls backend status endpoint or GiveButter API
Transaction Creation Backend webhook handler creates transaction Frontend calls backend API to create transaction after detecting success
QR Code Generation Backend webhook handler generates QR code Frontend calls backend API to generate QR code
Email Sending Backend webhook handler sends email Frontend calls backend API to send email
User Experience User waits for webhook to process (may be delayed) User sees immediate polling feedback (better UX)
Pattern Reference N/A (not used) /events/[id]/checkout flow (existing pattern)
