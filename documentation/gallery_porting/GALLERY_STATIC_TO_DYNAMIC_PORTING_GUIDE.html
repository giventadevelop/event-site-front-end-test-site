Gallery Static → Dynamic Porting Implemented

Bulk-import all MOSC static gallery albums and photos into the dynamic gallery_album + event_media database using existing REST APIs — no backend code changes required.

Static gallery (unchanged): http://localhost:3000/mosc-redesign/gallery

Dynamic gallery (API-driven): http://localhost:3000/gallery

Header submenu: Gallery → Gallery (static) + Gallery Dynamic (/gallery)

Scope: 29 albums, ~315 photos parsed from src/lib/gallery/moscStaticAlbums.ts and per-album page.tsx files.

2. Prerequisites

Required environment variables (in .env.local):

  • NEXT_PUBLIC_API_BASE_URL — Rust/Spring backend base URL
  • NEXT_PUBLIC_TENANT_ID — tenant for all DTOs
  • API_JWT_USER / API_JWT_PASS — service JWT credentials (same as other admin scripts)

Optional (file upload mode only):

  • MOSC_APP_BASE_URL or NEXT_PUBLIC_APP_URL — e.g. http://localhost:3000
  • Next.js dev server running for multipart proxy uploads
  • Image files present under public/images/mosc/gallery/{slug}/
Gallery categories are required. Albums need a gallery_category_id foreign key. If gallery_category is empty (common after a fresh schema rebuild), imports succeed but every album gets galleryCategoryId: null. See Section 4 — Gallery categories for the fix.
Cover images on S3: Default import uses local paths (/images/mosc/gallery/...) for cover_image_url. For production S3 covers and admin upload UX, see Gallery Album Cover S3 Upload PRD.
Event dates & location: Legacy cards show full date lines (e.g. October 24–26, 2015, Indore); the API today stores only album_year. Structured fields eventDateStart, eventDateEnd, eventLocation require schema + backend work first — see Event date & location PRD pack and Section 4.5.

4. Gallery categories — problem and fix

Symptom: After import, SQL shows gallery_category empty and gallery_album.gallery_category_id is NULL for every row. The dynamic gallery may hide category pills/filters.

Root cause

  • The import script looks up categories by slug; it does not create the category table rows unless the updated import path runs ensureGalleryCategories() (current npm run gallery:import).
  • A fresh schema creates the gallery_category table but does not insert the 8 seed rows unless you run SQL migration 002_seed_gallery_category.sql or the API seed script below.
  • Console during import: No category id for slug … → album created with galleryCategoryId: null. album_year and photos still import correctly.

Fix option A — API seed + backfill (recommended)

Script: scripts/gallery-porting/seed-gallery-categories-and-backfill-albums.mjs

  1. Step 1: POST eight gallery_category rows via /api/gallery-categories (skips slugs that already exist).
  2. Step 2: PATCH each imported album matched by static_slug= in description, using category mapping from static-gallery-manifest.json.
Seed categories + backfill albums
npm run gallery:seed-categories
Dry run (preview only)
npm run gallery:seed-categories:dry

Report written to scripts/gallery-porting/category-backfill-report.json.

Fix option B — SQL migrations

Apply in order (PostgreSQL):

  • documentation/gallery_album_category_year_enhancements/migrations/001_add_gallery_category_and_album_year.sql — schema (if not already in main schema file)
  • documentation/gallery_album_category_year_enhancements/migrations/002_seed_gallery_category.sql — insert 8 categories for tenant_demo_002
  • documentation/gallery_album_category_year_enhancements/migrations/004_backfill_gallery_album_category_year.sql — set gallery_category_id from static slug mapping

5. Step-by-step workflow

Step 1 — Extract manifest

Parses static TS sources into scripts/gallery-porting/static-gallery-manifest.json.

Extract manifest
npm run gallery:extract-manifest

Equivalent:

Direct node
node scripts/gallery-porting/extract-static-gallery-manifest.mjs

Step 2 — Dry run (validate auth + payload)

Full dry run
npm run gallery:import:dry
Single album test
node scripts/gallery-porting/import-static-gallery-to-api.mjs --dry-run --slug russia-visit

Step 3 — (Optional) Seed categories before import

Use this on a fresh DB if you prefer categories to exist before any album POST. Otherwise skip — npm run gallery:import calls ensureGalleryCategories() automatically.

Pre-import category seed
npm run gallery:seed-categories

Step 4 — Import albums + photos (URL mode, default)

Creates gallery_album rows and event_media rows with fileUrl pointing at existing static paths (e.g. /images/mosc/gallery/russia-visit/1.jpg). No file upload required if images are served from the Next app or CDN. Seeds categories first if the table is empty; patches category on existing albums when re-run and category was missing.

Full import
npm run gallery:import

Step 5 — Backfill categories (if albums imported with NULL category)

If you imported before categories existed, run the seed/backfill script once. Safe to re-run (idempotent for existing categories).

Backfill gallery_category_id
npm run gallery:seed-categories

Step 6 — Verify

  • Open http://localhost:3000/gallery — albums should appear.
  • Admin: /admin/gallery/albums
  • Review scripts/gallery-porting/import-report.json for per-album ids and error counts.

8. How albums are matched (idempotency)

  • Each imported album description includes static_slug={slug} for re-runs.
  • Default behavior skips albums/media that already exist (use --force to override).
  • --media-only looks up existing albums by parsing static_slug= from description.
