Homepage Cache Admin Refresh – Implementation Plan

This document is the implementation plan for an admin page that manually refreshes the homepage cache per tenant. It uses the cache-busting version parameter approach described in the Homepage Edge Cache Guideline (Option 1: no CloudFront API required). The plan covers frontend (Next.js admin page), backend (REST endpoint or PATCH extension), database (version storage), optional batch job (cache warming), and API contract. No code or config changes are made by this document—it is documentation only for developers to implement the feature.

Overview

Goal and approach

Goal: Allow an admin to trigger a “refresh” of the homepage cache for a specific tenant so that the next request for the homepage payload uses a new cache key and fetches fresh content from the origin (cache-busting).

Approach: Store a version per tenant (e.g. in tenant_settings or a dedicated table). When the admin clicks “Refresh homepage cache” for a tenant, the backend bumps that version. The homepage-payload URL includes the version (e.g. ?tenantId=X&v=...). Clients use the new version in the next request, so the CDN treats it as a new resource and serves fresh content. This matches Option 1: Cache-busting with a version parameter in HOMEPAGE_EDGE_CACHE_GUIDELINE.html.

Frontend

Admin page and components

Use layout max-w-5xl mx-auto px-8 py-8 and content card bg-white rounded-lg shadow-md p-6 per .cursor/rules/ui_style_guide.mdc.
  • Client component: Table or list of tenants with a “Refresh cache” button per row. On click: call a server action (e.g. refreshHomepageCacheServer(tenantId)); show loading state; on success show a success message (and optionally the new version). Use admin action button styling per .cursor/rules/admin_action_buttons_styling.mdc or .cursor/rules/icons_buttons_styles.mdc (e.g. w-full flex-shrink-0 h-14 rounded-xl bg-blue-100 hover:bg-blue-200 with icon and label).

  • Proxy (if backend adds a new path): If the backend adds POST /api/homepage-cache/refresh, add a proxy route e.g. src/pages/api/proxy/homepage-cache/[...slug].ts using createProxyHandler from src/lib/proxyHandler.ts. The frontend server action then calls /api/proxy/homepage-cache/refresh (or the proxy path that forwards to the backend).

Backend

Backend repo and endpoints

Repo: E:\project_workspace\malayalees-us-site-boot.

Option 1 – Extend tenant-settings PATCH: Add or extend PATCH for tenant-settings to support homepageCacheVersion (or homepage_cache_version per API schema). The frontend calls the existing tenant-settings proxy with a PATCH body that includes the bumped version; the backend persists and returns the updated resource. Reference documentation/Swagger_API_Docs/api-docs.json for the current tenant-settings path and schema.

Option 2 – New endpoint: Add e.g. POST /api/homepage-cache/refresh with body { "tenantId": "..." }. Backend validates that the caller is an admin, loads tenant_settings (or homepage_cache_meta) for that tenant, increments or sets the version, saves, and returns 200 with the new version or 204.

Auth and tenant isolation: Admin-only (same as tenant-settings update). Only allow refresh for tenants the user is allowed to manage; enforce tenant isolation so one tenant cannot bump another tenant’s version.

Frontend integration: The frontend will call the backend via a proxy (e.g. /api/proxy/tenant-settings/{id} for PATCH or /api/proxy/homepage-cache/refresh for POST). All calls must be made from server actions using fetchWithJwtRetry, not from the client.

Database

Schema changes

Option A (recommended for simplicity): Add a column to the existing tenant_settings table:

  • Column: homepage_cache_version bigint DEFAULT 0 NOT NULL (or varchar(32) for a timestamp string, if preferred).
  • The backend must expose this in the tenant-settings DTO and allow PATCH to update it (or expose a dedicated “bump” endpoint that updates this column).
  • Schema file to update: code_html_template/SQLS/Current_Sqls/Latest_Schema_Post__Blob_Claude_12.sql. New columns may be added via ALTER in a migration; document the ALTER and ensure the canonical schema file is updated when the migration is applied.

Option B – New table: Create homepage_cache_meta with id (bigint, sequence_generator), tenant_id (varchar 255, FK to tenant_organization.tenant_id), version (bigint or varchar), updated_at (timestamp). Follow .cursor/rules/database_schema_guidelines.mdc: add DROP TABLE in reverse dependency order at the top of the schema file, CREATE TABLE in dependency order, and add the table to the sequence_generator setval GREATEST block.

Migrations: Document whether the backend (e.g. Flyway/Liquibase in malayalees-us-site-boot) runs migrations or a separate process applies schema changes.

Batch Job (Optional)

Cache warming

Repo: E:\project_workspace\event-site-manager-batch-jobs.

Optional job: Use EventBridge (or cron) to run every 3 days (or other interval). For each tenant, perform an HTTP GET to the homepage-payload URL (e.g. https://your-app-url/api/cached/homepage?tenantId=...) or call the backend to generate the payload. This warms the edge cache so that user requests are served from the CDN. See the “Cache warming” section in HOMEPAGE_EDGE_CACHE_GUIDELINE.html.

API Contract

Refresh and payload

Refresh (bump version):

  • If using PATCH tenant-settings: PATCH /api/tenant-settings/{id} with body { "homepageCacheVersion": <newValue> }. Backend returns 200 with updated resource.
  • If using dedicated endpoint: POST /api/homepage-cache/refresh with body { "tenantId": "tenant_demo_002" }. Backend returns 200 with { "version": <newVersion> } or 204 No Content. On error: 400 (invalid tenant), 403 (not admin), 404 (tenant not found).

Homepage payload: The homepage-payload API (e.g. GET /api/cached/homepage?tenantId=X&v=...) must read the current version for the tenant (from tenant_settings or homepage_cache_meta) and include it in the response or in the cache key. Clients use the v query parameter so that after an admin refresh, the next request uses a new v and the CDN fetches fresh content.

Post-Implementation Notes

Client-side cache behavior and configuration

After implementing the admin refresh and env-prefixed cache keys, the following fixes were applied to improve behavior in local dev and to clarify configuration.

1. Instant cache on refresh (useLayoutEffect)

Issue: In local dev, the loading image and section delays were still visible even after the first load and a browser refresh. Cached data was only read in useEffect, which runs after the first paint, so the loading image appeared for at least one frame and sections waited for their deferred timers (300ms, 800ms, 1500ms) before checking cache.

Fix:

  • TenantSettingsProvider – Added a useLayoutEffect that runs before paint and performs a synchronous read of the tenant-settings cache. When the cache is warm, it sets settings and loading = false in that layout effect, so the loading image is never shown.
  • TeamSection, UpcomingEventsSection, OurSponsorsSection, FeaturedEventsSection – Added a useLayoutEffect in each that runs before paint and performs a synchronous read of their sessionStorage cache. When cache is warm, each sets its data and loading = false (or isVisible = true for Featured) in that layout effect, so sections render immediately without waiting for the deferred delays.
  • Hero section (HeroSection / DynamicHeroImage) – Added homepage_hero_section_cache to src/lib/homepageCacheKeys.ts and a useLayoutEffect in the hero component that reads cached dynamicImages, upcomingEvents, and imageDurations before paint. When cache is warm, hero images and rotation show immediately on refresh. After hero data is initialized (from useFilteredEvents('hero') plus standalone hero media fetch), the component writes the same payload to sessionStorage. Hero image rotation logic and data sources are documented in .cursor/rules/hero_section_image_rotation.mdc.

Result: On refresh or repeat visit (cache warm), the loading image does not appear and Events, Featured Events, Team, Sponsors, and Hero section (rotating images and overlay) show content immediately. Cold first load is unchanged (deferred fetch and optional loading UI).

2. NEXT_PUBLIC_APP_ENV values

Question: Does NEXT_PUBLIC_APP_ENV have to be exactly local | development | staging | production?

Answer: No. The code does not validate the value; it uses whatever string is set (or 'local' when unset) as the cache key prefix in src/lib/homepageCacheKeys.ts. The four values above are a convention, not a strict allowlist.

Rules:

  • Different environments → different values: Use a different value per deployment (e.g. dev vs prod) so cache namespaces do not collide.
  • Same environment → same value: Use the same value for all builds of that environment so they share the same cache namespace.
  • Unset: If the variable is not set, the prefix is 'local'; all such deployments share the same “local” cache.

You may use other names (e.g. dev, prod, test, uat, preview, or branch names) as long as they are consistent per environment.
