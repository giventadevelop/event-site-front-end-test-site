Homepage Edge Cache Guideline – AWS CloudFront / Amplify

This guideline describes how to improve homepage load by serving static or cached data from the AWS edge (CloudFront or Amplify CDN) so that homepage sections can render immediately when the cache is warm, without a loading screen. The application runs on AWS Amplify (Lambda SSR), so all options use AWS-native services.

Overview

Goal and cache behavior

Goal: When the edge has a cached copy of the homepage data, portions of the page should load immediately from that cache so the user sees content without a loading screen. When the cache is cold or the response is slow, the existing deferred-loading pattern remains as fallback.

  • Cache hit: Portions load immediately from edge; no loading screen for those sections.
  • Cache miss: Keep current behavior (deferred fetch and optional loading UI).
  • Freshness: Cache is updated regularly (e.g., short TTL or scheduled cache warming).

Problem Statement

Current limitations and desired outcome

Today, the homepage uses client-side deferred loading and per-section sessionStorage cache (5-minute TTL). On a first visit or after cache expiry, multiple API calls run after first paint, which can still result in a visible loading experience and Lambda/backend load. There is no edge-level cache, so every cold request hits the origin.

Desired outcome: small, static-like payloads for the homepage sections are served from the edge when possible, so the page (or key portions) loads immediately without a loading screen when the cache is warm.

Current Homepage Pattern

Existing deferred loading and data sources

The existing pattern is defined in the project rule homepage_deferred_loading_pattern.mdc. Summary:

  • Hooks: usePageReady() and useDeferredFetch(delayMs) gate API calls until after initial paint.
  • Staggered delays: TenantSettings (gate) → Hero/Live/Featured events (500 ms) → Upcoming events (300 ms after mount) → Team (800 ms) → Sponsors (1500 ms).
  • Per-section cache: Each section checks sessionStorage first (5-minute TTL); cache hit avoids network.

Data sources (all via /api/proxy/...):

  • tenant-settings – Section visibility gate
  • event-details + event-medias – Hero, live, featured, upcoming events
  • executive-committee-team-members – Team section
  • event-sponsors – Sponsors section

Recommended Architecture

Edge-cached homepage payload
  1. Single aggregated endpoint – e.g. /api/homepage-payload or /api/cached/homepage – returns a JSON blob with all data needed for above-the-fold and key sections: tenant settings, hero/featured/upcoming events (with media), team members, sponsors. Response must be scoped by tenant (e.g. query param ?tenantId=... or path segment).
  2. CloudFront or Amplify CDN – Cache this endpoint with a cache key that includes tenant ID (and optionally locale). Use a short TTL (e.g. 1–5 minutes) so content stays fresh, or longer TTL with invalidation on admin updates.
  3. Homepage client logic – On load, try to fetch the cached payload first. If the response is fast (e.g. < 200–300 ms) and OK, render from it immediately and skip per-section fetches for that load. Otherwise, show loading/placeholder and fall back to the existing deferred per-section fetch. Continue to use sessionStorage for repeat visits within TTL.
  4. Optional cache warming – A scheduled job (e.g. EventBridge rule every 5–15 minutes) triggers a server-side or Lambda request to the homepage-payload URL per tenant so the edge stays warm.

This keeps the existing deferred pattern as fallback and adds a fast path when the edge has the payload.

Option Comparison: Amplify Cache-Control vs S3 + CloudFront

For a strategy that uses periodic update once every three days and on-demand refresh per tenant (e.g. when an admin publishes changes), the two most relevant options are Amplify custom headers / Cache-Control and S3 + CloudFront with scheduled warming. Below is whether Amplify’s caching is built-in, how each option fits that use case, cost, ease of setup, and a concise recommendation.

Does Amplify custom headers / Cache-Control exist as a built-in feature?

Yes. AWS Amplify Hosting uses CloudFront in front of your app and supports caching of dynamic (SSR/API) responses in two ways:
  • Application-level: Your homepage-payload route (e.g. Next.js API Route or Route Handler) returns a response with Cache-Control: public, max-age=259200 (3 days in seconds). Amplify’s managed cache policy respects this and will cache the response at the edge.
  • Custom headers file: You can add a customHeaders.yaml in the project root (or configure in the Amplify console) with a pattern such as /api/cached/** and set Cache-Control: public, max-age=259200. This overrides framework defaults and applies to successful 200 responses.

So no separate CloudFront distribution is required; the CDN in front of Amplify is already there and will cache when you set appropriate Cache-Control. Default TTL for compute (Lambda) responses is 0; setting max-age or s-maxage enables caching.

Use case: 3-day refresh + on-demand push per tenant

Behavior by option
  • Amplify Cache-Control: Set max-age=259200 (3 days). Content refreshes automatically after 3 days. On-demand invalidation per tenant is not built-in: Amplify does not expose a per-path CloudFront invalidation API to the application. To “push” an update for one tenant you would need to either (a) accept that the cache expires in at most 3 days, (b) use a shorter TTL after updates (e.g. re-deploy or a separate path with short TTL), or (c) put your own CloudFront in front and run invalidations there.
  • S3 + CloudFront + scheduled warming: A job (e.g. EventBridge + Lambda) runs every 3 days, fetches data per tenant, and writes s3://bucket/homepage-payload/{tenantId}.json. CloudFront caches each object. For on-demand push per tenant, you overwrite the S3 object for that tenantId and optionally create a CloudFront invalidation for /homepage-payload/{tenantId}.json. So both “every 3 days” and “on demand per tenant” are fully supported.

Cost comparison (order-of-magnitude)

Assumptions: a few tenants (e.g. 5–10), homepage payload ~200–500 KB per tenant, and either low traffic (e.g. 10K page views/month) or moderate (100K views/month). Prices are US East–style and approximate.

Ease of setup

  • Amplify Cache-Control (easier): Implement the homepage-payload API (e.g. /api/cached/homepage?tenantId=...). In that route, set the response header Cache-Control: public, max-age=259200 (and optionally s-maxage=259200). Optionally add customHeaders.yaml for the same path. No new AWS resources. On-demand invalidation per tenant is not supported out of the box.
  • S3 + CloudFront (more setup): Create S3 bucket and object key layout (e.g. homepage-payload/{tenantId}.json). Create or reuse a CloudFront distribution with origin = S3. Add a Lambda (triggered by EventBridge every 3 days) that calls your backend or existing API to build the payload per tenant and writes to S3. Optionally add an admin-triggered action (e.g. API or Lambda) to write S3 and invalidate CloudFront for that tenant. More moving parts, but full control over 3-day refresh and on-demand push per tenant.

Recommendation for your scenario

When to use which
  • Use Amplify Cache-Control if you want the cheapest and simplest approach and can accept that updates propagate at most every 3 days (no true on-demand invalidation per tenant). Ideal when homepage content changes infrequently and you are already on Amplify.
  • Use S3 + CloudFront + scheduled warming when you need on-demand refresh per tenant (e.g. admin clicks “Publish” and that tenant’s homepage updates within seconds via S3 overwrite + optional CloudFront invalidation), or when you want strict separation between “content generation” (job) and “content delivery” (S3 + CDN). Cost remains low for typical traffic; setup is more involved.

If you need both “periodic every 3 days” and “push on demand per tenant,” S3 + CloudFront is the option that supports both natively; Amplify Cache-Control supports only the periodic TTL unless you add your own invalidation layer.

Manual push and invalidation when using Amplify

Because Amplify does not expose a built-in “invalidate cache for this tenant” button or API, you can add your own control in two ways: cache-busting with a version parameter (no AWS APIs, works with Amplify as-is), or programmatic CloudFront invalidation (requires the distribution ID and IAM permissions).

Option 1: Cache-busting with a version parameter (recommended, no AWS API)

How it works

Include a version in the cache key so that when you “push” an update, the client requests a new URL. The CDN treats it as a new resource (cache miss) and fetches fresh content from the origin. No invalidation API is needed.

  1. Store a version per tenant – e.g. in your database or config: homepage_cache_version or last_published_at (timestamp). When an admin publishes changes for a tenant, update this value (e.g. set to Date.now() or current ISO string).
  2. Payload URL includes version – The client requests /api/cached/homepage?tenantId=X&v=1730123456. The v value can come from (a) a small “meta” endpoint that returns the current version for the tenant, or (b) the payload response itself (e.g. payload returns { version: 1730123456, ... } and the client uses that version in the next request). For the first load, the client can call the payload URL without v (or with v=0); the server returns the payload and the current version; the client then stores the version and uses it in subsequent requests.
  3. Admin “Refresh” or “Publish” action – When the admin clicks “Refresh homepage cache” or “Publish” for a tenant, your backend updates the stored version for that tenant. The next time any client requests the payload, they get the new version (either from the meta endpoint or from the payload response); they then request ?tenantId=X&v=newVersion. That URL is a new cache key, so the CDN fetches from origin and caches the new response. Old cached URLs expire naturally after TTL.

Implementation outline: Add an API route (e.g. POST /api/admin/refresh-homepage-cache) that accepts tenantId, validates the caller is an admin, and updates the tenant’s homepage_cache_version (or equivalent) in the database. The homepage-payload API reads this version and either (a) redirects to /api/cached/homepage?tenantId=X&v=... or (b) includes the version in the JSON so the client can use it in the next request. The client always requests with tenantId and v when known; otherwise it gets the version from the payload or a small meta endpoint first.

Advantages: No CloudFront API, no distribution ID, no IAM; works entirely within your app and Amplify’s CDN. You control “push” by updating a version in your data store and having clients use that version in the URL.

Option 2: Programmatic CloudFront invalidation

When you need to clear the CDN cache directly

If you want to actually invalidate the cached response for a path (so the next request refetches from origin), you can call the CloudFront CreateInvalidation API. Amplify hosts your app behind a CloudFront distribution that Amplify creates and manages. To use this approach you need:

  • Distribution ID – Amplify does not expose this in the app console by default. You can find it in the AWS CloudFront console (look for the distribution whose origin is your Amplify app URL), or via AWS APIs (e.g. ListDistributions and match by origin). Store it in an environment variable (e.g. AMPLIFY_CLOUDFRONT_DISTRIBUTION_ID) so you can use it from a server-side route or Lambda.
  • IAM permissions – The role that runs your API route or Lambda needs cloudfront:CreateInvalidation. Create an IAM policy that allows CreateInvalidation on the specific distribution (or use a broader policy if you have multiple distributions).
  • Invalidation paths – For a tenant-scoped payload, invalidate the path that includes the tenant (e.g. /api/cached/homepage*tenantId=tenant_123* or the exact path your app uses). CloudFront supports path patterns; the first 1,000 invalidation paths per month are free, then a small charge per path.

Implementation outline: Add an API route (e.g. POST /api/admin/refresh-homepage-cache) that accepts tenantId, validates the caller is an admin, then calls the AWS SDK (e.g. @aws-sdk/client-cloudfront) CreateInvalidation with the distribution ID and paths like ["/api/cached/homepage?tenantId=tenant_123"]. Use a unique CallerReference (e.g. timestamp) per request. After the invalidation is created, the next request to that URL will hit the origin and the new response will be cached.

Note: Amplify may recreate or change the distribution on certain operations (e.g. major config changes). If the distribution ID changes, update your env. Option 1 (cache-busting) avoids this dependency.

Cache Key Design

Cache key requirements
  • Tenant ID (required): Every cache key must include the tenant identifier so responses are never shared across tenants. Use query parameter (e.g. ?tenantId=tenant_demo_002) or path segment (e.g. /api/cached/homepage/tenant_demo_002).
  • Locale (optional): If the payload varies by language, include locale in the cache key (query or path).
  • Path: Restrict caching to the homepage-payload path only so other API routes are not cached by mistake.

TTL and Invalidation

Suggested TTL and invalidation
  • Suggested TTL: 1–5 minutes for a good balance between freshness and hit rate. For infrequent updates, 3 days (max-age=259200) is also common and reduces origin load; use with Amplify Cache-Control or CloudFront.
  • Invalidation on admin updates: When an admin updates events, tenant settings, team, or sponsors, optionally invalidate the cache for that tenant (e.g. CloudFront invalidation /api/cached/homepage?tenantId=* or path pattern). With S3 + CloudFront, you can overwrite the object for that tenant and create an invalidation for /homepage-payload/{tenantId}.json to get on-demand refresh per tenant. With Amplify-only, invalidation is not exposed; rely on TTL.

Cache Warming

Scheduled cache warming

To keep the edge cache warm, use a scheduled job that calls the homepage-payload URL for each tenant (or, for S3 + CloudFront, writes the payload to S3 per tenant):

  • EventBridge (CloudWatch Events): Create a rule with a schedule expression (e.g. rate(5 minutes), cron(0/15 * * * ? *) for every 15 minutes, or cron(0 8 * * ? *) for once daily at 8:00 UTC). For once every 3 days, use e.g. cron(0 8 1/3 * ? *) (every 3 days at 8:00 UTC) or a fixed schedule that fits your release cycle.
  • Target: A Lambda function that performs an HTTP GET to https://your-app-url/api/cached/homepage?tenantId=... for each configured tenant (e.g. from environment variable or config).
  • Result: CloudFront (or Amplify CDN) serves the request from origin once per period and caches the response; subsequent user requests are served from the edge.

Ensure the Lambda uses the same origin URL and tenant IDs as production so the cache is populated correctly.

Client Behavior

Homepage client flow
  1. Check sessionStorage – If a valid cached payload exists (same structure as edge payload, within client-side TTL), use it and render immediately; skip network.
  2. Fetch payload with timeout – Request /api/cached/homepage?tenantId=... (or equivalent). Use a timeout (e.g. 300 ms).
  3. Fast and OK: If response returns within the timeout and status is 200, parse JSON and render all sections from the payload. Optionally store in sessionStorage for the next visit. Do not run the existing per-section deferred fetches for this load.
  4. Slow or error: If timeout fires or status is not OK, ignore the payload and run the existing deferred pattern (usePageReady / useDeferredFetch and per-section API calls). Show loading/placeholder as today.

This way, when the edge is warm, users get instant content; when it is cold, behavior matches the current implementation.

Security and Tenant Isolation

Security requirements
  • Cache key must include tenant: Never cache a response without tenant in the key; otherwise one tenant could see another tenant’s data.
  • Payload endpoint: The homepage-payload API must enforce tenant server-side (e.g. from NEXT_PUBLIC_TENANT_ID or a validated token). Do not rely only on client-supplied tenant ID for authorization; use it only for cache keying after the server has determined the tenant.
  • Public read-only: The payload is public (homepage data). Ensure it contains no sensitive or user-specific data.

Checklist and Best Practices

Implementation checklist
Implementation checklist
  • Implement the aggregated homepage-payload endpoint (e.g. /api/cached/homepage) that returns tenant-scoped JSON.
  • Configure CloudFront or Amplify CDN to cache this path with a short TTL and cache key that includes tenant (and optionally locale).
  • Add client-side logic: try payload first with a short timeout; if fast and OK, render from it and skip per-section fetch; else use existing deferred flow.
  • Optionally add cache warming (EventBridge + Lambda) to hit the payload URL per tenant on a schedule.
  • Optionally add cache invalidation when admins update events, settings, team, or sponsors.
  • Monitor cache hit rate (e.g. CloudFront metrics or application logs) to tune TTL and warming frequency.
