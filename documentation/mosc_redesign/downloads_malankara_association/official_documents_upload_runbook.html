Official Documents — Migration Upload Runbook

Layer: Data migration (Node scripts → Spring Boot API)  |  Version: 1.1  |  Date: June 27, 2026
Related: Malankara PRD  |  Gap analysis  |  Full migration session (June 2026)  |  scripts/mosc-in-migration/README.md  |  scripts/README.md (doc folder wrappers)

This runbook documents how bulk official document uploads are performed for the Malankara Association migration (and how the same pattern can be reused elsewhere). Scripts call the Rust/Spring backend directly at http://localhost:8080 using a service JWT — not the Next.js app at :3000 and not the /api/proxy/... layer.

June 2026 outcome (Malankara): 58 missing 2021/2022 files re-uploaded; 76 card thumbnails backfilled; 5 year-bundle covers set; public API X-Total-Count increased from 91 → 149 documents on /mosc-redesign/downloads.

1. Upload architecture

Legacy mosc.in PDF URLs
    → build-malankara-manifest.mjs  (categorySlug + year remap)
    → fetch-urls.mjs                (download to MOSC_DOWNLOAD_ROOT)
    → upload-manifest-to-official-docs.mjs
         │
         ├─ migration-api-lib.mjs
         │     POST http://localhost:8080/api/authenticate  (service JWT)
         │     All calls: Authorization: Bearer <token>
         │                X-Tenant-ID: <NEXT_PUBLIC_TENANT_ID>
         │
         ├─ POST /api/event-medias/upload/tenant-official-document  (multipart: file + metadata)
         ├─ PATCH /api/event-medias/{id}                            (hierarchy fields)
         ├─ POST /api/event-medias/{id}/upload-official-document-thumbnail  (optional cover JPG)
         └─ POST/PATCH /api/official-document-year-bundles            (year section cover)

PostgreSQL event_media + official_document_category
    → S3 (eventapp-media-bucket, us-east-2)
    → GET /api/event-medias/public-official-documents?tenantId=...
    → Next.js downloads hub (/mosc-redesign/downloads)

Why direct backend (:8080)? Migration runs outside the browser. Service JWT credentials (API_JWT_USER / API_JWT_PASS) live in .env.local and authenticate against the Spring Boot API. The Next.js proxy at :3000/api/proxy/... is for the web app; older mirror upload used it, but Malankara manifest upload uses migration-api-lib.mjs for reliability and fewer hops.

2. Backend API URL (localhost:8080)

The shared library resolves the API base URL in this order:

  1. MOSC_API_BASE_URL (optional override for scripts)
  2. NEXT_PUBLIC_API_BASE_URL from .env.local (same value the Next.js app uses)
  3. Default fallback: http://localhost:8080
SettingTypical local valueUsed by
NEXT_PUBLIC_API_BASE_URLhttp://localhost:8080Next.js server actions, migration scripts
MOSC_API_BASE_URL(optional) same as aboveScripts only — overrides without changing Next env
NEXT_PUBLIC_TENANT_IDmosc_malankara_orthodox_2Tenant scope on every API call
API_JWT_USER / API_JWT_PASSService account (e.g. jwtadmin)POST /api/authenticate → Bearer token

Every authenticated request from migration-api-lib.mjs sends:

Authorization: Bearer <service JWT from /api/authenticate>
X-Tenant-ID: <NEXT_PUBLIC_TENANT_ID>
Content-Type: application/json   (or multipart for uploads)

Production / staging: Set NEXT_PUBLIC_API_BASE_URL to the deployed backend URL (e.g. Amplify/API Gateway). Scripts read the same variable — no code change required.

3. Prerequisites

  • Spring Boot backend running and reachable at NEXT_PUBLIC_API_BASE_URL (local: http://localhost:8080).
  • AWS credentials on the backend process (AWS_ACCESS_KEY_ID, AWS_SECRET_ACCESS_KEY, AWS_REGION=us-east-2, bucket eventapp-media-bucket).
  • Service JWT user has admin authority (upload endpoint calls requireAdmin()).
  • Official document categories seeded for the tenant (Malankara slugs in DB).
  • Node.js 20+; repo root .env.local with JWT and tenant vars.
Environment variables (Malankara upload)
  • MOSC_DOWNLOAD_ROOT — local PDF folder (e.g. f:\project_workspace\mosc-temp\.mosc-downloads). No trailing space after set VAR=value on Windows.
  • MOSC_UPLOAD_LIMIT=0 — required for full runs (unset or 0; a limit of 3 only processes three items).
  • MOSC_UPLOAD_DELAY_MS — pause between uploads (default 400).
  • MOSC_UPLOAD_MAX_RETRIES — retry on S3/backend 500 (default 3).
  • MOSC_AUTO_CREATE_CATEGORIES=true — only if a category slug is missing.

5. Step-by-step: Malankara upload (Windows)

5.1 Set environment and verify backend

Health check — backend on :8080
curl -s http://localhost:8080/management/health
Windows single-line:
curl -s http://localhost:8080/management/health

5.2 Fetch PDFs (if not already on disk)

Download manifest PDFs to MOSC_DOWNLOAD_ROOT
set MOSC_DOWNLOAD_ROOT=f:\project_workspace\mosc-temp\.mosc-downloads && cd F:\project_workspace\mosc-temp && npm run mosc:migration:malankara:fetch
Windows single-line:
set MOSC_DOWNLOAD_ROOT=f:\project_workspace\mosc-temp\.mosc-downloads && cd F:\project_workspace\mosc-temp && npm run mosc:migration:malankara:fetch

Uses --insecure-tls --skip-existing for mosc.in TLS quirks on Windows.

5.3 Dry-run upload

Dry-run — no API writes
set MOSC_DOWNLOAD_ROOT=f:\project_workspace\mosc-temp\.mosc-downloads && set MOSC_UPLOAD_LIMIT=0 && cd F:\project_workspace\mosc-temp && npm run mosc:migration:malankara:upload:dry
Windows single-line:
set MOSC_DOWNLOAD_ROOT=f:\project_workspace\mosc-temp\.mosc-downloads && set MOSC_UPLOAD_LIMIT=0 && cd F:\project_workspace\mosc-temp && npm run mosc:migration:malankara:upload:dry

5.4 Upload missing files + thumbnails (recommended repair)

Missing-only upload + thumbnail backfill + year bundles
set MOSC_DOWNLOAD_ROOT=f:\project_workspace\mosc-temp\.mosc-downloads && set MOSC_UPLOAD_LIMIT=0 && cd F:\project_workspace\mosc-temp && npm run mosc:migration:malankara:upload:missing
Windows single-line:
set MOSC_DOWNLOAD_ROOT=f:\project_workspace\mosc-temp\.mosc-downloads && set MOSC_UPLOAD_LIMIT=0 && cd F:\project_workspace\mosc-temp && npm run mosc:migration:malankara:upload:missing

Equivalent flags: --missing-only --backfill-thumbnails --ensure-year-bundles --insecure-tls. Log: .mosc-downloads/upload-run-missing.log.

⚠ Avoid blind --force on full manifest
npm run mosc:migration:malankara:upload:force deletes existing rows before re-upload. An earlier bug in delete/dedupe logic (unfiltered API query) caused collateral deletion of 2021/2022 rows. The script is fixed, but prefer --missing-only for repairs unless you intentionally overwrite every file.

8. Dedupe, missing detection, and the force-upload bug (fixed)

Unique key: categorySlug | officialDocumentYear | hierarchyPath (case-insensitive).

Manifest: 165 entries → 134 unique keys (31 duplicate diocese list paths in manifest).

Fixed behavior: official-docs-query.mjs loads all rows with eventMediaType.equals=TENANT_OFFICIAL_DOCUMENT, then filters by officialDocumentCategoryId + officialDocumentYear in Node. Backend criteria alone were unreliable and caused --force to delete unrelated tenant media.

--missing-only: Diffs manifest unique keys against DB Malankara rows; uploads only gaps (e.g. 58 files in 2021/2022 categories).

10. Reusing this pattern for other migrations

The same stack works for any bulk ingest of tenant official documents (synod minutes, circulars, another legacy folder tree). Copy the pattern — not necessarily the Malankara-specific manifest.

10.1 Checklist for a new document set

  1. Manifest JSON — array of { url, categorySlug, year, filename, hierarchyPath, displayName } (see url-list.example.json).
  2. Categories — create slugs in admin or SQL seed before upload (/admin/official-document-categories).
  3. Fetch — fetch-urls.mjs --manifest your-list.json → flat layout under MOSC_DOWNLOAD_ROOT.
  4. Upload — upload-manifest-to-official-docs.mjs --manifest your-list.json (direct :8080).
  5. Optional covers — add a *-category-covers.json map slug → image URL; use --backfill-thumbnails.
  6. Verify — public API count + downloads hub filters.

10.2 When to use which upload script

ScenarioScriptAPI target
Flat manifest (categorySlug/year/file.pdf on disk)upload-manifest-to-official-docs.mjsDirect :8080 ✅
Windows Explorer tree (Downloads\Category\Section\file.pdf)upload-mirror-to-official-docs.mjsHistorically Next proxy; consider migrating to migration-api-lib
Admin one-off / QAAdmin UI bulk uploadNext.js server actions → :8080

10.3 Extending migration-api-lib.mjs

For new automation (e.g. bulk PATCH, category import, year bundle seeding):

import { assertEnv, apiFetch, TENANT_ID, API_BASE_URL } from './migration-api-lib.mjs';

assertEnv();
const { res, json, text } = await apiFetch('/api/your-endpoint', {
  method: 'POST',
  headers: { 'Content-Type': 'application/json' },
  body: JSON.stringify({ tenantId: TENANT_ID, ...payload }),
});

Do not hardcode http://localhost:8080 in new scripts — always use apiFetch so production env vars apply.
