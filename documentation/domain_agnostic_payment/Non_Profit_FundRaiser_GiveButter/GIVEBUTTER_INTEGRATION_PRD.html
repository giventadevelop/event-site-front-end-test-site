Givebutter Integration PRD
Zero-Fee Fundraising & Mass Offerings

1. Executive Summary

This PRD outlines the integration of Givebutter as a zero-fee payment provider for nonprofit fundraising events and church Mass offerings. The implementation will leverage the existing domain-agnostic payment infrastructure, following the same patterns established for Stripe payments.

Key Objectives

  • Enable zero-fee fundraising for nonprofit events and campaigns
  • Support Mass offerings and church collections with prayer intentions
  • Maintain consistency with existing Stripe payment flow architecture
  • Support multi-tenant configuration with per-tenant Givebutter accounts
  • Enable recurring donations and offerings
  • Provide QR code generation for in-person giving

2. Background & Context

2.1 Current State

The platform currently supports Stripe payments for ticket sales through a domain-agnostic payment system. The infrastructure includes:

  • Multi-tenant payment provider configuration (payment_provider_config table)
  • Payment orchestration service for routing payments to appropriate providers
  • Payment transaction tracking (payment_transaction table)
  • Webhook handling for payment status updates
  • Frontend payment components (UniversalPaymentCheckout)

2.2 Business Need

Nonprofit organizations and churches require zero-fee payment processing for:

  • Fundraising Events: Charity events, galas, and campaigns where 100% of donations should reach the cause
  • Mass Offerings: Weekly collections and special offerings with optional prayer intentions
  • Recurring Donations: Monthly or weekly recurring giving programs

2.3 Why Givebutter

✅ Givebutter Advantages

  • Zero Fees: True zero-fee model (optional donor tips)
  • Open REST API: Direct integration without Zapier dependency
  • Spring Boot Compatible: Standard REST API calls with API key authentication
  • Multi-Tenant Ready: Works seamlessly with existing tenant-based payment config
  • Feature Rich: Supports recurring donations, custom fields (prayer intentions), QR codes, and webhooks
  • Well Documented: Comprehensive API documentation

3. Scope & Requirements

3.1 In Scope

  • Integration of Givebutter as a payment provider type
  • Support for DONATION_ZERO_FEE payment type for fundraising events
  • Support for OFFERING payment type for Mass offerings
  • Multi-tenant configuration (per-tenant Givebutter accounts)
  • Webhook handling for donation status updates
  • Frontend components for donation/offering flows
  • QR code generation for in-person giving
  • Recurring donation/offering support
  • Prayer intention custom fields for Mass offerings

3.2 Out of Scope

  • Migration of existing Stripe transactions to Givebutter
  • Givebutter campaign management UI (use Givebutter dashboard)
  • Donor management features (handled by Givebutter)
  • Receipt generation (handled by Givebutter)

3.3 Payment Types Supported

Payment Type Use Case Givebutter Campaign Features Event Configuration
TICKET_SALE Fundraiser/charity events with tickets Event-specific campaign Ticket sales, zero fees, event-specific campaign isFundraiserEvent: true + donationConfig.useZeroFeeProvider: true
DONATION / DONATION_ZERO_FEE Nonprofit fundraising events Event-specific campaign Preset amounts, custom amounts, anonymous option, recurring donationConfig.useZeroFeeProvider: true
OFFERING Church Mass offerings Mass offering campaign Preset amounts, prayer intentions, QR codes, recurring, anonymous Always routes to Givebutter if configured

✅ Ticketed Fundraiser Events

Key Feature: Ticketed events can be configured as fundraisers, routing all ticket sales to Givebutter for zero-fee processing.

Configuration: Set isFundraiserEvent: true (or isCharityEvent: true) in event metadata along with donationConfig.useZeroFeeProvider: true and donationConfig.zeroFeeProvider: "GIVEBUTTER".

Flow: When users purchase tickets for a fundraiser event, the backend automatically routes the payment to Givebutter instead of Stripe, ensuring zero fees for nonprofit organizations.

4.2 Payment Flow

  1. Frontend: User selects donation/offering amount and fills form
  2. Frontend: Calls POST /api/proxy/payments/initialize with payment details
  3. Backend: PaymentOrchestrationService routes to GivebutterPaymentAdapter
  4. Backend: Adapter retrieves tenant's Givebutter config from database
  5. Backend: Adapter calls Givebutter API to create donation
  6. Backend: Returns checkout URL to frontend
  7. Frontend: Redirects user to Givebutter checkout page
  8. Givebutter: Processes payment and sends webhook
  9. Backend: Webhook handler updates payment transaction status

4.3 Multi-Tenant Configuration

Each tenant can have their own Givebutter account configuration stored in payment_provider_config:

  • API Key: Encrypted in config_data field
  • Webhook Secret: Encrypted in webhook_secret field
  • Campaign IDs: Stored in metadata JSONB field
  • Use Case Routing: Separate configs for OFFERING vs DONATION_ZERO_FEE

5. Database Schema Changes

5.1 Payment Provider Type Enum

-- Add GIVEBUTTER to payment_provider_type enum ALTER TYPE payment_provider_type ADD VALUE IF NOT EXISTS 'GIVEBUTTER';

5.2 Payment Type Enum

The following payment types already exist and will be used:

  • TICKET_SALE - Ticket sales (can be routed to Givebutter for fundraiser events)
  • DONATION - General donations (can be routed to Givebutter if zero-fee configured)
  • OFFERING - Mass offerings and church collections

Note: DONATION_ZERO_FEE may need to be added if not already present in the enum.

5.3 Event Metadata for Fundraiser Events

Events can be marked as fundraisers using metadata fields. The metadata column is of type TEXT (not JSONB) and stores JSON as a string, matching the existing codebase pattern.

-- Mark event as fundraiser and configure Givebutter (stored as JSON string) UPDATE event_details SET metadata = '{"isFundraiserEvent":true,"donationConfig":{"useZeroFeeProvider":true,"zeroFeeProvider":"GIVEBUTTER","givebutterCampaignId":"campaign_fundraiser_123"}}' WHERE id = {event_id}; -- Note: Parse JSON in application code using Jackson ObjectMapper -- ObjectMapper mapper = new ObjectMapper(); -- Map<String, Object> metadata = mapper.readValue(event.getMetadata(), Map.class);

⚠️ Metadata Column Type

Uses TEXT (not JSONB): Following the existing codebase pattern (see user_payment_transaction.metadata text), the metadata column uses TEXT type. JSON data is stored as a string and parsed in application code using Jackson ObjectMapper.

10. Risks & Mitigations

Risk Impact Mitigation
Givebutter API changes High Monitor Givebutter API documentation, version API calls, implement adapter pattern for easy updates
Webhook delivery failures Medium Implement webhook retry logic, log all webhook events, provide manual status update capability
Multi-tenant config errors Medium Validate config on save, provide clear error messages, implement config validation service
Payment flow inconsistencies Low Follow existing Stripe payment patterns, reuse UniversalPaymentCheckout component
