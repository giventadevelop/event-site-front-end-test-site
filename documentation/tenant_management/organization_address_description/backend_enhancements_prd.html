Organization Address & Description — Backend Enhancements (v2.0)

Layer: Rust/JHipster REST API  |  Version: 2.0  |  Date: June 17, 2026
Index: generic_enhancement_prd.html  |  Schema: database_schema_enhancements_prd.html

Extend TenantOrganization entity, DTO, mapper, and REST resource for identity fields. Do not extend TenantSettings with new identity columns in v2.0. Reject or ignore PATCH/PUT of deprecated settings identity fields.

1. Affected REST resources

ResourceBase pathv2.0 write scope
Tenant Organization/api/tenant-organizationsAll identity fields (description, address, website)
Tenant Settings/api/tenant-settingsOperational fields only; no address/description writes

2. TenantOrganizationDTO — fields (canonical)

{
  "description": "string | null, max 1000",
  "addressLine1": "string | null, max 255",
  "addressLine2": "string | null, max 255",
  "city": "string | null, max 255",
  "stateProvince": "string | null, max 255",
  "zipCode": "string | null, max 20",
  "country": "string | null, max 100",
  "websiteUrl": "string | null, max 1024"
}

Plus existing org fields: organizationName, contactEmail, contactPhone, branding, subscription, etc.

3. TenantSettingsDTO — deprecated identity fields

v2.0 rule: If clients PATCH addressLine1, description, city, etc. on tenant-settings, return 400 Bad Request with message: Identity fields moved to tenant-organization; use PATCH /api/tenant-organizations/{id}. Alternatively strip deprecated keys silently (prefer explicit 400 for admin clarity).

GET may still return legacy columns for backward compatibility until v2.1 DROP. Mark OpenAPI fields as deprecated: true.

Active settings fields (unchanged): email, phoneNumber, homepage toggles, social URLs, hero images, limits, WhatsApp/Twilio, etc.

4. Validation rules (organization)

  • All identity fields optional (nullable).
  • description: max 1000 characters.
  • websiteUrl: if non-empty, must start with http:// or https://.
  • PATCH merge-patch: omitted fields unchanged.

5. Optional composite read endpoint (future)

Not required v2.0. Frontend can fetch org + settings and merge via resolveTenantOrganizationIdentity. If backend adds convenience:

GET /api/tenant-organizations/by-tenant/{tenantId}/with-settings
→ { organization: {...}, settings: {...}, resolvedIdentity: {...} }

8. Implementation checklist

  • Run migration 001_tenant_org_address_description.sql.
  • Add identity columns to TenantOrganization entity + mapper + DTO.
  • Do not add identity columns to TenantSettings entity in v2.0.
  • Reject PATCH of deprecated settings identity fields (or document ignore policy).
  • Mark deprecated settings fields in OpenAPI.
  • Integration tests: org create/update with full address; settings PATCH without address succeeds; settings PATCH with address returns 400.
  • Verify multi-tenant tenantId scoping unchanged.
Proxy note (Next.js): Frontend writes identity via /api/proxy/tenant-organizations only. Settings proxy unchanged for operational fields.
