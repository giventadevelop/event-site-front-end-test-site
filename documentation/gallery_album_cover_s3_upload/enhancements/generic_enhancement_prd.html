Gallery albums store a cover thumbnail in gallery_album.cover_image_url. Today admins can paste a URL or pick cover from album media already on S3. Static gallery import sets local Next.js paths (/images/mosc/gallery/...) instead of S3 URLs. Production and CDN deployments need covers uploaded to AWS S3 using the same pipeline as focus groups, sponsors, and event media.

Non-breaking rule: Existing cover_image_url values (local paths, external URLs, or S3) continue to work. New upload flow adds S3-backed covers without schema migration in v1.0.

Reference implementation to clone: Focus group cover upload — backend /api/event-medias/upload/focus-group-cover-image, proxy src/pages/api/proxy/event-medias/upload/focus-group-cover-image.ts, UI src/components/FocusGroupCoverImageUpload.tsx.

Executive summary

Problem and solution

Problem: Album covers are often stored as plain text URLs or static site paths. There is no dedicated “upload cover to S3” action on album create/edit (unlike focus groups). Static porting script defaults to URL mode, so 29 MOSC albums reference /images/mosc/gallery/{slug}/... which may not resolve in production if assets are not bundled or if the gallery is served from a different host.

Solution (v1.0): Add POST /api/event-medias/upload/gallery-album-cover-image (backend) + Next.js proxy + GalleryAlbumCoverImageUpload component. After upload, backend sets gallery_album.cover_image_url to the S3 fileUrl and optionally creates an event_media row linked to the album. Extend porting script with --upload-covers to push cover files to S3 post-import.

Success criteria

  • Admin uploads a JPEG/PNG cover on album edit → file appears in S3 → cover_image_url is HTTPS S3 URL.
  • Public /gallery and admin album list render cover from S3 (Next.js images.remotePatterns already allow bucket).
  • “Set as cover” from album media grid updates album when media is already on S3.
  • Static import can optionally upload all 29 cover images to S3 and replace local paths.
  • No regression for albums that already have valid cover URLs.
