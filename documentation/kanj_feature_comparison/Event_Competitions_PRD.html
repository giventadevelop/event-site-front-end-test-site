1. Executive summary

Problem and solution

Organizations run events with one or many competitions—youth festivals (parents register children), adult talent nights (competitors register themselves), or mixed programs. After judging, organizers publish winners with placement, prize details, and photos.

Our platform supports ticket types, Stripe checkout, and optional isSportsEvent scorecards. It does not support a competition catalog, participant profiles, structured registrations, or rich published winners.

Proposed solution: is_competition_event on event_details plus event_competition_* tables, REST APIs, routes under /events/[id]/competitions/, and audience_mode (YOUTH | ADULT | MIXED) on settings.

Scope

In scope (v1 PRD)

  • Admin: enable competition mode; audience_mode; days/venues; deadline; results with prizes/photos
  • Admin: CRUD competition catalog (individual/group, fees, capacity)
  • Guardian (YOUTH/MIXED): child participants; register; pay; view registrations
  • Competitor (ADULT/MIXED): self-registration; view own registrations
  • Public: hub, schedule, rules, winners board + gallery

Out of scope (v1 suggestions)

  • Full duplicate of KANJ Wix member portal
  • Automated soundtrack upload workflow (email-only per KANJ today)
  • Judge scoring app / live tabulation
  • Waitlist and lottery when capacity exceeded (phase 4)

5. Functional requirements

5.1 Event classification

FR-1 — Event Competitions flag
  • FR-1.1: Add is_competition_event (boolean, default false) on event_details.
  • FR-1.2: Admin toggles flag in event create/edit form (alongside existing isSportsEvent).
  • FR-1.3: When true, public event page shows prominent Event Competitions entry (hub link, register CTA).
  • FR-1.4: When true, standard ticket purchase UI is hidden or secondary (configurable: event_competition_settings.allow_ticket_sales default false).

5.2 Festival days (multi-venue)

FR-2 — Festival days
  • FR-2.1: Admin can add 1..n festival days per event: label (e.g. "Day 1"), date, venue name, address, sort order, optional notes.
  • FR-2.2: Public hub displays all days with venue details (KGT'26 two-location pattern).
  • FR-2.3: Competitions may optionally link to a festival day and optional start/end time for schedule grid.

5.3 Competition catalog

FR-3.1: Group category resolution: effective category = max category among registered children in group (configurable in settings).

5.4 Participant profiles

FR-4 — event_competition_participant
  • FR-4.1 (YOUTH): Guardian creates multiple CHILD participants linked to guardian_user_profile_id.
  • FR-4.2 (ADULT): Competitor creates or reuses one ADULT self-linked participant (or uses user_profile directly per implementation).
  • FR-4.3: Fields: name, optional DOB; grade/school for YOUTH; division_label / age bracket for ADULT.
  • FR-4.4: Deactivate only; no hard-delete if registrations exist.
  • FR-4.5: Public site never lists full PII; use display_name on results per privacy setting.

5.5 Registration

FR-5 — Registration workflow
  • FR-5.1: Registration cart separate from ticket cart; line items = (participant, competition) pairs.
  • FR-5.1b: Branch wizard by audience_mode: YOUTH = parent + children; ADULT = self (+ team members for GROUP); MIXED = both.
  • FR-5.2: Enforce global registration deadline from event_competition_settings.registration_deadline.
  • FR-5.3: Enforce per-competition capacity (max_participants).
  • FR-5.4: Prevent duplicate registration: same child + same competition.
  • FR-5.5: Validate child grade/category against competition category rules (with admin override flag optional).
  • FR-5.6: Group registration: one "group leader" registration record + event_competition_group_member rows for each child.
  • FR-5.7: Status workflow: PENDING_PAYMENT → CONFIRMED | CANCELLED.
  • FR-5.8: Payment: Stripe metadata registrationType: event_competition, competitionRegistrationIds; or admin confirm with $0.
  • FR-5.9: No walk-in registrations (UI + API reject after deadline).

5.6 Results and published winners

FR-6 — event_competition_result
  • FR-6.1: Admin enters results per competition: participant_profile_id (or group), display_name, placement (1/2/3+), optional placement_label, points_awarded, notes.
  • FR-6.2: prize_title, prize_details for public winner cards.
  • FR-6.3: Winner photo via winner_media_id (FK event_medias) and/or winner_photo_url — same admin upload pattern as event gallery.
  • FR-6.4: Publish: is_published, published_at; public route /events/[id]/competitions/winners.
  • FR-6.5: Public winners board: cards with photo, name (per results_display_mode), placement, prize; optional winners gallery grid for 1st place.
  • FR-6.6: Optional champion rollup when enabled (YOUTH: exclude group points per KGT preset).
  • FR-6.7: Privacy: stricter default for YOUTH (initials vs full name). See Sports/Arts Results for orthogonal is_sports_event scorecards.

5.7 Festival content (CMS-lite)

FR-7 — Content blocks
  • FR-7.1: Event-scoped content blocks: RULES, FAQ, GUIDELINES, DISCLAIMER (markdown/HTML).
  • FR-7.2: Optional schedule PDF URL or image URL (KANJ uses images today).
  • FR-7.3: Eligibility text configurable per tenant/event (KANJ: Kerala origin, tristate, Malayalee parent).

13. Open questions (stakeholder sign-off)

Decisions needed before P1
  1. Fees: Per-competition fee only (KANJ model) vs optional single "festival pass" bundle?
  2. Payment: Is Stripe required at launch, or admin manual confirm / pay-at-door acceptable for v1?
  3. Categories: Fixed four grade bands (KANJ) vs tenant-configurable categories in DB?
  4. Group billing: One paying group leader vs per-parent split for group members?
  5. Privacy: Public results show full child name, first + last initial, or anonymous placement codes?
  6. Eligibility: Enforce Malayalee/Kerala rules in software, or display-only honor code?
  7. Max placements: 3 only vs extended podium?
  8. Winner photos: Required before publish or optional?
  9. Ticket coexistence: Can is_competition_event coexist with ticket sales (allow_ticket_sales=true)?
  10. MIXED cart: Same Stripe cart for tickets + competitions or separate?
