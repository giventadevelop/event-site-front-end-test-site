Organization Address & Description — Gap Analysis

Index: generic_enhancement_prd.html  |  Version: 2.0  |  Date: June 17, 2026

v2.0 consolidates identity fields on tenant_organization. tenant_settings address columns are legacy; they must not receive new writes after migration. Status key: Present · Partial · Missing · Deprecated.

1. Source-of-truth decision

Why organization, not settings?

  • tenant_organization is the parent identity entity (tenant_settings.tenant_id FK → org).
  • Name, domain, branding, and “where we are” belong with organization metadata — not app configuration.
  • Duplicating on both tables caused admin confusion and drift risk (v1.0 anti-pattern).
  • Settings remains correct home for email, phone_number, WhatsApp/Twilio, homepage toggles, and social URLs.

3. Read fallback (public / footer / contact)

Until deprecated settings columns are dropped, consumers use:

resolveTenantOrganizationIdentity({ organization, settings })
  → for each field: organization.value ?? settingsLegacy.value ?? null

Apply to: Footer.tsx, MoscRedesignFooter.tsx, contact sections, About pages (future). Admin organization detail reads org table only.

4. Frontend gaps (v2.0 refactor)

Tenant Organization (TenantOrganizationForm.tsx)

  • Target Description + full address + websiteUrl — sole edit surface.
  • Refactor Ensure PATCH/POST sends only org resource; view page shows all identity fields.

Tenant Settings (TenantSettingsForm.tsx)

  • Remove Description textarea and entire Address Information section from form.
  • Remove Address display from settings view (or show read-only org address with link “Edit on Organization”).
  • Keep email, phone_number, social URLs, feature flags, hero images, etc.
