GitHub Actions: Re-run Workflows and Trigger New Deploys

Location: documentation/devops/GITHUB_ACTIONS_RERUN_AND_WORKFLOW_DISPATCH_GUIDE.html
Repository example: giventadeveloporg/malayalees-us-site-boot · Workflow: deploy-dev.yml

This guide documents three related topics: how to re-run a GitHub Actions workflow from the UI, why the Re-run button may be missing, and what to do when gh run rerun fails because the run is older than 30 days (including triggering a new run with gh workflow run).

1. Re-run the last GitHub Actions workflow (UI)

📋 Steps in the GitHub web UI
  1. Open the repository on GitHub (for example giventadeveloporg/malayalees-us-site-boot).
  2. Click the Actions tab.
  3. In the left sidebar, click the workflow name (for example Deploy to Dev or deploy-dev.yml).
  4. Click the most recent run in the list (failed runs show a red ✗; successful runs show a green ✓).
  5. Top right → Re-run jobs or Re-run all jobs.
Button What it does
Re-run all jobs Runs every job in the workflow again from the start.
Re-run failed jobs Re-runs only jobs that failed (when GitHub offers this option).
💡 Important: A re-run uses the same commit and branch as the original run, but picks up current secrets and variables. Update Environment secrets (not variables) for sensitive values such as AWS_ACCESS_KEY_ID before re-running a deploy that uses AWS credentials.

2. Re-run not visible on the run page

⚠️ Common case: On some run summary pages (especially older successful runs), there is no large Re-run jobs button in the header. Check the ⋯ (three dots) menu at the top right of the run title first.

2.1 Use the ⋯ menu on the run page

  1. Open the specific workflow run (for example Merge pull request #18…).
  2. Click ⋯ next to the run title.
  3. Look for Re-run all jobs or Re-run failed jobs.

2.2 Re-run from the workflow runs list

  1. Actions tab → select the workflow in the left sidebar.
  2. In the table of runs, hover the right side of the row.
  3. Click ⋯ on that row → Re-run all jobs.

2.3 Why Re-run may be unavailable

Reason What to do
Run older than ~30 days GitHub does not allow re-run; start a new run (see Section 3).
Read-only access to the repo Need Write or Maintain permission, or ask an admin to re-run.
Fork without write access Re-run on the upstream repository, not the fork.
Wrong repository Confirm you opened the repo where the deploy failed (for example event-site-manager-batch-jobs vs malayalees-us-site-boot).
Archived repository Unarchive or use another trigger method.
Optional — re-run from a commit: On the Commits page, open a commit → status icon → Details → Re-run jobs if shown.

3. 30-day limit and gh run rerun

GitHub CLI error (example):
run 23694863798 cannot be rerun; Unable to retry this workflow run because it was created over a month ago

GitHub blocks re-running workflow runs that are older than approximately 30 days. This applies in both the web UI and the CLI. A run from two months ago (for example run 23694863798) cannot be retried; you must create a new workflow run.

Action Works on runs > 30 days old?
gh run rerun <run-id> No
Re-run all jobs in UI No
gh workflow run / Run workflow / git push Yes — creates a new run

4. Start a new deploy (recommended alternatives)

4.1 GitHub UI — Run workflow

▶ Manual dispatch (if enabled in YAML)
  1. Open https://github.com/giventadeveloporg/malayalees-us-site-boot/actions
  2. Left sidebar → click deploy-dev.yml (or the workflow display name).
  3. If Run workflow appears on the right → choose branch (for example release/v1.0.0) → Run workflow.

This requires workflow_dispatch: in the workflow file:

on:
  push:
    branches: [...]
  workflow_dispatch:

4.2 GitHub CLI — gh workflow run

Trigger deploy (GitHub CLI)
gh workflow run deploy-dev.yml --repo giventadeveloporg/malayalees-us-site-boot --ref release/v1.0.0
Windows single-line:
gh workflow run deploy-dev.yml --repo giventadeveloporg/malayalees-us-site-boot --ref release/v1.0.0

List recent runs after dispatch:

List recent workflow runs
gh run list --repo giventadeveloporg/malayalees-us-site-boot --workflow="deploy-dev.yml" --limit 5
Windows single-line:
gh run list --repo giventadeveloporg/malayalees-us-site-boot --workflow="deploy-dev.yml" --limit 5

View a specific run (replace RUN_ID):

View one workflow and run
gh run view 25995383556 --repo giventadeveloporg/malayalees-us-site-boot
Windows single-line:
gh run view 25995383556 --repo giventadeveloporg/malayalees-us-site-boot

4.3 Empty commit (when workflow runs on push)

Empty commit to trigger push workflow
git commit --allow-empty -m "chore: trigger deploy"
Windows single-line:
git commit --allow-empty -m "chore: trigger deploy"

Then push the branch your workflow listens to (for example release/v1.0.0 or main):

Push to remote
git push
Windows single-line:
git push

6. Before you trigger a deploy

AWS credentials in GitHub Environments: Store AWS_ACCESS_KEY_ID and AWS_SECRET_ACCESS_KEY under Environment secrets for the dev environment—not under Environment variables. Workflows that use ${{ secrets.AWS_ACCESS_KEY_ID }} will not read variable values correctly, which can cause The security token included in the request is invalid in the configure-aws-credentials step.
  • Confirm the correct repository and workflow file (deploy-dev.yml).
  • Align AWS_REGION with the region your ECS/ECR resources use.
  • For failed deploy debugging, open the latest run—not a months-old successful run.
