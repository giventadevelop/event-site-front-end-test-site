Gallery Album Cover S3 — Data Migration & Porting PRD

Version: 1.0  |  Date: June 18, 2026
Index: generic_enhancement_prd.html  |  Porting guide: GALLERY_STATIC_TO_DYNAMIC_PORTING_GUIDE.html

Static gallery import currently sets coverImageUrl to paths like /images/mosc/gallery/{slug}/cover.jpg. This PRD defines how to upload those files to S3 and update albums after import (or as part of a new CLI phase).

1. Current import behavior

ModeCoverPhotos
--media-mode url (default)Local path in cover_image_urlfileUrl = local /images/...
--media-mode fileStill URL from manifest unless extendedMultipart upload to S3 via Next proxy

2. Target: cover upload phase

Add to scripts/gallery-porting/import-static-gallery-to-api.mjs (or sibling script upload-gallery-album-covers.mjs):

FlagDescription
--upload-coversAfter album exists, read cover file from disk and POST to new cover upload endpoint
--covers-onlySkip album/photo import; only upload covers for albums matched by static_slug=
--dry-runLog S3 path and file existence only

3. Recommended porting order (full S3 migration)

  1. npm run gallery:extract-manifest
  2. npm run gallery:seed-categories (if needed)
  3. npm run gallery:import — creates albums + URL-mode media (local paths OK for dev)
  4. NEW: npm run gallery:upload-covers — requires backend C1 endpoint + local files under public/images/mosc/gallery/
  5. Optional: re-run npm run gallery:import:media --media-mode file to move all photos to S3 (separate from covers)
Proposed npm script
npm run gallery:upload-covers
// package.json
"gallery:upload-covers": "node scripts/gallery-porting/upload-gallery-album-covers.mjs",
"gallery:upload-covers:dry": "node scripts/gallery-porting/upload-gallery-album-covers.mjs --dry-run"

4. Script algorithm

  1. Load manifest; for each album with coverImageUrl or coverLocalPath.
  2. Resolve album id via GET /api/gallery-albums + parse static_slug= from description.
  3. Map manifest path to filesystem: public/images/mosc/gallery/{slug}/{filename}.
  4. If file missing, log warning and skip.
  5. POST multipart to backend cover upload endpoint (direct backend URL + service JWT, same as import script — not Next proxy).
  6. Verify response fileUrl starts with https:// and album PATCH reflects new URL.
  7. Write cover-upload-report.json.

5. Backfill existing imported albums

For albums already in DB with local cover_image_url like /images/mosc/gallery/...:

  • Run gallery:upload-covers --covers-only against dev/staging/prod API.
  • No SQL migration — only PATCH via upload endpoint.
Production run requires AWS credentials on backend and public/images/mosc/gallery/ files available on the machine running the script (or fetch from deployed static assets first).
