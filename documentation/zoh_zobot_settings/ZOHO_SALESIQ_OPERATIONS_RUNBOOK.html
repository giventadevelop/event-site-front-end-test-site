Zoho SalesIQ Operations Runbook

OAuth refresh token · bulk upload (REST API) · bulk delete · FAQ supplement CSV · FAQ cleanup · merge · Answer Bot testing

This runbook documents the repeatable operational workflow used to reset and reload the Giventa Inc. SalesIQ knowledge base for MOSC (giventainc portal). Recommended import path: post all articles from salesiq-articles-import.csv using npm run zobot:bulk-upload-articles (SalesIQ Articles REST API). Portal CSV import remains documented as an alternative. This runbook complements the architecture guide: Zoho SalesIQ Zobot — MOSC Redesign Knowledge Base Guide and the export script guide: SalesIQ Articles Export — Node Script.

1. End-to-end workflow (summary)

1. Run event-site-manager locally (OAuth host) 2. Open http://localhost:3000/oauth/zoho/start → Zoho consent (scopes incl. CREATE) → copy ZOHO_REFRESH_TOKEN 3. Save token + client id/secret + SalesIQ IDs in .env.local (mosc-temp and event-site-manager) 4. (Optional reset) npm run zobot:bulk-delete-articles -- --dry-run → --confirm 5. (Optional) Re-export: npm run zobot:export-articles 6. npm run zobot:fix-import-titles → npm run zobot:patch-import-csv → edit salesiq-faq-supplement.csv 7. npm run zobot:merge-faq-supplement 8. npm run zobot:list-portal-ids (verify category / department IDs) 9. npm run zobot:bulk-upload-articles -- --dry-run → --confirm (posts all CSV rows via API) 10. npm run zobot:bulk-delete-faqs -- --dry-run → delete SalesIQ defaults → --confirm 11. npm run zobot:bulk-delete-faqs -- --dry-run (list all) → delete any remaining Zoho stragglers 12. npm run zobot:bulk-upload-faqs -- --dry-run → --confirm (7 MOSC curated FAQs) 13. Re-train / refresh Answer Bot; verify widget (dashboard + View All + Home strip) 14. If Home FAQ preview is stale but View All is correct → section 11b ($zoho.salesiq.reset() or incognito)

3. Obtain ZOHO_REFRESH_TOKEN (event-site-manager)

Step 3a — Register redirect URIs in Zoho API Console

In Zoho API Console, open your client and add these Authorized Redirect URIs (exact match required):

https://www.event-site-manager.com/oauth/zoho/callback
http://localhost:3000/oauth/zoho/callback
Localhost rule: When you visit OAuth from localhost, the callback route automatically uses http://localhost:3000/oauth/zoho/callback — not the production URI in ZOHO_OAUTH_REDIRECT_URI. This prevents the “callback hit production without secrets → 500” failure.
Step 3b — Configure event-site-manager .env.local

Copy from event-site-manager/.env.example and fill in:

ZOHO_CLIENT_ID=your_client_id
ZOHO_CLIENT_SECRET=your_client_secret
ZOHO_OAUTH_REDIRECT_URI=https://www.event-site-manager.com/oauth/zoho/callback
ZOHO_OAUTH_SCOPES=SalesIQ.articles.READ,SalesIQ.articles.CREATE,SalesIQ.articles.DELETE,SalesIQ.faqs.READ,SalesIQ.faqs.CREATE,SalesIQ.faqs.DELETE,SalesIQ.departments.READ
ZOHO_SALESIQ_SCREEN_NAME=giventainc
# Optional FAQ category (Resources → FAQs, not Articles):
# ZOHO_SALESIQ_FAQ_CATEGORY_ID=...
# Optional: ZOHO_OAUTH_SETUP_SECRET=your-secret

ZOHO_OAUTH_SCOPES must include SalesIQ.articles.CREATE for bulk article upload and SalesIQ.faqs.CREATE / SalesIQ.faqs.DELETE for FAQ scripts. After changing scopes, re-run OAuth (section 3) and replace ZOHO_REFRESH_TOKEN.

Do not commit .env.local to git.

Step 3c — Start event-site-manager (not mosc-temp)
PowerShell — run OAuth host
Windows single-line
cd F:\project_workspace\event-site-manager; npm run dev

Wait until Next.js is ready (typically http://localhost:3000).

Step 3d — Complete OAuth in browser
  1. Open http://localhost:3000/oauth/zoho/start (add ?key=... if ZOHO_OAUTH_SETUP_SECRET is set).
  2. Sign in to Zoho and approve access (access_type=offline, prompt=consent ensures a refresh token).
  3. After redirect, the success page shows ZOHO_REFRESH_TOKEN=... — copy the full value immediately.
Production OAuth: You can also use https://www.event-site-manager.com/oauth/zoho/start once ZOHO_CLIENT_ID and ZOHO_CLIENT_SECRET are set in AWS Amplify for that app. Localhost is preferred for first-time setup.
Step 3e — CLI fallback (if browser callback failed)

If you have a code= in the URL but the page errored, exchange it manually from mosc-temp:

Exchange authorization code
Windows single-line
cd F:\project_workspace\mosc-temp; npm run zobot:exchange-zoho-code -- --redirect-uri "http://localhost:3000/oauth/zoho/callback" --code "PASTE_CODE_HERE"

The --redirect-uri must match exactly what was used when the code was issued.

10. Alternative: portal CSV import

Portal steps (if you cannot use API CREATE scope)

Use this only when OAuth cannot include SalesIQ.articles.CREATE. Detailed mapping tables live in the main guide — Section 7.

  1. Sign in to salesiq.zoho.com/giventainc.
  2. Create article Categories matching CSV Category column before import.
  3. Go to Settings → Articles (Settings sidebar) → Add → Import.

11. FAQ cleanup and bulk upload (widget FAQs)

Articles ≠ FAQs. The ~272 rows uploaded via zobot:bulk-upload-articles are Articles (Resources → Articles). The chat widget Home → FAQs strip is a separate resource type (Resources → FAQs).

New brand / Zobot startup: When you configure a new SalesIQ brand or enable the chat widget on a site like MOSC, Zoho often pre-seeds 10 default marketing FAQs (SalesIQ product questions). These appear in the widget even after you upload Articles. You must delete the default FAQs and upload your own from salesiq-faq-supplement.csv — the same workflow documented in section 11a (verified commands).

The curated MOSC Q&A in salesiq-faq-supplement.csv is also merged into the articles CSV for Answer Bot — that merge does not replace widget FAQs.

11b. Widget Home FAQ strip stale (View All is correct) — clear cache

After bulk FAQ delete/upload, the SalesIQ dashboard and the widget View All FAQ screen usually show the new MOSC content immediately. The Home tab FAQ preview strip (the short list shown when you first open chat) can still show old buffered SalesIQ questions until the visitor session cache is cleared. This is expected Zoho widget behavior — not a sign that zobot:bulk-upload-faqs failed.

How to tell upload succeeded vs cache lag

Where you lookExpected after uploadIf wrong here
SalesIQ dashboard → Resources → FAQs7 MOSC FAQs onlyRe-run section 11a (delete + upload), not a cache issue
Widget → FAQ block → View All7 MOSC FAQs onlyUsually still server-side; check dashboard first
Widget → Home tab → FAQ preview stripMay still show old SalesIQ textVisitor cache — use fixes below (verified June 2026)
Why View All updates but Home does not: The Home preview is bundled with the initial widget bootstrap / visitor session. View All loads the live FAQ list from Zoho’s API when you open that screen. Two surfaces, two load paths — stale Home + fresh View All means your portal data is correct.

Fix 1 — Incognito / private window (quick test)

  1. Open your site (e.g. https://www.mosc-temp.com) in a private/incognito window.
  2. Open the chat widget → Home → FAQ preview.
  3. If MOSC FAQs appear here, the issue is only your normal browser session cache.

Fix 2 — Clear SalesIQ cookies (browser)

  1. Open DevTools → Application → Cookies.
  2. Clear cookies for your site (e.g. www.mosc-temp.com) and Zoho widget hosts (e.g. salesiq.zohopublic.com, zoho.com) — look for names starting with zld, zsiq, etc.
  3. Hard refresh: Ctrl+Shift+R (Windows) or Cmd+Shift+R (Mac).
  4. Reopen the widget and check the Home FAQ strip.

Fix 3 — $zoho.salesiq.reset() (verified — fastest forced refresh)

Official SalesIQ JS API: clears cookies, reloads the widget iframe, and starts a new visitor session. Run in the browser console while the page with the widget is open:

$zoho.salesiq.ready(function () {
  $zoho.salesiq.reset();
});

Reference: Zoho SalesIQ — .salesiq.reset(). Verified June 2026: After FAQ bulk upload, this cleared the stale Home preview and showed MOSC FAQs without re-running upload scripts.

Do not call reset() on every page load in site code — it ends active chats and creates a new visitor each time. Use only for testing or manual recovery.

  2. Open Installation or Personalization, review widget settings, click Save / Publish if offered.
  3. Hard-refresh the live site and retest the Home FAQ strip.

Production visitors — do you have to wait?

  • For you (testing): No long wait — incognito, cookie clear, or reset() should fix it immediately.
  • For returning visitors: They may see the old Home preview until their widget session expires or they get a fresh embed load. Some SalesIQ FAQ-related cookies can persist up to ~24 hours; most users see updates sooner on a normal revisit.
  • New visitors typically see current FAQs once Zoho’s edge cache has propagated (often minutes, sometimes a few hours).
What this is not: Re-running zobot:bulk-upload-faqs, Answer Bot re-training, or article upload. Dashboard + View All already prove the FAQ API data is correct.

11d. Skip / suppress pre-chat name, email, phone

When a visitor starts a new chat, SalesIQ may show a pre-chat form (name, email, phone) before the Zobot / Answer Bot conversation. To let visitors ask questions immediately (especially for bot-only flows), change brand flow settings and/or hide fields in the embed code.

Option A — Dashboard: Flow Control (recommended first)

  1. SalesIQ → Settings → Brands → select your MOSC brand (e.g. www.mosc-temp.com).
  2. Open Flow Control (or Flow / chat window flow — label varies by SalesIQ version).
  3. Form style: choose Conversation or Inline instead of Traditional — visitors can start chatting first; contact fields can be collected later in the thread (see Flow Control setup).
  4. Or remove / disable fields you do not need: for each of Name, Email, Phone, uncheck Mark this field as mandatory or disable the field if the UI allows.
  5. Save / publish the brand widget settings.
Bot-only sites: For a homepage assistant that only uses Answer Bot (no live agent queue), Conversation/Inline style avoids blocking the first question behind email/phone.

Option B — Embed code: hide default fields (website)

Zoho’s JavaScript API removes the default pre-chat fields from the visitor chat window. Add inside your existing $zoho.salesiq.ready handler (after the widget script loads):

$zoho.salesiq.ready(function () {
  $zoho.salesiq.field.clear(['siq_name', 'siq_email', 'siq_phonenumber']);
});

Reference: .field.clear() — SalesIQ JS API.

In this repo, the embed lives in src/components/mosc-redesign/ZohoSalesIqWidget.tsx — merge the field.clear call into the existing SALESIQ_INIT_SCRIPT ready callback if dashboard settings alone are not enough.

Trade-off: Without email/phone you cannot identify visitors for follow-up or CRM sync until they volunteer details or escalate to an agent. Use Flow Control to keep email optional (not mandatory) rather than removing it entirely if you still want lead capture on handoff.

Option C — Zobot flow: avoid Visitor Info cards before Answer Bot

In the Codeless Zobot canvas, do not place a Visitor Info or Get User Input card for email/phone before the Answer bot Card. Flow should be: Welcome → Answer bot Card (see Knowledge Base Guide section 9c).

Verify

  1. Open the site in incognito (or run $zoho.salesiq.reset() — section 11b).
  2. Open chat → you should reach the bot greeting / question input without mandatory email/phone.
  3. If fields still appear, confirm you edited the correct brand and saved Flow Control; then add Option B to the embed.
