Liturgical Calendar API – Integration Guide

This document describes the liturgical calendar API used for daily readings in English and Malayalam, how the current frontend invokes and displays it, and how to implement it in a Next.js project.

Example Request

Response Shape

The API returns a JSON object with a message property that is an array of reading objects. Each object (and the first item in particular) carries day/season info and per-reading details.

FieldWhereDescription
messageRootArray of reading objects
liturgy_day_headingFirst item (typically)Heading for the day (e.g. date or day name)
season_nameFirst item (typically)Name of the liturgical season
liturgy_headingEach itemTitle of the reading
content_placeEach itemReference/location (e.g. book and chapter)

Note: The token (__) is required. It is currently embedded in client-side code; for production, consider obtaining a dedicated key from the API owner (SMCIM) and, if possible, using it from a server-only layer.

2. Current Frontend (How It’s Invoked & Displayed)

Where the API is configured

  • Base URL: Set in the page via a <constants> element: data-common-api="https://www.apiwebser.smcimprojects.com/api/" (see index.html).
  • Endpoint path: In assets/scripts/common.js, apiEndpoints.liturgy is "liturgy".

How it’s invoked

  • Script: assets/scripts/home.js.
  • Function: fetchLiturgy(language) with language defaulting to 'en'.
  • Request: $.get(commonApiURL + apiEndpoints.liturgy, parameters, callback) with:
    • parameters.__ = "dmdlMXBVWkNqcS95MkFDVmlEWExZQT09"
    • parameters.lng = language ('en' or 'ml')
  • Language switch: Buttons with class liturgy-button and data-id="en" or data-id="ml" call fetchLiturgy(selectedLanguage) on click.

Where and how it’s displayed

  • Container: The response is rendered inside an element with id liturgy_readings (#liturgy_readings).
  • Loading: Before the request, the container is filled with a loader (spinner) HTML.
  • Empty response: If readings.length === 0, the function returns without updating the container.
  • Structure built in the frontend:
    • From the first item: liturgy_day_heading in a <p>, season_name in an <h2>.
    • A <ul class="fa-ul"> with one <li> per reading: Font Awesome list icon (fa-li fa-solid fa-cross), liturgy_heading, and content_place in a <span>.

HTML structure the current frontend produces

<p>{liturgy_day_heading}</p>
<h2>{season_name}</h2>
<ul class="fa-ul">
  <li><i class="fa-li fa-solid fa-cross"></i>{liturgy_heading}<span> ( {content_place} )</span></li>
  <!-- one li per reading -->
</ul>

Section context in the page

The liturgy block lives inside a section with class liturgy, with a heading “Liturgical Calendar” and a main title “Liturgical Calendar”. The dynamic content is intended to go into #liturgy_readings; the existing site may also show a static image in .liturgy-image alongside or above it.

3. Next.js Implementation Guide

3.1 Data fetching

You can fetch in two ways:

  • Client-side: Use fetch or a data library (e.g. SWR, React Query) inside a Client Component, with lng from state or search params. Same URL and parameters as above.
  • Server-side: In a Server Component or in getServerSideProps / route handler, call the API with the same URL and query string. Prefer keeping the __ token in environment variables (e.g. LITURGY_API_TOKEN) and not in client bundle.

3.2 Environment variable (recommended)

# .env.local
LITURGY_API_TOKEN=dmdlMXBVWkNqcS95MkFDVmlEWExZQT09

Use process.env.LITURGY_API_TOKEN only on the server; for client-side fetch, use a Next.js API route that forwards the request with the token.

3.3 TypeScript types (example)

type LiturgyReading = {
  liturgy_day_heading?: string;
  season_name?: string;
  liturgy_heading: string;
  content_place: string;
};

type LiturgyApiResponse = {
  message: LiturgyReading[];
};

3.4 Fetch helper (server or client)

const LITURGY_API_BASE = 'https://www.apiwebser.smcimprojects.com/api';

async function fetchLiturgy(lng: 'en' | 'ml', token?: string) {
  const params = new URLSearchParams({
    __: token ?? process.env.LITURGY_API_TOKEN ?? '',
    lng,
  });
  const res = await fetch(`${LITURGY_API_BASE}/liturgy?${params}`);
  if (!res.ok) throw new Error('Liturgy API error');
  const data: LiturgyApiResponse = await res.json();
  return data.message;
}

3.5 React component (display)

Structure the UI to match the current frontend behavior: day heading, season name, then list of readings with optional icon and reference.

interface LiturgyReadingsProps {
  readings: LiturgyReading[];
}

function LiturgyReadings({ readings }: LiturgyReadingsProps) {
  if (!readings?.length) return null;
  const first = readings[0];
  return (
    <section className="liturgy" aria-label="Liturgical Calendar">
      <div className="container">
        <h6 className="section-title">Liturgical Calendar</h6>
        <h2><span className="imp-color">Liturgical</span> <span>Calendar</span></h2>
        <p>{first.liturgy_day_heading}</p>
        <h2>{first.season_name}</h2>
        <ul className="fa-ul">
          {readings.map((r, i) => (
            <li key={i}>
              <i className="fa-li fa-solid fa-cross" aria-hidden />
              {r.liturgy_heading}
              <span> ( {r.content_place} )</span>
            </li>
          ))}
        </ul>
      </div>
    </section>
  );
}

3.6 Language toggle

Keep language in state or in URL (e.g. ?lng=ml). When the user switches between English and Malayalam, set lng to en or ml and refetch (or pass to a server component). Use the same lng value in the API request.

3.7 Loading and error states

While loading, show a spinner or skeleton inside the liturgy section. On error (e.g. !res.ok or empty message), show a friendly message or hide the block, consistent with the current frontend’s early return when readings.length === 0.
