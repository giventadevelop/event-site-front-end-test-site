Liturgical Calendar – Project Requirements

Generic product and feature requirements for the liturgical calendar, independent of backend implementation (external SMCIM API or self-hosted Strapi). Covers scope, functional requirements, data model, bilingual support (English and Malayalam), tenant scoping, and non-functional requirements.

1. Scope and Objectives

Provide a Liturgical Calendar section on the MOSC / Syro-Malabar website that displays daily liturgical readings for the current date, in both English and Malayalam.

  • Show the day heading (date/day name), the liturgical season, and a list of readings with their scripture references.
  • Allow the user to toggle between English and Malayalam.
  • Support multi-tenant deployments where each tenant (e.g. a parish or diocese site) may have its own set of readings or share a common calendar.
  • Be backend-agnostic: the feature can be served by an external API (SMCIM) or by a self-hosted Strapi CMS.

2. Functional Requirements

2.1 Display

  1. Day heading – A human-readable string for the current day (e.g. “Thursday, 27 February 2026”).
  2. Season name – The name of the current liturgical season (e.g. “Great Lent”).
  3. Readings list – One or more readings, each with:
    • Liturgy heading – Title of the reading (e.g. “First Reading”, “Gospel”).
    • Content place – Scripture reference (e.g. “Genesis 1:1-5”).
  4. A decorative image or icon may accompany the section.

2.2 Language Selection

  1. Two languages: English (en) and Malayalam (ml).
  2. A toggle (two buttons) allows the user to switch languages without a page reload.
  3. The currently selected language is visually highlighted.
  4. When the language changes, the readings refresh to show the corresponding language content.

2.3 Data Source

  1. The readings are fetched from a backend API (external or Strapi).
  2. The frontend calls the API with a lng (or lang) parameter set to en or ml.
  3. The API returns data for the current date by default (or accepts a date parameter for a specific day).

2.4 Loading & Error States

  1. Loading: Show a spinner or skeleton while the API request is in progress.
  2. Error / Unavailable: Show a friendly message (e.g. “Daily readings are temporarily unavailable”) with a link to the official liturgy site (Syro-Malabar Commission for Liturgy).
  3. Empty: If the API returns no readings for the day, show “No readings available for this day.”

3. Data Model (Logical)

Regardless of backend, the data follows this logical structure:

3.1 Liturgy Day

FieldTypeDescription
dateDateCalendar date (YYYY-MM-DD)
day_headingStringHuman-readable day heading (language-specific)
season_nameStringLiturgical season name (language-specific)
readingsArrayOne or more readings (see below)

3.2 Reading

FieldTypeDescription
liturgy_headingStringTitle of the reading (e.g. “First Reading”) – language-specific
content_placeStringScripture reference (e.g. “Genesis 1:1-5”) – language-specific

3.3 Current API Field Names

The existing integration (external SMCIM API) uses these field names in the JSON response:

Field NameMaps To
liturgy_day_headingDay heading
season_nameSeason name
liturgy_headingReading title
content_placeScripture reference

The frontend expects a response shaped as { message: [ { liturgy_day_heading, season_name, liturgy_heading, content_place }, ... ] }. Any new backend (including Strapi) should either return this shape directly or be mapped to it by a proxy layer.

4. Bilingual Support (English & Malayalam)

  • Each field (day heading, season name, liturgy heading, content place) has an English and Malayalam variant.
  • The API (or proxy) returns only the requested language’s content based on the lng parameter.
  • In a Strapi implementation, both language variants are stored in a single record (e.g. dayHeadingEn / dayHeadingMl), and the proxy or frontend selects the appropriate fields based on the requested language.
  • Language codes: en = English, ml = Malayalam.

5. Tenant Scoping

  • In a multi-tenant deployment (e.g. multiple parishes sharing a Strapi instance), each liturgy day record is associated with a tenant.
  • The frontend passes the tenant identifier (e.g. NEXT_PUBLIC_TENANT_ID) when fetching readings, so only data for the current tenant is returned.
  • Tenant assignment in Strapi follows the existing pattern: lifecycle hooks auto-assign the editor’s tenant on create; the content-manager extension filters the list by tenant so editors only see their own data.
  • Tenants may share a common calendar or maintain separate ones. The data model supports both via the tenant relation.

6. Non-Functional Requirements

RequirementDetail
AvailabilityGraceful degradation: if the API is unavailable, show a friendly message with a link to the official liturgy site. Do not crash the page.
PerformanceCache API responses (e.g. 1-hour revalidation on the server side). Readings change daily, so a short cache is acceptable.
AccessibilitySection has aria-label="Liturgical Calendar". Language toggle buttons use aria-pressed and aria-label. Loading state uses aria-live="polite".
SecurityAPI tokens (if any) must not be exposed to the client. Use a server-side proxy (Next.js API route) to inject tokens and tenant IDs.
ResponsivenessThe liturgy section and language toggle must work on mobile devices.
