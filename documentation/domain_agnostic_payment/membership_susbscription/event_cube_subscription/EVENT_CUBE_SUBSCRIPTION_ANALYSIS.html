1. Executive Summary

This document analyzes Event Cube as a separate path for membership and subscription management in the MCEFEE application. The goal is to detach membership from the event-centric Stripe integration and instead use Event Cube as a managed membership platform that is associated with organizations and domains, not with individual events.

Key Concept

Event Cube provides an all-in-one platform for events, memberships, and recurring payments. Rather than building our own billing logic with Stripe, we can externalize membership management to Event Cube while keeping event ticketing on our current path. Membership becomes organization-scoped and domain-aware.

3. Event Cube Membership Features

Event Cube offers a built-in membership system that supports:

  • Create and manage membership plans – Multiple tiers, seasons, pricing levels
  • Recurring payments – Direct debit, monthly/annual subscriptions
  • Member status and renewals – Automatic tracking of expiration, renewal status, access levels
  • Member portal – Members can view and manage their membership online
  • Event integration – Member-only content, exclusive events, discounted tickets
  • White-label – Custom domain/SSL, remove Event Cube branding (Pro plan)
  • API – Events, tickets, basket, orders (Pro plan)
  • Zapier – Integration with other tools

Note: Event Cube uses Stripe or similar processors for payments, but wraps them into a higher-level membership workflow so you don't build your own billing logic.

Reference: https://www.eventcube.io/memberships

5. Domain-Agnostic Membership Model

Core Concept

Membership is tied to the organization (tenant), not to events. Each organization can have its own Event Cube account and membership plans.

Key Entities

  • tenant_organization – tenant_id, domain, organization_name, etc.
  • Domain/subdomain mapping – domain or subdomain → tenant_id → organization
  • Externalized membership – Event Cube account per organization (or per tenant)
  • User profile – user_profile + tenant_id → member in Event Cube for that org

Domain-to-Membership Association

Domain/Subdomain → tenant_id → tenant_organization → Event Cube Account ID
                                                    → membership_plan (sync)
                                                    → membership_subscription (sync)

6. Externalization Strategy

What to Externalize to Event Cube

  • Membership plans – Created in Event Cube, optionally synced to local membership_plan for display
  • Subscriptions – Primary source of truth in Event Cube; webhooks or API sync to local membership_subscription
  • Member portal – Event Cube hosted or embed (custom domain on Pro)

Per-Domain Configuration

  • tenant_organization.eventcube_account_id (or similar) – maps org to Event Cube account
  • Environment: NEXT_PUBLIC_TENANT_ID per deployment; subdomain/domain resolves to tenant
  • Each domain/subdomain deployment uses its tenant's Event Cube config

User Management Across Domains

When a user visits a subdomain (e.g., org1.mosc.com), the app resolves tenant_id from the domain. Membership lookup uses tenant_id + user_profile to fetch subscription status from Event Cube (or local cache). The user can manage membership for that organization in the Event Cube portal or via embedded UI.

7. Separate Path Feature Implementation Outline

Path A – Stripe Direct (Current)

  • Routes: /membership, /membership/plans, /membership/subscribe/[planId], /membership/manage
  • Uses Stripe Elements, Payment Intents, Subscriptions
  • Backend: membership_plan, membership_subscription, Stripe webhooks

Path B – Event Cube (New, Parallel)

  • /membership/eventcube or feature flag to route to Event Cube flow
  • Redirect to Event Cube checkout/portal, or embed Event Cube UI
  • Sync subscription status via Event Cube webhooks/API
  • Local cache: membership_subscription with source = 'EVENTCUBE' and eventcube_subscription_id

Coexistence

  • Tenant setting: membership_provider: 'STRIPE' | 'EVENTCUBE'
  • Routing logic selects Path A or Path B based on tenant config

8. Event Cube as Paid Managed Service

What Event Cube Charges

  • Monthly SaaS fee – $0 (Starter), $99–$125 (Pro), Custom (Enterprise)
  • Transaction fee – 5% (Starter), 3% (Pro), bespoke (Enterprise)

What You Avoid Building

  • Recurring billing logic
  • Member portal
  • Renewal tracking
  • Direct debit handling

What You Still Need

  • Auth (Clerk)
  • User profile
  • Tenant/org mapping
  • Optional local cache of subscription status for feature gating

Cost Comparison

Stripe direct: Development cost + infra cost + maintenance. Event Cube: Subscription fee + transaction % + lower dev cost. Choose based on scale, dev resources, and desired features.

9. When Event Cube Makes Sense

  • All-in-one platform – Events + memberships + recurring payments under one roof
  • Branded member portal – Without manually building your own system
  • Membership includes event perks – Integrated event ticketing (clubs, associations, organizations)

Best Fit

Clubs, associations, and organizations where membership offers event perks or integrated ticketing. Event Cube combines event and member-centric activity into a single platform.
