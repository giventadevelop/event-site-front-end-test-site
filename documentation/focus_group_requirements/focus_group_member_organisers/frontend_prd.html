PRD: Focus Group – Member Join & Organisers (Frontend)

Project: mosc-temp (Next.js 15+ App Router)
Document Version: 1.0
Date: March 2026
Status: Draft

Scope

This PRD specifies the frontend behaviour for end-user join focus group and executive/organising committee list. Do not build components in the initial docs phase; this document is the handoff for the frontend team to implement when the backend supports self-service join and committee filtering.

End-User Join

  • Where: Public focus group page: src/app/focus-groups/[slug]/page.tsx.
  • When user is logged in: Show a “Join” button if the current user is not already a member of this focus group. On success, show “Leave” (or “Requested” if status is PENDING).
  • When user is not logged in: Do not show Join; optionally show “Sign in to join” or link to sign-in.
  • Implementation: Use auth (e.g. Clerk) to get the current user; resolve user_profile_id via existing profile lookup (e.g. by userId). Call backend via server action or proxy: POST to create a focus_group_members row with focusGroupId, tenantId, and user_profile_id from the server/session. For “Leave”, call DELETE (or backend leave endpoint) for the current user’s membership row.

Executive / Organising Committee List

  • Where: Same public focus group page: src/app/focus-groups/[slug]/page.tsx.
  • What: A section “Executive / Organising Committee” that lists members whose role is EXECUTIVE or ORGANISER (and optionally LEAD, ADMIN) for this focus group.
  • Data: GET /api/proxy/focus-group-members with filters e.g. focusGroupId.equals=<id> and role.in=EXECUTIVE,ORGANISER (or equivalent per backend API). If the backend exposes a dedicated endpoint e.g. GET /api/focus-groups/{id}/organisers, use that. Resolve user names/photos from user_profile if the API returns only userProfileId (e.g. expand in server action or use nested DTO from backend).

API / Proxy

  • Use existing proxy: /api/proxy/focus-group-members (GET list, POST create, DELETE by id).
  • Follow nextjs_api_routes.mdc: all authenticated backend calls from server components or server actions (e.g. ApiServerActions.ts); use fetchWithJwtRetry from @/lib/proxyHandler for direct backend calls. Client components must not call the backend API URL directly; use server actions for mutations.
  • Do not add tenantId.equals when calling proxy endpoints; the proxy injects tenant.

UI Standards

  • Follow ui_style_guide.mdc: page container max-w-5xl mx-auto px-8 py-8, content cards bg-white rounded-lg shadow-md p-6, and existing button/form patterns.
No build in docs phase: This PRD is for the frontend team to implement when the backend supports self-service join and committee filtering. No components are built in the initial schema/docs phase.
