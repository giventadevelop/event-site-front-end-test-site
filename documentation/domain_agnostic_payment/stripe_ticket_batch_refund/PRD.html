Executive Summary

This document defines a batch refund feature for Stripe-based ticket transactions. The feature allows administrators to refund all eligible tickets for an event in a single operation via an on-demand batch job. This is particularly useful when events are canceled or require bulk refund processing. The feature follows the same pattern as the existing batch job trigger in the sales analytics page and extends the single ticket refund functionality to handle multiple transactions efficiently.

Key Concept: Batch refunds are processed asynchronously via an on-demand batch job. The system returns a job ID immediately and processes refunds in the background, allowing administrators to continue working while refunds are being processed.

Currently, administrators can refund individual tickets through the ticket list page using a single refund button per transaction. This process requires manual intervention for each ticket, which is time-consuming when events are canceled and all tickets need to be refunded. The batch refund feature automates this process by allowing administrators to trigger a single batch job that processes all eligible tickets for an event.

The feature follows the same architectural pattern as the existing Stripe Fees and Tax Update batch job, which is triggered from the sales analytics page. This ensures consistency in user experience and code maintainability.

2. Goals & Non-Goals

Goals

  • Enable administrators to refund all eligible tickets for an event in a single operation.
  • Process refunds asynchronously via an on-demand batch job to avoid UI blocking.
  • Provide real-time feedback on job status, progress, and completion.
  • Handle partial refund scenarios gracefully (continue processing if some refunds fail).
  • Maintain consistency with existing single refund flow and batch job patterns.
  • Ensure proper audit logging and tenant isolation for all refund operations.

Non-Goals

  • Scheduled/automatic batch refunds (this is an on-demand feature only).
  • Refunding manual payments (Zelle, Venmo, etc.) - only Stripe payments are supported.
  • Partial refunds for individual transactions (full refund only per ticket).
  • Refunding tickets from multiple events in a single batch operation.
  • Refund reason selection for batch operations (uses default reason: "Event canceled - Batch refund").

3. Scope & Eligibility

Eligible Tickets

Only tickets meeting all of the following criteria are eligible for batch refund:

  • Event ID: Must match the event ID from the ticket list page context.
  • Tenant ID: Must match the current tenant (enforced by backend).
  • Stripe Payment: Must have a stripePaymentIntentId (Stripe payments only).
  • Status: Must not already be refunded (status != 'REFUNDED').
  • Payment Status: Must have a completed Stripe payment status.

Excluded Tickets

  • Tickets with manual payments (Zelle, Venmo, Cash App, etc.).
  • Tickets that are already refunded.
  • Tickets with pending or failed payment status.
  • Tickets from different events or tenants.

Optional Filters

  • Date Range: Optional filter by purchase date range (future enhancement).
  • Status Filter: Optional filter by transaction status (future enhancement).

4. User Journeys

Administrator Workflow

  1. Navigate to ticket list page: /admin/events/{eventId}/tickets/list
  2. Review ticket statistics and confirm event details.
  3. Locate the "Batch Refund" section (collapsible, similar to sales analytics page).
  4. Click "Show Batch Refund Options" to expand the section.
  5. Review the number of eligible tickets that will be refunded.
  6. Click "Trigger Batch Refund" button.
  7. Confirm the action in the confirmation dialog (shows ticket count and warning).
  8. System returns job ID and estimated completion time immediately.
  9. Monitor job status via success message (job ID, status, estimated records).
  10. Refresh ticket list to see updated refund status after job completion.

System Processing Flow

  1. Frontend validates eventId and prepares request payload.
  2. Frontend calls server action: triggerStripeTicketBatchRefundServer().
  3. Server action calls backend endpoint: POST /api/cron/stripe-ticket-batch-refund.
  4. Backend validates request and returns job ID (HTTP 202 Accepted).
  5. Backend triggers batch job service asynchronously.
  6. Batch job queries eligible tickets from database.
  7. Batch job processes refunds in batches (100-500 tickets per batch).
  8. For each ticket: Call Stripe refund API, update database record.
  9. Batch job logs success/failure for each transaction.
  10. Batch job updates job status with progress and completion summary.

5. Data Model & Schema

Primary Table: event_ticket_transaction

The batch refund feature operates on the existing event_ticket_transaction table. No schema changes are required, but the following fields are critical:

Field Type Purpose
id Integer Unique transaction identifier
event_id Integer Event identifier for filtering
tenant_id String Tenant isolation and filtering
stripe_payment_intent_id String Required for Stripe refund API call
status String Transaction status (COMPLETED, REFUNDED, PENDING, etc.)
final_amount Decimal Amount to refund (in dollars)
refund_amount Decimal Refunded amount (updated after refund)
refund_date Timestamp Timestamp when refund was processed
refund_reason String Reason for refund (default: "Event canceled - Batch refund")
stripe_payment_status String Stripe payment status (updated to 'refunded' after refund)

Query Criteria

SELECT * FROM event_ticket_transaction
WHERE event_id = ?
  AND tenant_id = ?
  AND stripe_payment_intent_id IS NOT NULL
  AND status != 'REFUNDED'
  AND stripe_payment_status IN ('succeeded', 'paid')
ORDER BY created_at ASC;

6. Integration Points

Stripe API Integration

  • Endpoint: stripe.refunds.create()
  • Parameters: payment_intent (from stripePaymentIntentId), reason (optional)
  • Response: Refund object with id, amount, status
  • Error Handling: Handle rate limits, invalid payment intents, already refunded payments

Backend REST API

  • Endpoint: POST /api/cron/stripe-ticket-batch-refund
  • Authentication: Service JWT (Bearer token)
  • Request Body: StripeTicketBatchRefundRequest DTO
  • Response: StripeTicketBatchRefundResponse with job ID (HTTP 202 Accepted)

Batch Job Service

  • Job Name: StripeTicketBatchRefundJob
  • Trigger: On-demand via backend REST endpoint
  • Processing: Asynchronous, processes tickets in batches
  • Status Updates: Reports progress, success count, failure count

Frontend Integration

  • Page: /admin/events/[id]/tickets/list
  • Server Action: triggerStripeTicketBatchRefundServer() in ApiServerActions.ts
  • UI Component: Collapsible batch refund section (similar to sales analytics)
  • State Management: Loading, success, error states with job status display

10. Security & Compliance

Access Control

  • Admin-only access (enforced by backend authorization)
  • Tenant isolation: only refund tickets for current tenant
  • Event validation: ensure eventId belongs to tenant

Confirmation Requirements

  • Confirmation dialog before triggering batch refund
  • Display ticket count and warning message
  • Require explicit user confirmation

Audit Logging

  • Log all batch refund operations with timestamps
  • Record job ID, eventId, tenantId, ticket count
  • Track success/failure for each transaction
  • Store admin user ID who triggered the refund

Data Protection

  • No sensitive data in logs (mask payment intent IDs)
  • Secure API communication (HTTPS, JWT authentication)
  • Rate limiting for Stripe API calls

11. Error Handling & Edge Cases

Partial Refund Scenarios

  • Some refunds fail: Continue processing remaining tickets, log failures
  • Stripe API errors: Log error, mark transaction as failed, continue
  • Database errors: Rollback individual transaction update, continue
  • Network timeouts: Implement retry logic for Stripe API calls

Validation Errors

  • Invalid eventId: Return 400 Bad Request with error message
  • No eligible tickets: Return 200 OK with message "No eligible tickets found"
  • Event not found: Return 404 Not Found
  • Tenant mismatch: Return 403 Forbidden

Stripe API Errors

  • Already refunded: Skip ticket, log warning, continue
  • Invalid payment intent: Log error, mark as failed, continue
  • Rate limit exceeded: Implement exponential backoff retry
  • Insufficient funds: Log error, mark as failed, continue

Job Status Reporting

  • Report total tickets processed
  • Report successful refunds count
  • Report failed tickets count with error details
  • Provide job completion summary
