Member Social Feature – Feasibility & Integration Analysis

This document explores options for adding a member-only social experience to the MOSC platform. The goal is to provide a lightweight social layer—member profiles, mutual visibility, media sharing, and comments—exclusively for users whose profile designates them as members. The analysis focuses on solutions that can integrate with our existing Next.js + Clerk + proxy API architecture with minimal lift.

Current System Snapshot

  • Identity & Auth: Clerk is used for authentication; Next.js middleware controls public vs protected routes. Session data is exposed to server actions and proxy APIs.
  • Member Data: `user_profiles` table (via proxy endpoints) tracks roles such as member; tenant scoping is enforced via `tenantId` injection helpers.
  • Content Storage: Event media, sponsor media, and gallery uploads are served through existing API/proxy layers.
  • Frontend Stack: Next.js App Router with client/server components, Tailwind UI, and reusable feature cards.

Feature Objectives

  • Provide each member with a dedicated profile page visible only to other members.
  • Allow members to post short updates, upload media, and comment within the private community.
  • Keep the experience embedded within the MOSC UI while reusing Clerk sessions for SSO.
  • Maintain tenant isolation and reuse existing API proxy & JWT patterns.
  • Achieve “minimal viable social” with limited engineering investment.

Capability Gap Assessment

Our current backend does not provide a ready-made social graph or timeline. Building the entire stack (posts, reactions, media feeds, notifications) would require significant backend work, new database tables, and moderation tooling. A lightweight embedded community SaaS or headless social API can accelerate delivery while we retain control over authentication.

Hyvor vs. Bettermode – Cost & Scalability Notes

  • Hyvor Talk Communities: Entry plans start around $49/month with generous MAU limits (50k+ monthly visitors depending on tier). Higher tiers scale into hundreds of thousands of monthly users while staying under $249/month. Hyvor supports custom CSS/JS, JWT SSO, and API endpoints for exporting data— making it the most cost-effective option that still accommodates thousands of members and incremental customization. Limitations: primarily widget-based UI; deeper feature tweaks require working within the embed.
  • Tribe / Bettermode: Pricing typically begins at $599/month (Business plan) with support for advanced workflows, white-label experiences, and GraphQL APIs. Enterprise tiers handle millions of members and allow full UI overrides, but the cost is 10–12× higher than Hyvor. Bettermode is ideal if we need enterprise SSO, granular permissions, or custom feeds built on top of their headless APIs.

Summary: For a budget-conscious rollout that must still welcome thousands of authenticated members, Hyvor Talk offers the best price-to-scale ratio. Bettermode is more expensive but excels when we need deep integration (native React components via GraphQL) and enterprise-grade customization out of the gate.

Blogging & Long-Form Content Options

Many of our tenants/domains want a light blog or announcements area in addition to the member-only social feed. Neither Hyvor Talk nor Bettermode replaces a full CMS, but both can complement or host lightweight blogging:

  • Hyvor Ecosystem: Hyvor Talk focuses on discussions/comments. Hyvor also offers a separate product, Hyvor Blogs, which supports custom domains, multi-author posts, and Markdown. We can configure one blog per tenant and embed snippets or link directly from MOSC. SSO is not automatic between Talk & Blogs, so we would manage access by publishing public posts while keeping member-only conversations inside Hyvor Talk.
  • Bettermode Articles: Bettermode includes an Articles module (aka Knowledge Base) that supports rich-text posts, tagging, and collections. Articles can be restricted to specific member groups and exposed via Bettermode’s GraphQL API to render a fully custom blog UI inside MOSC. This is the closest “built-in” blogging option.
  • External Headless CMS: For stronger editorial tooling, we can integrate Sanity, Contentful, or Strapi. Each tenant/domain would map to a space/environment. We would render blog pages via Next.js server components and protect member-only posts with our existing middleware. The community plugin remains responsible for comments and social engagement, while the headless CMS stores the canonical content.

Recommendation: If we need only lightweight news updates, Bettermode’s Articles (or a small Hyvor Blog) may suffice. For a structured editorial calendar or multi-site branding requirements, pair the community plugin with a headless CMS so each domain retains control over layout and SEO while reusing the same auth and theming.

Recommendation: A hosted community platform with JWT-based SSO (e.g., Hyvor Talk or Bettermode) offers the fastest path. It keeps core data (member identity) inside MOSC while outsourcing feed storage, reactions, and moderation. We can progressively replace the embed with native UI using the provider’s REST/GraphQL endpoints if we outgrow the widget.

Integration Blueprint (JWT-SSO Community)

  1. Eligibility Check: Add server-side guard (middleware + API) ensuring only users whose profile has role === 'member' can access the community routes (e.g., /member-community).
  2. SSO Token Endpoint: Create a server action or API route (e.g., /api/member-community/sso) that:
    • Validates Clerk session
    • Fetches membership profile via proxy (/api/proxy/user-profiles/by-user/:id)
    • Generates provider-specific JWT payload with member id, name, avatar
    • Returns short-lived signed token to the frontend
  3. Embed Component: Build a client component that:
    • Calls the SSO endpoint on mount
    • Initializes the community widget with the token
    • Applies MOSC theming (background, typography)
  4. Member Directory: Reuse existing sponsor/event card patterns to render a member directory view using data from our backend; clicking a member card opens the plugin profile page.
  5. Media Governance: Configure the provider’s storage or use existing MOSC media proxy endpoints to avoid duplicating uploads. Some providers allow overriding media storage via webhooks.
  6. Moderation & Auditing: Assign staff accounts inside the community dashboard. Optionally send moderation webhooks back to MOSC for audit logs.

Security & Compliance Notes

  • Ensure JWT SSO secret is stored using our environment variable pattern (`withTenantId`, Amplify-safe config).
  • Limit API route to authenticated members; never expose provider secrets to the browser.
  • Document data residency & retention for the chosen SaaS provider (important for religious org privacy).
  • Maintain tenant isolation by including tenantId in the SSO payload or by provisioning separate community spaces per tenant.

Extension Idea: Member Matrimonial Module

A matrimonial board adds profile listings (bride/groom details), discoverability, and contact requests. Neither Hyvor Talk nor Bettermode ships a turnkey matrimonial vertical, but both platforms can be extended to cover the use case:

  • Hyvor Talk: Supports creating dedicated “community spaces” or threads. We can host a “Matrimonial” category where members post structured cards (title, description, media). To keep the UI consistent, we can render a custom Next.js listing page that consumes the Hyvor API (e.g., reading posts with a specific tag) and augments it with filters for age, location, occupation. Contact could flow through private comments or external links (WhatsApp, email).
  • Bettermode: Offers Collections, Custom Fields, and Workflow Automations. We can construct a “Matrimonial” collection with custom schema (birth year, parish, language, etc.). Bettermode’s GraphQL API lets us build a bespoke MOSC frontend for browsing/filtering while all submissions live within their moderation tools. We can also configure automated contact-request flows (e.g., button triggers DM or sends email).

If neither hosted option satisfies the exact matrimonial experience, we can still leverage the SSO community for the feed portion while storing structured profiles inside our own database. Members would publish “matrimonial posts” to generate awareness, and the detailed profile/contact flow could live on a secured MOSC page driven by our existing proxy APIs.
