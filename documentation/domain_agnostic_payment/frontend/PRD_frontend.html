Domain-Agnostic Payment Refactor – Frontend PRD

Repository: Current Next.js workspace (this project)
Goal: Deliver a multi-domain payment experience driven by backend orchestration while preserving premium UI/UX across tenants.

1. Context

The Stripe Elements integration is live today and tied to domain-specific publishable keys. Backend transformations (Payment Orchestrator, provider adapters) will centralize payment creation. The frontend must consume the new APIs, support Instant Checkout (ACP) when it becomes available, and expose alternative gateways (Zeffy, Zelle manual flow, Revolut, PayPal) with consistent UI patterns.

References:
  • cursor_rules/nextjs_api_routes.mdc – required proxy pattern for REST API calls.
  • @ui_style_guide.mdc – typography, colors, spacing, component guidance.
  • documentation/domain_agnostic_payment/PRD.html – master backend/architecture PRD.
  • documentation/Swagger_API_Docs/api-docs.json – backend contract definitions (use for typings).

2. Scope

In Scope

  • Update checkout forms to read payment configuration from backend (tenant-aware) and remove embedded Stripe publishable keys.
  • Implement UniversalPaymentCheckout client component that renders provider-specific UIs based on backend response.
  • Expose optional Instant Checkout CTA (ACP) once flagged by backend; maintain Stripe Elements flow as default.
  • Embed Zeffy donation CTA for events flagged as zero-fee fundraisers.
  • Surface Zelle manual payment option with instructions and pending status messaging.
  • Render Revolut checkout button for tenants configured with Revolut adapter; ensure PSD2/SCA prompts are handled via redirect.
  • Update membership management screens to support subscription pricing (monthly/annual) once backend endpoints are ready.
  • Ensure all new API calls use proxy endpoints and follow error handling/logging patterns.

Out of Scope

  • Backend orchestration, settlement logic, or database migrations.
  • Tenant billing invoicing UI (handled in admin backoffice later).
  • Offline payment reconciliation workflows (admin-only screens).

3. User Flows

  1. Standard Ticket Purchase: Shopper visits event page → selects tickets → UniversalPaymentCheckout determines provider:
    • Stripe (default): show wallet buttons + Elements card form.
    • PayPal: render PayPal Buttons (embedded) with redirect fallback.
    • Revolut: show “Pay with Revolut” button that redirects to hosted page for SCA; poll backend for completion.
  2. Instant Checkout (ACP) Preview: If backend returns { providerType: "STRIPE", sessionUrl }, expose “Instant Checkout (Beta)” button. Clicking opens modal with hosted checkout; once Stripe returns success we redirect to success page.
  3. Fundraising with Zeffy: Event flagged as DONATION_ZERO_FEE → UI displays Zeffy embed. If Zeffy fails, fallback message invites user to use Stripe or PayPal.
  4. Zelle Manual Flow: For events enabling Zelle, display instructions + confirm dialog. After confirmation, call backend to create a pending transaction and show “Awaiting confirmation” status to the user.

4. UI Components & Pages

  • Checkout Route:
    • Replace direct Stripe calls with /api/proxy/payments/initialize.
    • Handle provider-specific metadata and render appropriate sub-component (Stripe, PayPal, Revolut, Zeffy, Zelle manual).
  • Event Page (Upcoming Events Section):
    • Display badges for supported payment methods based on backend event metadata.
    • Ensure hero image logic remains unchanged (homepage hero > hero > placeholder).
  • Admin Tenant Settings:
    • Add toggles for “Instant Checkout beta”, “Enable Zeffy for fundraisers”, “Enable Zelle manual payments”, “Enable Revolut checkout”.
    • Form should call new proxy endpoints (to backend) and obey @ui_style_guide.mdc layout.
  • Membership Portal:
    • Implement subscription plan selection UI (monthly/yearly) and feed data to backend /api/proxy/memberships/subscriptions endpoints.
    • Show current subscription status, renewal date, and allow cancellation requests.

5. API Integration Requirements

  1. All fetches must use proxy routes defined under src/pages/api/proxy, following patterns in cursor_rules/nextjs_api_routes.mdc.
  2. Use typings derived from documentation/Swagger_API_Docs/api-docs.json for PaymentSession, PaymentStatusResponse, etc.
  3. Handle polling for Revolut/Zelle pending transactions via /api/proxy/payments/{transactionId}.
  4. Ensure sensitive provider keys are never exposed client-side; rely on backend responses for publishable keys or session URLs.

6. Accessibility & UX

  • All buttons must meet contrast requirements defined in @ui_style_guide.mdc.
  • Hosted flows (Stripe Instant Checkout, PayPal, Revolut) should open in modal/redirect with clear loading and success messaging.
  • Provide localized currency and date formats based on tenant settings (GBP/EUR support for UK/EU tenants).
  • Ensure keyboard navigation and screen-reader announcements when switching providers.

7. Risks & Mitigations

  • Dependency on Backend Completion: Frontend work requires finalized endpoints. Mitigation: mock responses based on Swagger schema.
  • Hosted Checkout Branding Loss: Instant Checkout reduces design control. Mitigation: keep Stripe Elements as default; highlight ACP as optional beta.
  • Zelle Manual Confirmation: Users might expect instant confirmation. Mitigation: clear messaging and follow-up email templates.
  • Zeffy Availability: Some events may not qualify. Mitigation: fallback logic automatically shows Stripe/PayPal if Zeffy returns errors.

8. Deliverables & Timeline (Frontend)

  • Week 1-2: Refactor checkout to consume backend initialization endpoint; implement UniversalPaymentCheckout skeleton.
  • Week 3: Integrate provider UIs (Stripe Elements fallback, PayPal Buttons, Revolut redirect, Zelle manual flow, Zeffy embed).
  • Week 4: Admin toggles, membership subscription UI, final QA.
  • Week 5: Pilot tenants (Stripe/PayPal), capture feedback, polish ACP beta surfaces.

9. Acceptance Criteria

  • All payment providers selectable via backend configuration with no client redeploy.
  • Checkout page renders correct provider-specific UI and handles success/cancel/error flows per provider.
  • ACP beta button only appears when backend indicates support; falls back gracefully otherwise.
  • Membership subscriptions show correct status and interact with backend endpoints.
  • All new API calls use proxy routes and pass lint/tests.
