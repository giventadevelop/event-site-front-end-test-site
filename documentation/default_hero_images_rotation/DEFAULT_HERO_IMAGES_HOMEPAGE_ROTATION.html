Default hero images — homepage & satellite hero rotation

This document describes the hybrid default hero image architecture implemented for multi-tenant satellite deployments (one Next.js codebase, many NEXT_PUBLIC_TENANT_ID values). It covers the shared resolver, homepage slideshow rotation fix, charity hero parity, onboarding seeding, admin configuration, and the platform-wide emergency fallback asset.

Problem before the fix

What was broken or inconsistent
  • Single hardcoded default — Homepage used /images/hero_section/default_hero_section_second_column_poster.webp baked into src/components/HeroSection.tsx. Every satellite deploy would need different files under public/, which does not scale for multi-tenant.
  • Missing JPEG fallback — Event, checkout, and success pages referenced /images/default_placeholder_hero_image.jpeg, which was not in the repo, causing broken images on those surfaces.
  • Charity hero drift — src/components/charity-sections/HeroSection.tsx used Builder.io CDN URLs as fallback instead of tenant-scoped settings.
  • No tenant defaults when events absent — When no upcoming event had isHomePageHeroImage media, rotation could show nothing useful or the wrong brand for a new satellite tenant.
Do not bundle per-tenant hero images in public/ per satellite deploy. Use cloud URLs in tenant_settings (primary) plus one small emergency asset in the repo (safety net). Do not download S3 objects to server disk at runtime — serve HTTPS CDN URLs and let the browser cache them (same pattern as logoImageUrl).

Shared resolver & fallback chain

All hero surfaces use helpers in src/lib/hero/defaultHeroImages.ts:

Function / constant Purpose
BUNDLED_EMERGENCY_HERO_IMAGE '/images/hero_section/hero_images/fallback/default-hero.webp' — single platform-wide emergency asset
parseTenantDefaultHeroUrls() Reads defaultHeroImageUrls array or defaultHeroImageUrlsJson string from tenant settings
resolveHeroImages() Homepage slideshow: events → tenant defaults → bundled; returns imageUrls, durationsMs, defaultSlideCount
resolveSingleHeroFallbackUrl() Single-image surfaces (checkout, success, tickets): prefers event URL, then tenant default, then bundled
applyTenantDisplayMode() slideshow (all URLs), random (one per resolve), single (first URL only)

Tenant settings fields (frontend DTO)

  • defaultHeroImageUrlsJson — JSON array of HTTPS URLs (ordered for slideshow)
  • defaultHeroDisplayMode — slideshow | random | single
  • defaultHeroIncludeWithEvents — when true, append tenant default slides after event hero images (legacy behavior, now explicit)

Homepage hero rotation (HeroSection.tsx)

Main homepage (src/components/HeroSection.tsx)

DynamicHeroImage was updated to:

  1. Wait for both useFilteredEvents('hero') and useTenantSettings() before initializing slides (avoids flashing wrong defaults).
  2. Build eventImageUrls from upcoming events with hero media (existing recurrence / date logic preserved).
  3. Call resolveHeroImages({ eventImageUrls, tenantSettings }) to produce the final dynamicImages array and durations.
  4. Track failedImageUrls and use onError on images to fall back to BUNDLED_EMERGENCY_HERO_IMAGE when a CDN URL fails.
  5. Keep existing rotation timers, pause/hover, and overlay logic tied to upcomingEvents (see overlay specification).
Rotation behavior: Event slides use per-media homePageHeroDisplayDurationSeconds (default 8s). Tenant default slides use a fixed 8s interval. Display mode random picks one tenant URL when there are no event images; with events, appended defaults still follow includeWithEvents.

Charity hero section

charity-sections/HeroSection.tsx

The charity homepage hero uses the same resolver as the main hero:

  • Imports resolveHeroImages and BUNDLED_EMERGENCY_HERO_IMAGE.
  • Loads tenant settings via useTenantSettings().
  • When no events are returned from the proxy, explicitly calls resolveHeroImages({ eventImageUrls: [], tenantSettings }) (fixes empty rotation).
  • On fetch error, same resolver path instead of Builder.io CDN fallback.

Satellite / multi-tenant behavior

One codebase, many tenants

Each Amplify app (primary or satellite) sets NEXT_PUBLIC_TENANT_ID. Tenant settings are loaded once by TenantSettingsProvider, which normalizes defaultHeroImageUrlsJson into defaultHeroImageUrls for client components.

Hero URLs are not read from config/satellites.json or public/images per tenant. Branding changes are a settings + S3 update, not a redeploy.

Server-side fallback: Pages that render on the server use getTenantHeroFallbackUrl() in src/lib/hero/getTenantHeroFallback.ts (fetches tenant settings with service JWT). Client pages use useHeroFallbackUrl() from src/hooks/useHeroFallbackUrl.ts.

Onboarding & seed script

When onboarding a new tenant_id:

  1. Copy template pack from s3://eventapp-media-bucket/tenant-templates/default-hero/ to tenants/{tenantId}/hero-defaults/.
  2. Seed tenant_settings with public HTTPS URLs via script or admin create flow.

Verification checklist

Manual test plan
  • Homepage with upcoming event hero media — event slides rotate; overlays match current event.
  • Homepage with no events — tenant default URLs rotate (or random/single per mode); else bundled webp.
  • New satellite tenant after onboarding script — settings contain URLs; homepage shows tenant branding without redeploy.
  • Broken S3 URL in pool — onError shows bundled fallback on homepage.
  • Event checkout / success pages — no 404 for old default_placeholder_hero_image.jpeg; hero shows event or tenant or bundled image.
  • Admin form — save URLs, reload edit page, previews and homepage reflect changes.
  • Charity route — same resolver behavior when events list is empty.
