Frontend Requirements: Stripe Ticket Batch Refund

This document specifies the UI/UX and client requirements for the batch refund feature in the Next.js frontend. Align all UI with @ui_style_guide.mdc and follow the pattern established in the sales analytics batch job trigger section.

1. Page Location & Context

  • Target Page: /admin/events/[id]/tickets/list
  • File Path: src/app/admin/events/[id]/tickets/list/page.tsx
  • Event ID Source: Extracted from URL params (params.id)
  • Placement: Add batch refund section above or below the ticket table, after statistics dashboard

2. UI Component Structure

2.1 Batch Refund Section Container

  • Collapsible section (similar to sales analytics page)
  • Default state: Collapsed (hidden)
  • Toggle button: "Show Batch Refund Options" / "Hide Batch Refund Options"
  • Styling: White background, gray border, rounded corners
  • Reference: src/app/admin/sales-analytics/SalesAnalyticsClient.tsx lines 448-597

2.2 Section Header

  • Title: "Stripe Ticket Batch Refund"
  • Toggle button on the right side of header
  • Icon: Sync/Refresh icon (FaSync from react-icons/fa)
  • Styling: Follow admin page header patterns

2.3 Information Banner

  • Blue background banner explaining the batch refund process
  • Text: "This batch job will refund all eligible Stripe tickets for this event. The job runs asynchronously in the background and will return a job ID when started."
  • Warning: "Only tickets with Stripe payments (not manual payments) will be refunded."
  • Styling: bg-blue-50 border-2 border-blue-200 rounded-lg p-4

2.4 Eligibility Summary

  • Display count of eligible tickets (optional - can be fetched from backend)
  • Show criteria: "Tickets with Stripe payments that are not already refunded"
  • Styling: Info box with gray background

3. State Management

State Variable Type Purpose
showBatchRefundSection boolean Controls visibility of batch refund options
batchRefundLoading boolean Loading state during API call
batchRefundSuccess boolean Success state after job trigger
batchRefundError string | null Error message if job trigger fails
batchRefundResponse StripeTicketBatchRefundResponse | null Response data with job ID and status
showConfirmDialog boolean Controls confirmation dialog visibility

4. Confirmation Dialog

  • Trigger: When user clicks "Trigger Batch Refund" button
  • Content:
    • Title: "Confirm Batch Refund"
    • Message: "Are you sure you want to refund all eligible tickets for this event? This action cannot be undone."
    • Ticket count: "X eligible tickets will be refunded"
    • Warning: "This will process refunds via Stripe. Manual payments will not be refunded."
  • Actions:
    • Cancel button: Close dialog, do nothing
    • Confirm button: Proceed with batch refund trigger
  • Styling: Follow dialog button patterns from dialog_button_styling.mdc
  • Reference: Similar to refund modal in TicketTableClient.tsx

5. Trigger Button

  • Label: "Trigger Batch Refund" (default) / "Starting Batch Job..." (loading)
  • Icon: FaSync (default) / FaSpinner with animate-spin (loading)
  • Styling:
    • Full-width button: w-full flex-shrink-0 h-14 rounded-xl
    • Background: bg-red-100 hover:bg-red-200 (red for destructive action)
    • Text: text-red-700 font-semibold
    • Icon container: w-10 h-10 rounded-lg bg-red-200
    • Disabled state: disabled:opacity-50 disabled:cursor-not-allowed
    • Hover effect: hover:scale-105 transition-all duration-300
  • Disabled When: Loading state is true
  • Reference: Follow admin action button pattern from admin_action_buttons_styling.mdc

6. Status Messages

6.1 Success Message

  • Display When: batchRefundSuccess && batchRefundResponse
  • Styling: bg-green-50 border-2 border-green-200 rounded-lg p-4
  • Content:
    • Icon: FaCheckCircle (green)
    • Title: "Batch refund job started successfully!"
    • Details:
      • Job ID: {batchRefundResponse.jobId}
      • Status: {batchRefundResponse.status}
      • Estimated Records: {batchRefundResponse.estimatedRecords}
      • Estimated Completion: {new Date(batchRefundResponse.estimatedCompletionTime).toLocaleString()}
      • Message: {batchRefundResponse.message}

6.2 Error Message

  • Display When: batchRefundError is not null
  • Styling: bg-red-50 border-2 border-red-200 rounded-lg p-4
  • Content:
    • Icon: FaExclamationTriangle (red)
    • Title: "Error"
    • Message: {batchRefundError}
  • Error Handling: Map technical errors to user-friendly messages (similar to sales analytics)

7. API Integration

7.1 Server Action

  • File: src/app/admin/events/[id]/tickets/list/ApiServerActions.ts
  • Function: triggerStripeTicketBatchRefundServer()
  • Signature: 
    export async function triggerStripeTicketBatchRefundServer(
  request: StripeTicketBatchRefundRequest = {}
): Promise<StripeTicketBatchRefundResponse>
  • Implementation:
    • Validate request parameters (eventId required)
    • Prepare payload with eventId, tenantId (from environment)
    • Call backend endpoint: POST ${API_BASE_URL}/api/cron/stripe-ticket-batch-refund
    • Use fetchWithJwtRetry from @/lib/proxyHandler
    • Handle errors and return user-friendly messages
    • Return response with job ID and status
  • Reference: src/app/admin/sales-analytics/ApiServerActions.ts lines 785-926

7.2 Request/Response Types

export interface StripeTicketBatchRefundRequest {
  eventId: number; // Required: Event ID from URL params
  tenantId?: string; // Optional: Defaults to NEXT_PUBLIC_TENANT_ID
  startDate?: string; // Optional: ISO 8601 format (future enhancement)
  endDate?: string; // Optional: ISO 8601 format (future enhancement)
}

export interface StripeTicketBatchRefundResponse {
  jobId: string;
  status: 'STARTED' | 'IN_PROGRESS' | 'COMPLETED' | 'FAILED';
  tenantId: string | null;
  eventId: number | null;
  estimatedRecords: number | null;
  estimatedCompletionTime: string | null;
  message: string;
}

7.3 API Call Pattern

  • Follow nextjs_api_routes.mdc rules
  • Use fetchWithJwtRetry for backend calls (NOT proxy endpoint)
  • Direct backend call: ${API_BASE_URL}/api/cron/stripe-ticket-batch-refund
  • Method: POST
  • Headers: Content-Type: application/json
  • Body: JSON stringified request payload
  • Cache: cache: 'no-store'

8. Event Handler

8.1 Handle Trigger Function

  • Function: handleTriggerBatchRefund()
  • Steps:
    1. Set loading state to true
    2. Clear previous errors and success states
    3. Prepare request payload with eventId from URL params
    4. Call server action: triggerStripeTicketBatchRefundServer(request)
    5. On success: Set response, success state, clear loading
    6. On error: Set error message, clear loading
  • Reference: src/app/admin/sales-analytics/SalesAnalyticsClient.tsx lines 330-414

9. Component Integration

9.1 Page Component Updates

  • Add batch refund section after statistics dashboard
  • Before ticket table or after (designer's choice)
  • Pass eventId from server component to client component if needed
  • Ensure proper state management and error boundaries

9.2 Client Component

  • Create or extend existing client component for batch refund UI
  • Can be added directly to page.tsx or as separate component
  • Use 'use client' directive if creating separate component
  • Import required icons: FaSync, FaSpinner, FaCheckCircle, FaExclamationTriangle

10. UI Standards Compliance

  • Follow @ui_style_guide.mdc for all styling
  • Use admin action button patterns from admin_action_buttons_styling.mdc
  • Use dialog patterns from dialog_button_styling.mdc for confirmation
  • Follow color coding: Red for destructive actions (refund)
  • Ensure accessibility: ARIA labels, keyboard navigation, screen reader support
  • Responsive design: Works on mobile, tablet, desktop

11. Error Handling

  • Map backend errors to user-friendly messages
  • Handle network errors gracefully
  • Display validation errors (invalid eventId, etc.)
  • Show timeout errors if job trigger takes too long
  • Reference error handling pattern from sales analytics (lines 831-926)

12. Testing Considerations

  • Test with events that have no eligible tickets
  • Test with events that have many eligible tickets
  • Test error scenarios (invalid eventId, network errors)
  • Test confirmation dialog (cancel and confirm flows)
  • Test loading states and button disabled states
  • Test success and error message display
  • Verify eventId is correctly passed from URL params
