GiveButter Event Campaign Embed Checkout
Event Hero + Embedded Widget, Redirection & Setup

1. Overview

This PRD defines the event campaign embed checkout: a new page that shows the event hero at the top and an embedded GiveButter widget in the center. It is distinct from:

  • Generic donate button: Site-wide or non-event GiveButter campaigns (homepage donate button).
  • Donation-checkout: Existing /events/[id]/donation-checkout page that opens GiveButter in a popup or new tab (no embed).

The new page uses the same event-scoped URL pattern and eligibility rules as donation-checkout for ticketed fundraiser/charity events with GiveButter, but renders the GiveButter widget inline (no Stripe, no mixed flow).

X-Frame-Options: GiveButter sets X-Frame-Options: sameorigin, so the full GiveButter page cannot be iframe'd. Only the widget (script + custom element <givebutter-widget id="..."> or <givebutter-form campaign="...">) can be embedded on our site. This page is our layout (hero + center area) with the GiveButter widget embedded.

2. Checkout URL Format

Pattern: /events/[id]/givebutter-checkout

Examples:

  • Local: http://localhost:3000/events/4202/givebutter-checkout
  • Production: https://your-domain.com/events/4202/givebutter-checkout

No query parameters required; the event id from the path is sufficient. This keeps parity with /events/[id]/donation-checkout and makes the page clearly event-specific.

Comparison: Currently donation-checkout is http://localhost:3000/events/4202/donation-checkout. The new givebutter-checkout follows the same pattern: http://localhost:3000/events/4202/givebutter-checkout.

3. User Flow & Redirection

Entry points:

  1. Events listing – User clicks “Buy ticket” or fundraiser CTA on an event card.
  2. Event detail page – Hero image or “Buy ticket” / fundraiser image has a URL to this checkout page.

Flow: Events listing or event detail (hero) → user clicks “Buy ticket” / fundraiser CTA → /events/[id]/givebutter-checkout → event hero at top, GiveButter widget in center. User completes donation/ticket purchase in the embedded widget.

When to redirect to givebutter-checkout: Only for events that are ticketed + fundraiser/charity with GiveButter (same eligibility as donation-checkout). For these events, “Buy ticket” and fundraiser hero links should point to /events/[id]/givebutter-checkout (or optionally remain donation-checkout if “embed” vs “popup” is configurable per event).

5. Page Layout and Behavior

  • Route: src/app/events/[id]/givebutter-checkout/ (page.tsx + server data + client component).
  • Top: Event hero (reuse same hero resolution as donation-checkout, e.g. from DonationCheckoutServerData or a shared server data module).
  • Center: GiveButter widget only (no Stripe, no donation-checkout form).

Data loading: Server fetches event by id, resolves hero image, reads donation_metadata via getDonationMetadata. Pass to client: heroImageUrl, eventId, givebutterCampaignId, givebutterWidgetId.

Widget rendering (client):

  • If givebutterWidgetId is set → <GivebutterWidget widgetId={givebutterWidgetId} /> (renders <givebutter-widget id="...">).
  • Else if givebutterCampaignId is set → <GivebutterWidget campaignId={givebutterCampaignId} /> (renders <givebutter-form campaign="...">).
  • Else show a message that GiveButter is not configured and link back to event or donation-checkout.

Access control: Only allow for events that are ticketed + fundraiser/charity with GiveButter. Reuse isTicketedFundraiserEvent and donation metadata checks; 404 or friendly error otherwise.

12. GiveButter Configuration from Dashboard

From the GiveButter dashboard you get two values:

  1. Event ticketing page link: https://givebutter.com/FwSx70 → campaign slug / ID = FwSx70 → store as givebutterCampaignId.
  2. Widget component: <givebutter-widget id="j1ek6j"></givebutter-widget> → widget ID = j1ek6j → store as givebutterWidgetId.

Both can be configured in the Event details page (EventForm) under the fundraiser/charity section and are persisted in donation_metadata.
