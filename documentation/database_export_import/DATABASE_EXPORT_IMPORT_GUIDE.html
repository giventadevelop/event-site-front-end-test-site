1. Quick-start workflow

📋 End-to-end steps
  1. Fast path: run database_export_import_refresh.bat — see §1a Automated batch script.
  2. cd E:\project_workspace\mosc-temp\code_html_template\SQLS
  3. Find Postgres container: docker ps | findstr postgres
  4. Export: docker exec -it <container_id> pg_dump ... > export.sql
  5. Reorder: node reorder_sql_inserts_final.cjs → corrected_event_media_inserts.ordered.sql
  6. Optional: Renumber IDs + trim Spring Batch — see §3a → sequence_fix_inserts/corrected_event_media_inserts.renumbered.sql
  7. Prod IDs (repo root): node scripts/replace-user-profile-ids-in-sql.mjs → corrected_event_media_inserts.ordered_PROD.sql
  8. In generated SQL: Find/Replace pg_dump → -- pg_dump (comment out dump metadata lines)
  9. Import ordered SQL into target database (empty schema first)
  10. Run sync_sequence_after_inserts.sql so new rows get IDs above the imported max

WORKSPACE_ROOT (required)

This is the parent folder of your mosc-temp checkout—not the mosc-temp folder itself. Example: F:\project_workspace when the repo lives at F:\project_workspace\mosc-temp.

The script uses this path to find SQL files under mosc-temp\code_html_template\SQLS, Node scripts under mosc-temp\scripts, and (for remote mode) credentials in mosc-temp\.env.local. If the path is wrong, the batch stops immediately with “mosc-temp not found”.

CONTAINER_ID (optional, positional)

The Docker container ID (or short ID) of your local PostgreSQL instance—the database you export from in steps 0–1. Example: a6fb865ce8e0.

If you omit it, the script runs docker ps | findstr postgres and uses the first matching container. That is fine when you only have one Postgres container running; pass the ID explicitly when you have several containers and need a specific one.

Not used for remote import: With /REMOTE, steps 5–7 connect to RDS (or other remote host) from .env.local. The container ID still matters for local export unless you also use /SKIP-EXPORT.

/FORCE — skip local confirmation prompt

Steps 5–7 are destructive: they drop and recreate the schema, import all data, and reset sequences. By default, on local Docker, the script pauses and asks you to type Y or YES before it continues.

/FORCE skips that local prompt so the script runs straight through after steps 1–4—useful in CI, scripted runs, or when you already know you want a full refresh.

Important: /FORCE does not bypass remote production safety. If you use /REMOTE, you must still type PRODUCTION at the prompt, even with /FORCE.

If you cancel at the prompt (or type anything other than the expected word), steps 1–4 still finish and their output files remain on disk; only steps 5–7 are skipped. Re-run later with /FORCE or /SKIP-EXPORT to apply without re-exporting.

/PROD — use production Clerk user IDs in the import file

After reordering, the workflow produces two import-ready files:

  • corrected_event_media_inserts.ordered.sql — dev/staging Clerk IDs from your local export
  • corrected_event_media_inserts.ordered_PROD.sql — same data, but user_profile rows use production Clerk IDs (via replace-user-profile-ids-in-sql.mjs)

Without /PROD, step 6 imports the non-PROD file—appropriate for refreshing a local Docker database that uses dev Clerk accounts. With /PROD, step 6 imports the _PROD file so user profiles match production sign-in after you load data into AWS RDS or another production-like environment.

Step 3 (PROD ID replacement) always runs during the batch; this flag only chooses which prepared file gets imported in step 6.

/SKIP-EXPORT — reuse existing export.sql

Normally step 1 runs pg_dump inside Docker and overwrites code_html_template\SQLS\export.sql. That requires Docker to be running and a reachable Postgres container.

/SKIP-EXPORT tells the script: do not dump again—use the export.sql already on disk, then continue with reorder (step 2), PROD copy (step 3), and the rest. The file must exist or step 1 fails with “export file not found”.

Typical uses: you already exported manually; export took a long time and you are retrying steps 2–7 after a failure; or you are pushing prepared SQL to production with /SKIP-EXPORT /REMOTE (no local Docker needed for step 0–1 when combined with /REMOTE—the script skips container detection in that case).

/REMOTE — target production Postgres instead of local Docker

By default, steps 5–7 run against your local Docker database via docker exec … psql. With /REMOTE, those same steps connect to a remote PostgreSQL server using credentials from mosc-temp\.env.local (RDS_ENDPOINT, DB_NAME, DB_USERNAME, DB_PASSWORD, etc.). See §1b.

Steps 1–4 still run on your PC (export from Docker unless skipped, reorder, PROD file generation, comment out pg_dump lines). Only schema apply, data import, and sequence sync hit the remote database.

Auto-enables PROD import: If you pass /REMOTE without /PROD, the script turns on /PROD anyway so step 6 uses corrected_event_media_inserts.ordered_PROD.sql.

Stricter confirmation: You must type PRODUCTION (exact word) before the remote schema rebuild runs. This applies even with /FORCE.

Prerequisites: Valid .env.local (or fallback database_target.local.env), network access to the host, and psql on PATH—or Docker Desktop so the batch can run psql inside a postgres:16 container.

/SCHEMA-ONLY — rebuild schema without importing data

Applies Latest_Schema_Post__Blob_Claude_12.sql (step 5) and skips export (steps 1–4), data import (step 6), and sequence sync (step 7). Use when you only need DDL changes applied to an empty or disposable database, or when data is already loaded and you want a schema refresh without touching INSERT files.

Automatically implies /SKIP-EXPORT behavior for the export pipeline. /IMPORT-FILE is ignored if both flags are passed.

/IMPORT-FILE — override the default import SQL

By default, step 6 imports either corrected_event_media_inserts.ordered.sql or corrected_event_media_inserts.ordered_PROD.sql (when /PROD or /REMOTE is set). Pass a custom path to import a different prepared SQL file:

  • /IMPORT-FILE "F:\path\to\custom.sql" — path as the next argument (quote if spaces)
  • /IMPORT-FILE=F:\path\to\custom.sql — path in the same token

The file must exist before the batch runs. Step 4 still patches pg_dump header lines in that file unless /SKIP-EXPORT is also set. Strict import: when /IMPORT-FILE is set, step 6 uses ON_ERROR_STOP=1 and the batch fails on the first SQL error (instead of logging a warning and continuing). Use /SKIP-EXPORT with custom files to skip export/reorder.

Common flag combinations
  • Full local refresh: WORKSPACE_ROOT — export, reorder, import into local Docker (confirm with Y/YES).
  • Full local refresh, no prompt: add /FORCE.
  • Local DB with prod Clerk IDs: add /PROD.
  • Schema only (local): WORKSPACE_ROOT /SCHEMA-ONLY /FORCE.
  • Custom import file: WORKSPACE_ROOT /SKIP-EXPORT /IMPORT-FILE "path\to\file.sql" /FORCE (recommended).
  • MOSC full custom import: /REGEN-MOSC /SKIP-EXPORT /IMPORT-FILE "...\mosc_malankara_orthodox_2.sql" /FORCE — see §1c.
  • MOSC fast refresh (no schema rebuild): /SKIP-EXPORT /DATA-ONLY /IMPORT-DUP-ONLY /FORCE — see §1c.
  • Production RDS push (reuse export): /SKIP-EXPORT /REMOTE — type PRODUCTION when asked.
  • Remote schema only: /SCHEMA-ONLY /REMOTE — type PRODUCTION when asked.
  • Azure VM, known container: WORKSPACE_ROOT CONTAINER_ID /FORCE.

1b. Import into production PostgreSQL (/REMOTE)

By default, steps 5–7 target your local Docker Postgres. With /REMOTE, the batch file applies the schema, imports the corrected_event_media_inserts.ordered_PROD.sql file, and syncs sequences on your AWS RDS (or other remote Postgres) using the same variables already in mosc-temp\.env.local.

Steps 1–4 still run on your machine (export from Docker, reorder, PROD user-ID copy). Only the destructive apply/import/sync steps hit the remote database.

⚠️ Production safety: Remote mode rebuilds the entire schema and replaces data on the target database. You must type PRODUCTION at the confirmation prompt. .env.local is gitignored — never commit it.

3a. Renumber IDs and trim Spring Batch (optional)

After reordering, the ordered file may still contain high tenant-specific IDs (for example MOSC event_type_details at 600001+), thousands of Spring Batch rows (batch_job_execution_params, batch_job_execution_log), and FK values that reference those old IDs. The script renumber_sql_insert_ids.cjs produces a separate import file with per-table IDs starting at 1, FK columns rewritten column-aware, and Spring Batch trimmed to a small sample — without overwriting the canonical corrected_event_media_inserts.ordered.sql in SQLS/.

⚙️ What the script does
  1. Parse every INSERT INTO public.<table> (including multiline VALUES).
  2. Trim Spring Batch: keep the first N batch_job_execution rows and cascade to instance, step, context, and params; keep the first N batch_job_execution_log rows (default N=3).
  3. Pass 1 — PK map: assign each table’s primary key 1, 2, 3, … in encounter order (--start-id=1 default). Spring Batch uses job_instance_id, job_execution_id, step_execution_id where applicable.
  4. Pass 2 — FK rewrite: update FK columns using schema FOREIGN KEY metadata plus manual overrides (junction tables, batch chain, event_live_update.event_id, etc.). Supports CREATE TABLE IF NOT EXISTS in the schema file (promotion email tables).
  5. Orphan repair: synthetic aliases for demo event IDs (500001–500003); rel_event_details__discount_codes rows with wrong event_details_id are corrected via linked discount_code.event_id.
  6. Write corrected_event_media_inserts.renumbered.sql and id-mapping-report.json.

7. Sync sequences after import

After manual INSERTs with explicit IDs, PostgreSQL sequences may still point at low values. Run sync_sequence_after_inserts.sql so the next auto-generated primary key starts above the highest imported ID (typically MAX(id) + 50). If you imported Spring Batch rows (including the trimmed set from §3a), also run sync_spring_batch_sequences.sql.
