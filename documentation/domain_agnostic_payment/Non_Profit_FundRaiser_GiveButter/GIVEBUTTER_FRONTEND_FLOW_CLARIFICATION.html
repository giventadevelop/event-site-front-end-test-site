GiveButter Frontend Control Flow Clarification

📋 Overview: This document clarifies how the GiveButter payment flow works from the frontend perspective, including API credential usage, webhook handling, and payment status determination.

Key Questions Answered

1. Are GiveButter API Key and Secret Used in Frontend?

❌ NO - Frontend Does NOT Use GiveButter API Credentials Directly

The frontend never accesses GiveButter API keys or secrets directly. All GiveButter API calls are made through backend proxy endpoints:

  • Frontend calls: /api/proxy/givebutter/donations/...
  • Backend proxy: Forwards requests to GiveButter API using server-side credentials
  • Backend credentials: Stored in environment variables or database (encrypted)
// ✅ CORRECT: Frontend calls backend proxy
const response = await fetch(`${baseUrl}/api/proxy/givebutter/donations/${donationId}/status`);

// ❌ WRONG: Frontend should NEVER call GiveButter API directly
// const response = await fetch('https://api.givebutter.com/v1/donations/...', {
//   headers: { 'Authorization': 'Bearer gb_live_xxx' } // NEVER expose API keys in frontend
// });

2. Are We Listening on Webhooks in Frontend?

✅ YES - Frontend Listens to Webhooks (Similar to Stripe Flow)

The frontend now listens to GiveButter webhooks and forwards them to the backend, similar to the Stripe checkout flow pattern. This ensures transaction data is persisted immediately when webhooks arrive.

  • Frontend webhook endpoint: /api/webhooks/givebutter/route.ts (Next.js API route)
  • Frontend webhook handler: Receives GiveButter webhook events, forwards to backend with JWT authentication
  • Backend webhook processing: Backend verifies webhook signature, creates/updates donation transactions in database
  • Frontend polling (fallback): Polls backend payment status endpoint if webhook hasn't processed yet (desktop/mobile flow)
// ✅ CORRECT: Frontend webhook handler (similar to Stripe webhook handler)
// Location: src/app/api/webhooks/givebutter/route.ts

export async function POST(req: NextRequest) {
  // 1. Read raw body and signature header
  const signature = req.headers.get('x-givebutter-signature');
  const rawBody = Buffer.from(await req.arrayBuffer());
  
  // 2. Get JWT token for backend authentication
  let jwt = await getCachedApiJwt();
  if (!jwt) jwt = await generateApiJwt();
  
  // 3. Forward webhook to backend with JWT authentication
  const backendResponse = await fetch(`${API_BASE_URL}/api/webhooks/givebutter`, {
    method: 'POST',
    headers: {
      'Content-Type': 'application/json',
      'X-Givebutter-Signature': signature || '',
      'Authorization': `Bearer ${jwt}`, // CRITICAL: JWT authentication required
    },
    body: rawBody,
  });
  
  // 4. Return backend response to GiveButter
  return new NextResponse(await backendResponse.text(), {
    status: backendResponse.status,
  });
}

3. Is Payment Status Determined by Webhooks?

✅ YES - Frontend Listens to Webhooks and Calls Backend to Persist Data

Payment status is determined through a webhook-based approach (similar to Stripe checkout flow):

  1. Frontend receives webhook from GiveButter → POST /api/webhooks/givebutter (frontend endpoint)
  2. Frontend forwards webhook to backend with JWT authentication → Backend verifies signature and creates/updates transaction
  3. Frontend success page polls backend payment status endpoint → Reads transaction data from database
  4. Frontend detects success → Generates QR code, sends email (or uses data from webhook-created transaction)
📋 Pattern Match: This follows the same pattern as the Stripe checkout flow:
  • Stripe: Frontend webhook handler at /api/webhooks/stripe/route.ts forwards to backend
  • GiveButter: Frontend webhook handler at /api/webhooks/givebutter/route.ts forwards to backend
  • Both: Use JWT authentication when forwarding to backend
  • Both: Success pages poll for transaction data (desktop/mobile flow)

Complete Control Flow

Donation-Based Checkout Flow (from /donation-checkout page)

1 User initiates payment on /donation-checkout page
POST /api/proxy/payments/initialize with paymentProvider: 'GIVEBUTTER'
2 Backend initializes GiveButter payment
Backend uses GiveButter API key (from env/database) to create payment session
Returns checkoutUrl or sessionUrl to frontend
3 Frontend redirects user to GiveButter
User completes payment on GiveButter's hosted checkout page
4 GiveButter processes payment and sends webhook
GiveButter → POST /api/webhooks/givebutter (frontend endpoint)
Frontend webhook handler: Receives webhook, forwards to backend with JWT authentication
Backend webhook processing: Verifies webhook signature, creates/updates donation transaction in database
5 GiveButter redirects user back to frontend
Redirect URL: /events/[id]/donation/success?transactionId=xxx or ?donationId=xxx
6 Frontend success page (desktop/mobile detection)
Desktop flow: Polls GET /api/events/donation/success/process?transactionId=xxx&eventId=xxx
Mobile flow: Uses POST /api/events/donation/success/process as fallback if webhook hasn't processed
Maximum 10 attempts (30 seconds total for desktop), similar to Stripe checkout flow
7 Backend returns transaction data
Transaction may already be created by webhook, or GET/POST endpoint creates it if webhook delayed
Desktop flow can create transaction immediately if webhook hasn't processed (similar to Stripe desktop flow)
8 Frontend detects transaction created
Stops polling, displays transaction details (donation or ticket transaction)
9 Frontend generates QR code and sends email
Calls backend endpoints:
  • POST /api/proxy/events/{eventId}/donations/{transactionId}/qrcode
  • POST /api/proxy/events/{eventId}/donations/{transactionId}/send-email
Or uses QR code/email data already returned from success processing endpoint

Webhook vs Polling Architecture

Location Frontend endpoint: /api/webhooks/givebutter/route.ts → Backend: /api/webhooks/givebutter Frontend success page: /events/[id]/donation/success
Trigger GiveButter sends webhook when payment completes → Frontend receives and forwards to backend Frontend polls every 3 seconds (desktop) or uses POST fallback (mobile) after redirect
Purpose Frontend forwards webhook to backend → Backend verifies signature and creates/updates transaction in database Check if transaction exists in database, create if webhook hasn't processed (desktop only)
Credentials Used JWT token (for backend authentication) + Webhook secret (backend verifies signature) JWT token (for backend API authentication)
Data Flow GiveButter → Frontend Webhook Handler → Backend (with JWT) → Database Frontend → Backend API (GET/POST) → Database → Frontend

Payment Status Determination Flow

How Payment Status is Determined

Scenario A: Webhook Arrives First (Primary Path)
  1. GiveButter sends webhook → Frontend receives at /api/webhooks/givebutter
  2. Frontend forwards webhook to backend with JWT → Backend verifies signature and creates transaction
  3. User redirected to success page
  4. Frontend polls backend → Immediately finds transaction (created by webhook)
  5. Frontend proceeds with QR code and email
Scenario B: Desktop Flow Creates Transaction if Webhook Delayed
  1. User redirected to success page
  2. Frontend polls backend (GET endpoint) → Transaction not found
  3. Desktop flow: GET endpoint creates transaction from GiveButter data if webhook hasn't processed
  4. Frontend gets transaction data → Proceeds with QR code and email
  5. Webhook arrives later (redundant, but safe - backend checks for existing transaction)
Scenario C: Mobile Flow Uses POST Fallback
  1. User redirected to success page
  2. Frontend detects mobile → Uses POST endpoint as fallback
  3. Mobile flow: POST endpoint creates transaction from GiveButter data if webhook hasn't processed
  4. Frontend gets transaction data → Proceeds with QR code and email
  5. Webhook arrives later (redundant, but safe - backend checks for existing transaction)

  • ✅ NO webhook listeners in frontend - Webhooks handled by backend

📋 Architecture Pattern (Matches Stripe Checkout Flow)

This is a webhook-based architecture with frontend listener (same pattern as Stripe):

  • Frontend Webhook Handler: Receives GiveButter webhooks, forwards to backend with JWT authentication
  • Backend Webhook Processing: Verifies webhook signature, creates/updates donation transactions in database
  • Desktop Flow (GET): Polls GET endpoint, creates transaction if webhook hasn't processed (fallback)
  • Mobile Flow (POST): Uses POST endpoint as fallback to create transaction if webhook hasn't processed
  • Idempotency: Both webhook and GET/POST endpoints check for existing transactions before creating (prevents duplicates)

This ensures reliability - the system works even if webhooks fail or are delayed, matching the proven Stripe checkout flow pattern.

Key Similarity to Stripe Flow:

  • Stripe: /api/webhooks/stripe/route.ts → Forwards to backend → Backend creates transaction
  • GiveButter: /api/webhooks/givebutter/route.ts → Forwards to backend → Backend creates transaction
  • Both: Success pages poll GET endpoint (desktop) or use POST fallback (mobile)
  • Both: Desktop flow can create transaction immediately if webhook delayed
