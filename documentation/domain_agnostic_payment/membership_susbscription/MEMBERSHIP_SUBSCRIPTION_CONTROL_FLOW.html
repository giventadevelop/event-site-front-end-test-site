Membership Subscription Control Flow - Recommended Solution

Detailed control flow addressing first-time users, incomplete data, and profile creation failures

Overview

This document provides a comprehensive control flow for the recommended solution that ensures user profiles exist before enabling payment. It covers all edge cases including first-time users, missing data, and profile creation failures.

1. Complete Control Flow Diagram

START: User visits /membership
  • User can browse plans without authentication
  • Plans are fetched from backend (public endpoint)
DECISION: User clicks "Subscribe"
  • Check: Is user authenticated? (userId exists?)
IF NOT AUTHENTICATED:
  • Redirect to: /sign-in?redirect_url=/membership/subscribe/[planId]
  • After sign-in, user will be redirected back to subscription page
IF AUTHENTICATED: Navigate to /membership/subscribe/[planId]
  • Server component checks authentication (already done)
  • Fetches membership plan details
  • Attempts to fetch user profile (non-blocking)
DECISION: Does user profile exist?
  • Check: userProfile?.id exists?
  • If not found server-side, client component will check again
IF PROFILE EXISTS:
  • ✅ Profile confirmed
  • ✅ Enable payment buttons
  • ✅ User can proceed with payment
  • → Continue to Payment Flow (Section 2)
IF PROFILE DOES NOT EXIST: Profile Creation Flow
  • Show loading state: "Setting up your account..."
  • Attempt to create user profile from Clerk data
  • → Continue to Profile Creation Flow (Section 3)

2. Profile Creation Flow (Critical Section)

Scenario A: First-Time User with Complete Clerk Data

Conditions: User has signed in with Clerk, has email, firstName, lastName

Step 1: Extract Clerk User Data
const email = user?.emailAddresses?.[0]?.emailAddress || user?.primaryEmailAddress?.emailAddress || ''; const firstName = user?.firstName || ''; const lastName = user?.lastName || ''; const phone = user?.phoneNumbers?.[0]?.phoneNumber || ''; const imageUrl = user?.imageUrl || '';
DECISION: Does email exist?
  • Email is REQUIRED for user profile creation
  • Email is REQUIRED for payment processing
IF EMAIL EXISTS: Create Profile
  • Call server action: createUserProfileFromClerkUser()
  • Payload includes: userId, email, firstName, lastName, phone, imageUrl
  • Backend creates user_profile record
// Server Action: createUserProfileFromClerkUser const profileData = { userId: userId, email: email, firstName: firstName || 'User', lastName: lastName || '', phone: phone || '', profileImageUrl: imageUrl || '', tenantId: getTenantId(), userRole: 'MEMBER', userStatus: 'PENDING_APPROVAL', createdAt: new Date().toISOString(), updatedAt: new Date().toISOString(), }; const response = await fetch('/api/proxy/user-profiles', { method: 'POST', headers: { 'Content-Type': 'application/json' }, body: JSON.stringify(profileData), });
DECISION: Profile creation successful?
  • Check: response.ok === true
  • Check: Response contains profile.id
IF SUCCESS:
  • ✅ Profile created successfully
  • ✅ Store profile in state
  • ✅ Enable payment buttons
  • ✅ Hide loading state
  • → Continue to Payment Flow
IF FAILURE:
  • ❌ Profile creation failed
  • Show error message: "Unable to set up your account. Please try again."
  • Show retry button
  • Log error for debugging
  • → Continue to Error Handling Flow (Section 4)

Scenario B: First-Time User with Missing Email

Conditions: User signed in but Clerk user has no email address

Note: This is rare but possible with certain OAuth providers or incomplete Clerk setup

Step 1: Extract Clerk User Data
  • Check: user?.emailAddresses?.[0]?.emailAddress
  • Check: user?.primaryEmailAddress?.emailAddress
DECISION: Email is missing or empty?
IF EMAIL MISSING:
  • ❌ Cannot create user profile (email is required)
  • ❌ Cannot proceed with payment (email required for Stripe)
  • Show error message: "Email address is required to complete your subscription. Please update your account settings."
  • Show button: "Update Account Settings"
  • Redirect to: /profile?redirect_url=/membership/subscribe/[planId]
  • On profile page, user must add email address
  • After email added, redirect back to subscription page
Profile Page Flow:
  • User lands on /profile page
  • Profile form shows email field (required)
  • User enters email address
  • On save, create/update user profile with email
  • After successful save, redirect to subscription page
  • Subscription page now has profile with email
  • → Payment buttons enabled

Scenario C: Profile Creation Fails (Network/Backend Error)

Conditions: Email exists, but profile creation API call fails

DECISION: Profile creation API call failed?
  • Check: response.status !== 200
  • Check: Network error or timeout
  • Check: Backend validation error
IF NETWORK ERROR:
  • Show error: "Network error. Please check your connection and try again."
  • Show retry button
  • On retry, attempt profile creation again
  • Maximum 3 retry attempts
IF BACKEND ERROR (400/500):
  • Show error: "Unable to set up your account. Please contact support if the problem persists."
  • Show support contact information
  • Log error details for debugging
  • Allow user to retry or contact support
IF TIMEOUT:
  • Show error: "Request timed out. Please try again."
  • Show retry button
  • Increase timeout for retry (e.g., 30 seconds)
FALLBACK: After 3 Failed Attempts
  • Show message: "We're having trouble setting up your account. Please try again later or contact support."
  • Show support email/phone
  • Allow user to save subscription plan for later
  • Option to receive email when issue is resolved

3. Payment Flow (After Profile Confirmed)

Profile Confirmed: Enable Payment
  • ✅ User profile exists (userProfile.id confirmed)
  • ✅ Email address available
  • ✅ Payment buttons enabled
Payment Options Available:
  • Desktop: Stripe Elements (Apple Pay, Google Pay, Link, Card)
  • Mobile: Payment Request Button (Apple Pay/Google Pay) + Checkout Session fallback
User Completes Payment:
  • Payment Intent or Checkout Session created
  • Payment processed by Stripe
  • Redirect to: /membership/success?pi=pi_xxx or ?session_id=cs_xxx
DECISION: Payment successful?
IF PAYMENT SUCCESSFUL:
  • Success page processes subscription
  • Looks up user profile (should exist now)
  • Creates membership_subscription record
  • Links subscription to userProfile.id
  • Shows success confirmation
IF PAYMENT SUCCESSFUL BUT PROFILE MISSING:
  • ⚠️ Edge case: Profile was deleted or not found
  • Success page attempts to create profile from payment data
  • Uses email from Payment Intent metadata
  • Creates profile, then creates subscription
  • → Fallback Profile Creation (Section 5)

4. Error Handling & Recovery

Error Scenario User Action System Response Recovery Path
No Email in Clerk User clicks "Subscribe" Show error, redirect to profile page User adds email → Profile created → Return to subscription
Profile Creation Fails Retry button clicked Attempt profile creation again Success → Enable payment | Failure → Show support contact
Network Error Retry button clicked Retry with increased timeout Success → Enable payment | 3 failures → Show support
Backend Validation Error Contact support Log error, show support info Admin fixes issue → User retries
Payment Success, Profile Missing Automatic Create profile from payment data Profile created → Subscription created → Success

5. Fallback Profile Creation (Success Page)

Edge Case: Payment Succeeded But Profile Missing
  • This should rarely happen if we enforce profile before payment
  • But we need fallback for edge cases
Success Page Processing:

Now proceed with subscription creation if (!userProfile?.id) { throw new Error('Unable to create user profile. Please contact support.'); } // 5. Create subscription... }

IF SUCCESS:
  • ✅ Profile created from payment data
  • ✅ Subscription created and linked
  • ✅ Show success message
IF FAILURE:
  • ❌ Cannot create profile (missing email or userId)

Please update your account settings.'); setCanEnablePayment(false); return; } // Create profile if (retryCount < 3) { setIsCreatingProfile(true); setProfileError(null); try { const newProfile = await createUserProfileFromClerkUser({ userId, email, firstName: user?.firstName || 'User', lastName: user?.lastName || '', phone: user?.phoneNumbers?.[0]?.phoneNumber || '', imageUrl: user?.imageUrl || '', }); if (newProfile?.id) { setUserProfile(newProfile); setCanEnablePayment(true); setIsCreatingProfile(false); setRetryCount(0); } else { throw new Error('Profile creation returned no ID'); } } catch (err) { setIsCreatingProfile(false); const errorMessage = err instanceof Error ? err.message : 'Failed to create profile'; setProfileError(`Unable to set up your account: ${errorMessage}`); if (retryCount < 2) { // Auto-retry after 2 seconds setTimeout(() => { setRetryCount(prev => prev + 1); }, 2000); } } } else { setProfileError('Unable to set up your account after multiple attempts. 8. Summary & Key Points

Recommended Solution Summary

  1. Require Profile Before Payment: Ensure user profile exists before enabling payment buttons
  2. Auto-Create Profile: If profile doesn't exist, create it from Clerk user data
  3. Email Validation: If email is missing, redirect to profile page to add email
  4. Error Handling: Handle profile creation failures with retry logic and user-friendly messages
  5. Fallback Safety: Add fallback profile creation in success page for edge cases

Key Decision Points

  • Email Required: Email is mandatory for both profile creation and payment processing
  • Profile Before Payment: Profile must exist before payment buttons are enabled
  • Retry Logic: Maximum 3 retry attempts for profile creation
  • Fallback Creation: Success page can create profile if missing (edge case)
  • User Experience: Show loading states and clear error messages
