Backend Fix: GiveButter Routing for Donation-Based Checkout Flow

⚠️ CRITICAL REQUIREMENT: This fix must NOT interfere with or disrupt the existing Stripe checkout flow. The donation-based checkout flow should be a completely separate control path.

Problem Statement

Event 4202 is currently routing to Stripe instead of GiveButter when accessed through the donation-based checkout flow (/events/[id]/donation-checkout).

Current Behavior (Incorrect):
  • Event ID: 4202
  • Admission Type: ticketed
  • Payment Provider: STRIPE ❌
  • Payment Use Case: TICKET_SALE
  • Success URL: /event/success?transactionId=xxx&eventId=4202
Expected Behavior (Correct):
  • Event ID: 4202
  • Admission Type: ticketed (with fundraiser metadata)
  • Payment Provider: GIVEBUTTER ✅
  • Payment Use Case: TICKET_SALE (but routed to GiveButter)
  • Success URL: /events/4202/donation/success?transactionId=xxx

Root Cause Analysis

The backend payment orchestration service is not correctly routing to GiveButter despite the frontend explicitly requesting it. Based on backend logs, the issues are:

  1. GiveButter Provider Not Configured: The backend logs show: "Event 4202 is configured for GiveButter but provider not available for tenant tenant_demo_002 - falling back to other providers"
    • This indicates GiveButter provider configuration is missing in the payment_provider_config table
    • The backend needs GiveButter API credentials configured for tenant tenant_demo_002
  2. Frontend Request Parameters Ignored: The frontend is explicitly sending:
    • paymentType: 'DONATION_ZERO_FEE'
    • paymentProvider: 'GIVEBUTTER'
    • returnUrl: '/events/4202/donation/success'
    But the backend is ignoring these and routing to Stripe instead.
  3. Success URL Override: The backend is setting its own success URL (/event/success?transactionId=7801&eventId=4202) instead of using the returnUrl provided by the frontend.

Frontend behavior when GiveButter is not configured

When the backend falls back to Stripe (e.g. GiveButter provider missing), it returns a clientSecret (Stripe PaymentIntent) and no checkoutUrl/sessionUrl. The donation-checkout page (/events/[id]/donation-checkout) expects a GiveButter checkout URL. The frontend now:

  • Detects missing checkout URL and presence of clientSecret (Stripe fallback)
  • Shows a clear error: "GiveButter is not configured for this organization..."
  • Offers a "Use standard ticket checkout (Stripe)" link to /events/[id]/checkout

Users will not be redirected to Stripe from the donation-checkout page; they must use the standard checkout link or the organizer must configure GiveButter.

Required Event Configuration

For an event to route to GiveButter (instead of Stripe), the event's donationMetadata must contain: 

{
  "isFundraiserEvent": true,
  "zeroFeeProvider": "GIVEBUTTER",
  "givebutterCampaignId": "<campaign-id>" // Optional but recommended
}

The backend payment initialization endpoint should check these fields in the request:

Backend Routing Logic Requirements

1. Payment Provider Selection Logic (Priority Order)

The payment orchestration service should check the following conditions in order when determining which payment provider to use:

  1. Check Explicit Request Parameter (HIGHEST PRIORITY):
    • If request.paymentProvider === 'GIVEBUTTER' OR request.paymentType === 'DONATION_ZERO_FEE', route to GiveButter immediately (do not check event metadata or provider config)
    • This ensures the frontend's explicit request is honored
  2. Check Event Metadata (SECONDARY): If no explicit provider is specified, check:
    • If donationMetadata.isFundraiserEvent === true AND donationMetadata.zeroFeeProvider === "GIVEBUTTER", route to GiveButter
  3. Default to Stripe: If none of the above conditions are met, route to Stripe (existing behavior).

  • Different success URL handling for donation-based transactions

// Pseudo-code for payment provider selection
function selectPaymentProvider(request, event) {

4. Webhook Handling Architecture

Desktop vs Mobile Flow Patterns

The frontend follows the same desktop/mobile flow patterns as Stripe (see .cursor/rules/desktop_payment_flow.mdc and .cursor/rules/mobile_payment_flow.mdc):

  • Desktop: Stays on success page, polls via GET endpoint, creates transaction as fallback if webhook delayed
  • Mobile: Shows brief success, redirects to QR page, uses POST endpoint as fallback
  • Webhook: Primary method - frontend receives webhook, forwards to backend, backend creates transaction
  • Idempotency: Both webhook and fallback paths check for existing transactions before creating

The backend should support the same patterns for GiveButter donations as it does for Stripe ticket sales.
