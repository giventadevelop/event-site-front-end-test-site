Batch Job Requirements: Stripe Ticket Batch Refund

This document defines the batch job implementation for processing Stripe ticket refunds asynchronously. The job is triggered on-demand via the backend REST API and processes refunds in batches to handle large volumes efficiently. It applies to the batch job repository at E:\project_workspace\event-site-manager-batch-jobs and integrates with the backend at E:\project_workspace\malayalees-us-site-boot.

1. Job Type & Trigger

  • Job Type: On-demand (triggered via API, NOT scheduled)
  • Job Name: StripeTicketBatchRefundJob
  • Trigger Method: Backend REST endpoint calls batch job service
  • Trigger Endpoint: POST /api/cron/stripe-ticket-batch-refund (in backend)
  • Job ID Format: stripe-batch-refund-{timestamp}-{eventId}

2. Job Parameters

Parameter Type Required Description
eventId Long Yes Event ID to process refunds for
tenantId String Yes Tenant ID for data isolation
jobId String Yes Unique job identifier
startDate LocalDateTime No Optional start date filter for purchase date
endDate LocalDateTime No Optional end date filter for purchase date

3. Job Processing Steps

3.1 Initialization Phase

  1. Read job parameters (eventId, tenantId, jobId, optional date filters)
  2. Validate parameters (eventId and tenantId are required)
  3. Query total count of eligible tickets
  4. Initialize job status tracking (processed count, success count, failure count)
  5. Set job status to IN_PROGRESS

3.2 Ticket Query Phase

  • Query: Select eligible tickets matching criteria
  • Criteria:
    • event_id = :eventId
    • tenant_id = :tenantId
    • stripe_payment_intent_id IS NOT NULL
    • status != 'REFUNDED'
    • stripe_payment_status IN ('succeeded', 'paid')
    • (startDate IS NULL OR purchase_date >= :startDate)
    • (endDate IS NULL OR purchase_date <= :endDate)
  • Ordering: ORDER BY created_at ASC (process oldest first)
  • Batch Size: Process in chunks of 100-500 tickets per batch

3.3 Refund Processing Phase

For each ticket in the eligible list:

  1. Validate Ticket:
    • Verify ticket still meets eligibility criteria (idempotency check)
    • Skip if already refunded (status = 'REFUNDED')
    • Skip if stripePaymentIntentId is null
  2. Call Stripe Refund API:
    • Endpoint: stripe.refunds.create()
    • Parameters:
      • payment_intent: Ticket's stripePaymentIntentId
      • reason: "requested_by_customer" (optional)
      • metadata: Include ticket transaction ID and job ID
    • Handle Stripe API response
  3. Update Database:
    • Only update if Stripe refund succeeds
    • Set status = 'REFUNDED'
    • Set refund_amount = refund.amount / 100 (convert cents to dollars)
    • Set refund_date = NOW()
    • Set refund_reason = 'Event canceled - Batch refund'
    • Set stripe_payment_status = 'refunded'
    • Set updated_at = NOW()
  4. Log Result:
    • Success: Log ticket ID, refund ID, amount
    • Failure: Log ticket ID, error message, error type
    • Increment success or failure counter
  5. Update Progress:
    • Increment processed count
    • Update job status with progress percentage
    • Log progress every 10-50 tickets (configurable)

4. Error Handling

4.1 Stripe API Errors

Error Type Handling Strategy Action
Already refunded Skip ticket Log warning, increment skipped count, continue
Invalid payment intent Mark as failed Log error, increment failure count, continue
Rate limit exceeded Retry with backoff Wait and retry (exponential backoff), max 3 retries
Insufficient funds Mark as failed Log error, increment failure count, continue
Network timeout Retry Retry up to 3 times, then mark as failed

4.2 Database Errors

  • Update failure: Log error, mark transaction as failed, continue processing
  • Connection error: Retry database operation, fail job if persistent
  • Constraint violation: Log error, skip ticket, continue

4.3 Job-Level Errors

  • Critical errors: Set job status to FAILED, log error, stop processing
  • Partial failures: Continue processing, report failures in completion summary
  • No eligible tickets: Set status to COMPLETED with message "No eligible tickets found"

5. Progress Tracking

5.1 Status Updates

  • Update job status at key milestones:
    • STARTED - Job initiated
    • IN_PROGRESS - Processing tickets
    • COMPLETED - All tickets processed
    • FAILED - Critical error occurred
  • Log progress every N tickets (configurable, default: 50)
  • Include in log: processed count, success count, failure count, percentage complete

5.2 Metrics Tracking

  • Total Tickets: Count of eligible tickets
  • Processed: Number of tickets processed (success + failure)
  • Successful: Number of successful refunds
  • Failed: Number of failed refunds
  • Skipped: Number of tickets skipped (already refunded, etc.)
  • Total Refund Amount: Sum of all successful refund amounts

6. Database Updates

6.1 Transaction Update

UPDATE event_ticket_transaction
SET status = 'REFUNDED',
    refund_amount = :refundAmount,
    refund_date = :refundDate,
    refund_reason = 'Event canceled - Batch refund',
    stripe_payment_status = 'refunded',
    updated_at = NOW()
WHERE id = :transactionId
  AND status != 'REFUNDED'  -- Idempotency check
  AND tenant_id = :tenantId;

6.2 Update Conditions

  • Only update if Stripe refund API call succeeds
  • Use database transaction for atomicity
  • Include idempotency check (status != 'REFUNDED')
  • Include tenant_id in WHERE clause for safety
  • Verify update affected exactly 1 row

7. Stripe API Integration

7.1 Refund API Call

Refund refund = stripe.refunds().create(
    RefundCreateParams.builder()
        .setPaymentIntent(ticket.getStripePaymentIntentId())
        .setReason(RefundCreateParams.Reason.REQUESTED_BY_CUSTOMER)
        .putMetadata("ticket_transaction_id", ticket.getId().toString())
        .putMetadata("batch_job_id", jobId)
        .putMetadata("event_id", eventId.toString())
        .putMetadata("refund_reason", "Event canceled - Batch refund")
        .build()
);

7.2 Rate Limiting

  • Stripe API rate limit: 100 requests per second (check current limits)
  • Implement throttling: Process max 50-80 refunds per second
  • Add delay between batches if needed
  • Handle 429 (Too Many Requests) with exponential backoff

7.3 Retry Logic

  • Retry on transient errors (network, timeout, rate limit)
  • Max retries: 3 attempts
  • Backoff strategy: Exponential (1s, 2s, 4s)
  • Do not retry on permanent errors (invalid payment intent, already refunded)

8. Job Completion

8.1 Completion Summary

  • Generate completion summary with:
    • Total tickets processed
    • Successful refunds count
    • Failed refunds count
    • Skipped tickets count
    • Total refund amount
    • List of failed ticket IDs with error messages
    • Job duration (start time to end time)
  • Set job status to COMPLETED
  • Log completion summary

8.2 Failure Reporting

  • Store failed ticket details in job execution context or separate table
  • Include: ticket ID, error message, error type, timestamp
  • Allow administrators to view failed tickets and retry individually if needed

9. Audit Logging

  • Log job start: jobId, eventId, tenantId, ticket count, start time
  • Log each refund attempt: ticket ID, Stripe refund ID, amount, status
  • Log errors: ticket ID, error type, error message, timestamp
  • Log job completion: end time, summary statistics
  • Use structured logging (JSON format recommended)
  • Include correlation ID (jobId) in all log entries

10. Performance Considerations

  • Batch Processing: Process 100-500 tickets per batch to avoid memory issues
  • Database Connections: Use connection pooling, close connections properly
  • Stripe API: Implement rate limiting and throttling
  • Memory Management: Stream tickets instead of loading all into memory
  • Timeout Handling: Set appropriate timeouts for Stripe API calls
  • Parallel Processing: Consider parallel processing for large batches (future enhancement)

11. Idempotency

  • Job should be idempotent: Running the same job twice should not cause duplicate refunds
  • Check ticket status before processing: Skip if already refunded
  • Use database constraints: Ensure status updates are atomic
  • Track processed tickets: Maintain list of processed ticket IDs in job context
