Liturgical Calendar – Frontend Integration (Strapi)

How the Next.js / MOSC frontend (mosc-temp) calls the Strapi-backed liturgical calendar API, maps the response to the existing LiturgyReading format, and displays the data. Designed so the existing SyroLiturgySection component requires no changes.

1. Overview & Architecture

The frontend component (SyroLiturgySection) calls /api/liturgy?lng=en|ml and expects the response { message: LiturgyReading[] }. This contract stays the same regardless of backend. The Next.js API route acts as a proxy that:

  1. Receives lng (and optionally date) from the frontend.
  2. Calls Strapi’s /api/liturgy-days with tenant and date filters.
  3. Maps the Strapi response (bilingual fields + readings component) to the flat LiturgyReading[] array using the requested language.
  4. Returns { message: LiturgyReading[] } to the frontend.

Data Flow

Browser (SyroLiturgySection) | | GET /api/liturgy?lng=en v Next.js API Route (src/app/api/liturgy/route.ts) | | GET ${STRAPI_URL}/api/liturgy-days | ?filters[tenant][tenantId][$eq]=${tenantId} | &filters[date][$eq]=${today} | &populate[readings]=* v Strapi (strapi-editorial-template) | | Returns: { data: [{ dayHeadingEn, dayHeadingMl, seasonNameEn, seasonNameMl, readings: [...] }], meta } v Next.js API Route | | Maps to: { message: [{ liturgy_day_heading, season_name, liturgy_heading, content_place }, ...] } v Browser (SyroLiturgySection) | | Renders: day heading, season, list of readings v User sees Liturgical Calendar

3. Next.js API Route (/api/liturgy)

The existing route at src/app/api/liturgy/route.ts currently calls the external SMCIM API. To support Strapi, add a conditional branch based on LITURGY_DATA_SOURCE.

3.1 Updated Route (Pseudocode)

import { NextRequest, NextResponse } from 'next/server';

const LITURGY_DATA_SOURCE = process.env.LITURGY_DATA_SOURCE || 'external';
const STRAPI_URL = process.env.STRAPI_URL || 'http://localhost:1337';
const TENANT_ID = process.env.NEXT_PUBLIC_TENANT_ID || '';

export async function GET(request: NextRequest) {
  const lng = request.nextUrl.searchParams.get('lng') ?? 'en';

  if (LITURGY_DATA_SOURCE === 'strapi') {
    return fetchFromStrapi(lng);
  } else {
    return fetchFromExternalApi(lng); // existing logic
  }
}

async function fetchFromStrapi(lng: string) {
  try {
    // Today's date in YYYY-MM-DD
    const today = new Date().toISOString().split('T')[0];

    const params = new URLSearchParams({
      'filters[tenant][tenantId][$eq]': TENANT_ID,
      'filters[date][$eq]': today,
      'populate[readings]': '*',
      'sort': 'order:asc',
    });

    const headers: Record<string, string> = {
      Accept: 'application/json',
    };
    if (process.env.STRAPI_API_TOKEN) {
      headers['Authorization'] = `Bearer ${process.env.STRAPI_API_TOKEN}`;
    }

    const res = await fetch(`${STRAPI_URL}/api/liturgy-days?${params}`, {
      method: 'GET',
      headers,
      next: { revalidate: 3600 }, // cache 1 hour
    });

    if (!res.ok) {
      console.error('[Liturgy API] Strapi error:', res.status);
      return NextResponse.json(
        { error: 'Liturgy service unavailable' },
        { status: 502 }
      );
    }

    const json = await res.json();
    const strapiDay = json.data?.[0];

    if (!strapiDay) {
      return NextResponse.json({ message: [] });
    }

    const message = mapStrapiToLegacy(strapiDay, lng);
    return NextResponse.json({ message });
  } catch (err) {
    console.error('[Liturgy API] Error:', err);
    return NextResponse.json(
      { error: 'Failed to fetch liturgy readings' },
      { status: 500 }
    );
  }
}

function mapStrapiToLegacy(day: any, lang: string) {
  const isEn = lang === 'en';
  const dayHeading = isEn ? day.dayHeadingEn : day.dayHeadingMl;
  const seasonName = isEn ? day.seasonNameEn : day.seasonNameMl;

  const readings = (day.readings || []).map((r: any) => ({
    liturgy_day_heading: dayHeading || '',
    season_name: seasonName || '',
    liturgy_heading: isEn ? r.liturgyHeadingEn : r.liturgyHeadingMl || '',
    content_place: isEn ? r.contentPlaceEn : r.contentPlaceMl || '',
  }));

  if (readings.length === 0) {
    readings.push({
      liturgy_day_heading: dayHeading || '',
      season_name: seasonName || '',
      liturgy_heading: '',
      content_place: '',
    });
  }

  return readings;
}
Note: The route retains the existing fetchFromExternalApi function for the legacy SMCIM API. Switching is controlled by the LITURGY_DATA_SOURCE environment variable. Set it to strapi to use the Strapi backend.

5. Tenant Handling

  • The tenant ID comes from process.env.NEXT_PUBLIC_TENANT_ID (server-side) or from auth context.
  • It is injected into the Strapi query as filters[tenant][tenantId][$eq]=<tenantId>.
  • The frontend component does not send the tenant ID – the API route handles it.
  • This follows the same pattern as other proxy routes in the project (see proxyHandler.ts).
Important: Never omit the tenant filter. Without it, readings from all tenants will be returned.

6. Frontend Component (No Changes)

Because the API route maps Strapi’s response to the same { message: LiturgyReading[] } format, the existing SyroLiturgySection component requires no modifications.

6.1 Current Component Behavior

src/app/mosc/components/SyroLiturgySection.tsx

  1. Maintains lng state (en or ml).
  2. On mount / language change, fetches /api/liturgy?lng={lng}.
  3. Expects { message: LiturgyReading[] } in the response.
  4. Renders: day heading (liturgy_day_heading), season (season_name), list of readings with cross icon.
  5. Shows loading spinner, error message, or “no readings” as appropriate.
Zero UI changes required. The API route contract is preserved. The component is backend-agnostic.

6.2 Alternative: Direct Strapi Call (Not Recommended)

If a project wants to bypass the Next.js API route and call Strapi directly from the client component, the component would need to:

  1. Know the Strapi URL (requires NEXT_PUBLIC_STRAPI_URL environment variable).
  2. Include the tenant filter in the fetch URL.
  3. Map the Strapi response to LiturgyReading[] in the component (duplicating the mapping logic).

This approach is not recommended because it exposes the Strapi URL to the client, requires tenant handling on the client, and duplicates mapping logic.

8. Switching Between External API and Strapi

The LITURGY_DATA_SOURCE environment variable controls which backend is used:

ValueBackendWhen to Use
external (default)External SMCIM APIWhen Strapi liturgy content type is not yet set up or populated.
strapiStrapi /api/liturgy-daysWhen Strapi is configured and has liturgy day entries.

8.1 Migration Steps

  1. Set up the liturgy-day content type and liturgy.reading component in Strapi (see Strapi Server Specification).
  2. Configure public permissions for find and findOne on liturgy-day.
  3. Create at least one liturgy day entry for today’s date in Strapi.
  4. Set STRAPI_URL and LITURGY_DATA_SOURCE=strapi in .env.local.
  5. Restart the Next.js dev server.
  6. Verify the liturgy section displays Strapi data.
  7. For production: set STRAPI_URL, LITURGY_DATA_SOURCE=strapi in AWS Amplify environment variables and add to next.config.mjs env block.

8.2 Rollback

To revert to the external API, set LITURGY_DATA_SOURCE=external (or remove it entirely). No code changes needed.
