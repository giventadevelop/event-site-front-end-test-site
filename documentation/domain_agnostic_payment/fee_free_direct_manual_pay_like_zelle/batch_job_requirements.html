PRD: Batch Job Requirements for Manual Payment Reporting

This PRD defines the batch job scope for aggregating fee-free manual payment data into a reporting table, exposing summary APIs for the admin dashboard, and ensuring the backend can serve summary metrics. It applies to the batch job repository at E:\project_workspace\event-site-manager-batch-jobs and integrates with the backend at E:\project_workspace\malayalees-us-site-boot. Notification features are out of scope.

1. Goals & Non-Goals

Goals

  • Nightly aggregation of manual payment data into a reporting table.
  • Provide dashboard-ready summaries grouped by event, status, and payment method.
  • Expose backend APIs to fetch summary data for admin dashboards.
  • Keep data consistent with tenant isolation.

Non-Goals

  • No reminder/notification emails for pending requests.
  • No auto-void or expiry enforcement for manual payments.
  • No cross-tenant aggregation.

2. Data Sources & Tables

  • manual_payment_request (source of manual payment records)
  • event_details (to verify event_id and manual payment eligibility)
  • manual_payment_summary_report (target reporting table)

3. Batch Job: Manual Payment Summary Aggregation

  • Schedule: nightly (e.g., 2:00 AM tenant-local).
  • Input: manual_payment_request rows where event_id is not null.
  • Grouping keys: tenant_id, event_id, payment_method_type, status, snapshot_date.
  • Metrics: total_amount (sum of amount_due), transaction_count (row count).
  • Filtering: only include events with manual payments enabled.
  • Output: insert new snapshot rows into manual_payment_summary_report.

4. Backend APIs (to be implemented)

These APIs live in the backend and are consumed via proxy routes in the frontend.

  • GET /api/manual-payment-summary
  • Query params: tenantId.equals, eventId.equals, snapshotDate.equals, status.equals
  • Returns summary rows for the admin dashboard.

5. Admin Dashboard Consumption

  • Frontend uses proxy routes per cursor_rules/nextjs_api_routes.mdc.
  • Dashboard should show totals grouped by status and method per event.
  • UI must follow @ui_style_guide.mdc styling.

6. Retention Policy

  • Keep daily snapshots for 18 months; archive thereafter.
  • Retention policy is documented in the schema for manual_payment_summary_report.

7. Audit & Monitoring

  • Log failed aggregation runs and store job status metadata.
  • Surface batch job health in admin analytics dashboard.
  • Record last successful snapshot date per tenant for monitoring.
