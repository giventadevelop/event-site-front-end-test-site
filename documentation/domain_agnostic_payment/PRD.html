Executive Summary

This PRD outlines the transformation of the MCEFEE event management platform from a single-domain, Stripe-centric payment implementation to a domain-agnostic, multi-tenant payment architecture that supports multiple payment providers (Stripe, PayPal, and future providers) while maintaining a consistent, user-friendly checkout experience across all client domains.

Current Pain Points

  • Each domain requires separate Stripe account configuration
  • Payment processing logic is tightly coupled to frontend (client-side Stripe integration)
  • Limited payment provider flexibility
  • Inconsistent payment flows for different use cases (tickets, donations, offerings)
  • Manual domain-specific setup for each new tenant

Proposed Solution

Backend-centric payment orchestration layer that abstracts payment provider logic, enabling any client domain to process payments through the centralized backend API without provider-specific frontend code.

2. Goals & Objectives

Primary Goals

  1. Domain Agnosticism
    • Support multiple client domains pointing to single backend
    • Zero frontend changes when adding new domains
    • Tenant-specific payment provider configuration
  2. Provider Abstraction
    • Support multiple payment providers (Stripe, PayPal, future providers)
    • Unified payment API regardless of provider
    • Easy provider switching per tenant
  3. Consistent User Experience
    • One-click checkout where possible
    • Consistent flow for tickets, donations, and offerings
    • Modern payment methods (Apple Pay, Google Pay, Link, Crypto)
  4. Backend-Centric Processing
    • Move payment logic from frontend to backend
    • Frontend becomes thin presentation layer
    • Backend maintains payment provider credentials
  5. Future-Ready Architecture
    • Support for emerging standards (ACP, AP2)
    • Webhook-driven state management
    • Extensible for new payment types

Secondary Goals

  1. Improved payment analytics and reporting
  2. Better refund and chargeback handling
  3. Support for recurring payments (subscriptions)
  4. Multi-currency support
  5. Enhanced fraud detection

Key Architectural Principles

  1. Backend-Centric Payment Processing
    • All payment provider interactions happen in backend
    • Frontend only receives generic payment tokens/sessions
    • Payment provider credentials stored securely in backend
  2. Provider Adapter Pattern
    • Each payment provider has dedicated adapter implementing PaymentService interface
    • Adapters translate generic payment requests to provider-specific API calls
    • New providers can be added without changing core logic
  3. Tenant Configuration
    • Each tenant configured with preferred payment provider(s)
    • Multiple providers per tenant for fallback/routing
    • Configuration stored in database, cached for performance
  4. Webhook-Driven State Management
    • All payment state changes driven by provider webhooks
    • Idempotent webhook processing
    • Reliable eventual consistency
  5. Frontend Provider Agnosticism
    • Single universal checkout component
    • Dynamic loading of provider-specific UI elements (if needed)
    • Consistent UX regardless of backend provider

15. Addendum: Multi-Domain Charity & Emerging Payment Considerations (November 2025)

15.1 CeFi / Charity-Focused Payment Flows

Several tenants require charity-exclusive payment journeys (locally branded as “Cephi”) where donations are accepted without tax receipts or with domain-specific treatment. To support this within the domain-agnostic architecture:

  • Provider Routing: Extend PaymentProviderConfig with a payment_use_case field (e.g., TICKET, DONATION_CEFI, MEMBERSHIP). The orchestration layer maps charity events to providers that support CeFi rails (bank transfer, ACH, regional gateways).
  • Provider Abstractions: Add a CharityPaymentAdapter implementing the same PaymentService interface. It will call the third-party CeFi API (REST/GraphQL) and translate results into our unified PaymentTransaction records.
  • Webhook Handling: CeFi providers often rely on batch settlement files. We should add a PaymentSettlementService that ingests CSV/webhook payloads, updates transaction state, and records settlement metadata in payment_transaction.metadata.
  • Tenant Opt-In: Add admin tooling so each tenant can toggle CeFi support per event. The frontend “Donate via CeFi” CTA would invoke /api/payments/initialize with paymentType='DONATION_CEFI'.

15.2 Agentic Payment Protocols (Google AP2 vs. OpenAI ACP)

Both protocols promise AI-assisted purchasing, but integration complexity differs:

Feature ACP (OpenAI + Stripe) AP2 (Google) Feasibility Notes
Current Maturity Production-ready (Stripe Instant Checkout) Developer preview (limited access) ACP is implementable today; AP2 requires early-access partnerships.
Implementation Effort Low: extend Stripe adapter with SPT creation endpoints Medium-High: new adapter, mandate management, crypto optionality ACP is the fastest addition once Stripe refactor ships; AP2 should remain Phase 4 R&D.
UX Impact ChatGPT / agent checkout flows, instant confirmations Agent-mediated payments across Google ecosystem Both require clear user consent flows; ACP aligns with our existing Stripe credentials.

Recommendation: Land end-to-end Stripe ACP support in Phase 3 (after adapter refactor). Keep AP2 in a research backlog until Google publishes stable SDKs and partner onboarding guidelines.

ACP Benefits for Our Platform

  • Faster Mobile Conversions: Stripe’s Instant Checkout page is optimized for one-tap wallets, reducing friction compared to embedding and maintaining custom Elements UI.
  • Agent-Ready: Enables AI assistants (ChatGPT, in-app agents) to complete purchases using Shared Payment Tokens while we remain in control of fulfillment and customer relationship.
  • Consistent Multi-Tenant Behavior: ACP honors our tenant-based Stripe keys and provider configs; we can turn it on per tenant without custom code per domain.
  • Lower Maintenance: Stripe owns the hosted UI. We spend less time chasing Elements/Wallet changes, yet can fall back to Elements when we need deep branding.
  • Compliance & Fraud Protection: ACP leverages Stripe’s Radar, SCA, and compliance stack automatically—no new certifications beyond our existing Stripe integration.
  • Future-Proofing: Positions the platform for conversational commerce and voice agent payments, matching the roadmap for multi-domain event/donation flows.

15.3 Membership Subscription Billing

Tenants want to charge recurring membership fees to members identified in user_profiles. Enhancements required:

  • Schema: Introduce membership_plan and membership_subscription tables (linking tenant, user, plan, status, next billing date). The provided SQL snapshot (code_html_template/SQLS/Latest_Schema_Post__Blob_Claude_11.sql) already includes payment_transaction; extend it with subscription columns (billing_interval, auto_renew).
  • Backend: Add SubscriptionPaymentService wrapper so adapters can create recurring profiles (Stripe Billing, PayPal Subscriptions, CeFi recurring ACH).
  • Frontend: For Next.js routes, follow CLI rules in cursor_rules/nextjs_api_routes.mdc—all membership REST calls should go through proxy endpoints that forward to the backend subscription controllers.
  • Member Eligibility: Enforce tenant-specific membership via user_profiles.tenant_id and the existing member role flag. Subscriptions should only appear for users flagged as member within the tenant.

15.4 Service Fees & Settlement

We must charge domain owners a platform fee while letting funds settle directly to their accounts:

  • Fee Columns: Ensure payment_transaction captures amount_platform_fee and amount_processing_fee (already present). Backend adapters should calculate these per provider configuration.
  • Reporting: Add daily settlement reports grouped by tenant_id and provider, exposing net payout vs. platform fees for finance reconciliation.
  • Invoicing: Schedule a monthly job that invoices tenants for platform fees (if fees are not netted out by provider). Output stored in a new platform_invoice table.

15.6 Zeffy™ Zero-Fee Fundraising Gateway

Some tenants run fundraising events that rely on zero-fee donation platforms such as Zeffy™. To accommodate this:

  • Gateway Adapter: Implement a ZeffyPaymentAdapter (CeFi style) that calls Zeffy’s REST APIs. The adapter should read event-specific configuration (API key, campaign ID, optional donor “tip” settings) from payment_provider_config with provider_type='ZEFFY' and payment_use_case='DONATION_ZERO_FEE'.
  • Dynamic Event Routing: Extend event metadata to flag “fundraising” events and optionally select the preferred gateway. During /api/payments/initialize, the orchestration service resolves the event’s gateway priority (e.g., Zeffy first, fallback to Stripe donations if unavailable).
  • AI-Friendly Routing: For ACP/AP2 integrations, include the gateway hint in the metadata returned by PaymentSession. Agentic checkouts can then surface “Zero-fee donation (Zeffy)” as the recommended option and gracefully downgrade to card providers if Zeffy is offline.
  • Settlement & Reporting: Zeffy covers transaction fees, but we still record the gross donation and $0 processing fee in payment_transaction. Platform fees remain configurable per tenant if they want to add optional surcharges or donor-paid tips.

Limitation: Zeffy targets registered nonprofits/charities. Their terms currently require proof of charitable status and are geared toward donations and fundraising ticketing (galas, benefit concerts). Standard commercial events without a nonprofit sponsor are ineligible. For non-fundraising ticket sales we should fall back to Stripe/PayPal/CeFi adapters unless the tenant qualifies as a Zeffy-approved organization.

15.7 Zelle® Payment Support (Feasibility & Considerations)

Tenants frequently request Zelle as a “free” person-to-person option. While Zelle excels at P2P transfers, it lacks a merchant API comparable to Stripe or PayPal. Key findings:

  • Integration Model: Zelle does not expose open APIs for programmatic charge initiation. Businesses must enroll with partner banks and often receive transfers manually. Therefore, the platform can only surface Zelle as an offline payment choice (display tenant’s Zelle email/phone) and mark the transaction as “pending” until the admin confirms receipt.
  • Buyer Experience: When a user chooses Zelle, the UI should display:
    • Tenant’s Zelle handle (email/phone) and amount owed.
    • Instructions to include event name/registration ID in the memo field.
    • A confirmation step where the user acknowledges that their registration becomes active only after manual verification.
  • Event vs. Fundraiser Usage:
    • Ticketed Events: Zelle is risky because ticket inventory is limited; reservations must remain soft until payment clears. Recommended only for VIP/manual approval scenarios.
    • Fundraisers/Donations: Zelle is more acceptable. Admins can reconcile deposits and mark payment_transaction.status as CONFIRMED after funds arrive.
  • Platform Automation: Add a PaymentService subtype (ZelleManualAdapter) that:
    • Creates a payment_transaction in status='INITIATED' with metadata capturing Zelle handle and memo.
    • Triggers email instructions to the buyer and admin.
    • Exposes an admin endpoint to mark payments as confirmed/cancelled once funds are verified.
  • Pros & Cons:
    • Pros: Zero processing fees, popular among US-based donors/attendees, immediate bank-to-bank transfer for participants.
    • Cons: No API automation, manual reconciliation required, chargeback/fraud protections limited, no support for international attendees.

Recommendation: Offer Zelle as an optional manual payment method (similar to “Pay by Check”) with clear UI messaging about manual confirmation. Encourage tenants to use automated gateways (Stripe, Zeffy, CeFi adapters) for real-time ticket issuance, while reserving Zelle chiefly for donations or one-off special cases.

15.8 ACP vs. Stripe Checkout – Mobile & Wallet Support

The Agentic Commerce Protocol (ACP) builds on Stripe’s Instant Checkout foundation. For our architecture, ACP is not a replacement for Stripe—it is an additional mode of initiating Stripe payments, primarily triggered by AI agents (e.g., ChatGPT).

  • Compatibility with Existing Setup: ACP uses the same Stripe account, secret/publishable keys, and PaymentIntent objects we already rely on. Our backend Stripe adapter simply adds new endpoints to mint Shared Payment Tokens (SPTs) and Instant Checkout sessions. No need to re-architect multi-domain support.
  • Wallet Support (Apple Pay / Google Pay): ACP adopts Stripe’s wallet support. If we enable Apple Pay and Google Pay in the Stripe dashboard, ACP-triggered checkouts automatically honor those payment methods. Therefore, mobile users on iOS/Android still see one-tap wallet buttons just like in our current Stripe Elements flow.
  • UI Implications:
    • Current UX: Our frontend renders the Stripe PaymentElement and wallet buttons (Apple/Google Pay) using Stripe.js.
    • ACP UX: When an AI agent initiates payment, Stripe serves a hosted Instant Checkout page. The page enforces the same wallet availability but requires minimal UI on our side—we simply redirect or embed the Stripe-hosted session.
    • Manual Users: We can still expose a “Pay with ACP (beta)” button that invokes the backend to create an Instant Checkout session. This is especially useful for mobile because the hosted page is fully responsive.
  • Mobile-Friendliness: Yes. ACP Instant Checkout is mobile responsive and supports Google Pay on Android (via Chrome, etc.) and Apple Pay on Safari/iOS, provided the device/browser meets Stripe’s eligibility checks. Users tap the wallet button and complete payment in the native sheet, similar to our existing frontend integration.
  • Advantages vs. Current Stripe Elements Flow:
    • Offloads more UI maintenance to Stripe (fewer Elements updates for us).
    • Allows AI agents to complete purchases on behalf of users (future-facing use case).
    • Supports the same fraud tools (Radar), wallet coverage, and SCA compliance as Stripe Elements.

    Trade-offs: We lose some UI customization because the checkout page is hosted by Stripe. For highly branded experiences, we still need the direct Elements integration.

  • Recommendation: Continue using Stripe Elements for primary web checkout (gives us full control, matches existing UI) and implement ACP Instant Checkout as an optional flow—ideal for agent-driven payments and fast mobile conversions. Ensure the backend Stripe adapter exposes both.

15.9 Settlement & Service Fee Automation

Because gateway payouts flow directly to the tenant’s merchant account (Stripe Connect, PayPal Business, bank ACH for CeFi), we cannot intercept funds per ticket sale. Instead we must reconcile and invoice/auto-debit platform fees post-settlement:

  • Per-Transaction Fee Capture: Populate amount_platform_fee and amount_processing_fee on each payment_transaction. Store fee percentages/flat amounts in payment_provider_config per tenant/event.
  • Daily Settlement Job: Nightly, aggregate transactions by tenant/provider where status = SUCCEEDED and settlement_batch_id IS NULL. Produce a settlement record (platform_settlement table) capturing gross amount, platform fee, and net.
  • Tenant Invoicing: If the tenant’s gateway auto-deducts our fee (Stripe Connect application_fee), mark the fee as collected. Otherwise, generate a monthly invoice (platform_invoice) and auto-charge the tenant’s on-file payment method or ACH debit using Stripe Billing/GoCardless.
  • Admin Dashboard: Provide a “Revenue & Fees” tab listing settlements, allowing finance teams to export CSVs and track unpaid invoices.
  • Event-Level Visibility: Event owners can view net proceeds per event (gross sales – platform fee – processing fee) once the batch job posts the settlement entry.

This approach keeps our backend as system-of-record for fee calculation while respecting the provider’s direct payout model. Stripe Connect or PayPal Partner APIs can automate platform fee deduction; otherwise we rely on invoicing + auto-debit to recover fees after settlement.

15.10 International Tenants & Revolut Gateway Option

When onboarding tenants outside North America (e.g., UK-based organizations serving UK attendees), we must ensure currency coverage, local wallet support, and compliance with PSD2/SCA rules:

  • Existing Stripe Setup: Stripe supports multi-currency PaymentIntents, GBP settlement, Apple Pay, Google Pay, and 3D Secure. Our current multi-tenant configuration already handles this if the tenant’s Stripe account is registered in the UK. Ensure payment_provider_config includes the desired settlement currency and the adapter sets currency on PaymentRequest appropriately.
  • Revolut Business Gateway: Revolut offers card acquiring across 25+ currencies with competitive FX. To add Revolut:
    • Implement a RevolutPaymentAdapter using their Merchant API for GBP/EUR processing.
    • Store API keys/OAuth tokens in payment_provider_config with provider_type='REVOLUT'.
    • Handle Strong Customer Authentication by redirecting the shopper to Revolut’s hosted SCA challenge (adapter must expose redirect URLs similar to PayPal).
    • Map Revolut webhooks into payment_webhook_log for settlement state tracking.
  • Settlement & Fees: Revolut payouts deposit into the tenant’s Revolut Business account in GBP/EUR. Our platform fee logic remains the same: record per-transaction platform fee and invoice tenants monthly if GPay/Apple Pay is handled via Revolut tokenization.
  • Regulatory Considerations: UK/EU merchants must comply with PSD2. Revolut and Stripe both enforce SCA. For CeFi ACH-style flows we must verify that the provider supports UK Faster Payments or SEPA before enabling the tenant.

Recommendation: For UK tenants, Stripe remains the quickest path (existing adapter, global wallet support). Revolut is a strong alternative when tenants demand multi-currency FX savings or already operate on Revolut Business. Both integrate cleanly with the domain-agnostic orchestration layer via dedicated adapters.

  5. Reconciliation:
    • Use Zeffy payout reports to mark payment_transaction entries as settled (import CSV/webhook).

15.12 Current Stripe Client Implementation vs. Target Architecture

Current State (2025): The frontend already embeds Stripe Elements via StripeDesktopCheckout.tsx and StripePaymentRequestButton.tsx, relying on publishable keys per domain. The backend stores ticket transactions in existing tables but lacks unified provider abstractions.

  • Frontend Changes: Retain Elements/UI components for branded checkouts, but add conditional logic to invoke Instant Checkout / ACP sessions when enabled. Remove hard-coded publishable keys from the client; instead fetch tenant-specific keys via backend-provided config (domain-agnostic).
  • Backend Changes: Use the new payment_transaction, payment_provider_config, and adapter layer to encapsulate all Stripe interactions. The existing Stripe PaymentIntent creation in Next.js API routes will migrate to the Spring Boot orchestrator.
  • Migration Plan: Update proxy routes to call backend endpoints, deprecate direct client-side PaymentIntent creation, and run DB migrations to align legacy data with the new schema. Ensure multi-tenant configuration loads the correct Stripe account settings at runtime.
