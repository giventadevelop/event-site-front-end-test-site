1. What Is This Batch Job?

The Subscription Renewal Batch Job is an automated process that checks all active membership subscriptions in your database and ensures they are synchronized with Stripe (the payment processor). Think of it as a "health check" that runs periodically to make sure your database and Stripe are in sync.

Simple Analogy: Imagine you have a library with books (subscriptions). Some books might have been checked out or returned (renewed) by Stripe, but your library records (database) might not be updated yet. This batch job is like a librarian who goes through all the books, checks with the actual checkout system (Stripe), and updates your records to match.

2. When Does It Run?

2.1 Scheduled Execution (Automatic)

The batch job runs automatically on a schedule:

  • Default Schedule: Every 6 hours (at midnight, 6 AM, noon, and 6 PM)
  • Configuration: Controlled by batch.subscription-renewal.schedule-cron in application.yml
  • Can Be Disabled: Set batch.subscription-renewal.enabled=false to turn off automatic runs
Scheduled Execution (BatchJobScheduler.java)
@Scheduled(cron = "${batch.subscription-renewal.schedule-cron:0 0 */6 * * *}")
public void scheduledSubscriptionRenewal() {
    // Runs automatically every 6 hours
    batchJobOrchestrationService.runSubscriptionRenewalJob(null, null, null);
}

2.2 Manual Trigger (On-Demand)

You can also trigger the batch job manually at any time:

  • Via API: POST /api/batch-jobs/subscription-renewal
  • Via Script: node scripts/test-subscription-renewal/trigger-batch-job.js
  • Use Case: When you need to immediately sync subscriptions (e.g., after a webhook fails or for testing)
Manual Trigger Example
node scripts/test-subscription-renewal/trigger-batch-job.js \
  --tenant-id=tenant_demo_002 \
  --subscription-id=sub_1SeifsK5BrggeAHMBvg2XE93 \
  --batch-size=10 \
  --max-subscriptions=100

3. What Subscriptions Does It Process?

The batch job doesn't process all subscriptions. It only looks at subscriptions that meet specific criteria:

3.1 Selection Criteria

Criteria What It Means Why
Status Must Be ACTIVE or TRIALING Only active subscriptions (not cancelled, expired, or past due) No point checking subscriptions that are already cancelled
Not Scheduled for Cancellation cancel_at_period_end = false Skip subscriptions that are already set to cancel at the end of the period
Renewal Date Within Window current_period_end is within 7 days (or 30 days for Stripe-checked subscriptions) Only check subscriptions that are approaching renewal or recently renewed
Tenant-Scoped Only subscriptions for the specified tenant Multi-tenant system - each tenant's subscriptions are processed separately
Database Query (MembershipSubscriptionRepository.java)
SELECT * FROM membership_subscription
WHERE tenant_id = :tenantId
  AND subscription_status IN ('ACTIVE', 'TRIALING')
  AND cancel_at_period_end = false
  AND (
    current_period_end <= CURRENT_DATE + 7 days  -- Database date check
    OR (stripe_subscription_id IS NOT NULL
        AND current_period_end <= CURRENT_DATE + 30 days)  -- Extended window for Stripe-checked
  )
ORDER BY current_period_end ASC

4. What Happens During Processing? (Step-by-Step)

Processing Flow

1 Reader Phase: Find Subscriptions

The batch job queries the database to find all subscriptions that match the criteria above. It processes them in batches (default: 100 subscriptions at a time) to avoid overwhelming the system or Stripe's API.

2 Processor Phase: Check Each Subscription

For each subscription found, the processor does the following:

  • If subscription has Stripe ID:
    • Fetches the latest subscription data from Stripe
    • Compares Stripe's dates with database dates
    • Checks if the subscription is within the renewal window (7 days)
    • Updates database dates if they don't match Stripe (self-healing)
  • If subscription has NO Stripe ID:
    • Uses only database dates
    • Checks if renewal date is within 7 days
    • Marks for processing if within window

3 Writer Phase: Save Updates

After processing, the writer saves any changes back to the database:

  • Updates current_period_start and current_period_end dates
  • Updates reconciliation_status (SYNCED, UPDATED, PROCESSED, or ERROR)
  • Records last_reconciliation_at timestamp
  • Records last_stripe_sync_at timestamp (if Stripe was checked)

5. What Gets Updated in the Database?

The batch job updates the following fields in the membership_subscription table:

Field What Gets Updated When
current_period_start Start date of current billing period If Stripe's date differs from database
current_period_end End date of current billing period (renewal date) If Stripe's date differs from database
reconciliation_status Status: SYNCED, UPDATED, PROCESSED, or ERROR Always updated to reflect processing result
last_reconciliation_at Timestamp of when this subscription was last checked Always updated
last_stripe_sync_at Timestamp of when Stripe was last checked Only if subscription has Stripe ID
reconciliation_error Error message if processing failed Only if an error occurred
Important: The batch job does NOT charge customers or create new subscriptions. It only reads data from Stripe and updates your database to match. Stripe handles the actual billing and renewal automatically.

6. Self-Healing Feature

One of the most important features of this batch job is its "self-healing" capability:

What Is Self-Healing?

If the database dates don't match Stripe's dates, the batch job automatically updates the database to match Stripe (the source of truth). This fixes any discrepancies that might have occurred due to:

  • Webhook failures (Stripe sent an update but your system didn't receive it)
  • Manual changes in Stripe dashboard
  • System errors or downtime
  • Time zone differences

6.1 Example Scenario

Before Batch Job Runs:

  • Database says: current_period_end = 2025-01-15
  • Stripe says: current_period_end = 2025-01-22 (subscription was renewed)

After Batch Job Runs:

  • Database is updated to: current_period_end = 2025-01-22
  • reconciliation_status = "UPDATED"
  • last_stripe_sync_at = 2025-12-31 12:00:00

7. Error Handling and Retry Logic

The batch job is designed to be resilient:

7.1 Retry Logic

  • Stripe API Failures: If Stripe API call fails, the job retries up to 3 times
  • Network Errors: Temporary network issues are retried automatically
  • Rate Limiting: If Stripe rate limits the request, the job waits and retries

7.2 Fallback Behavior

If Stripe cannot be reached (network outage, API down, etc.), the batch job falls back to using database dates only:

  • Checks if database current_period_end is within 7 days
  • Marks subscription as "PROCESSED" (not "SYNCED" since Stripe wasn't checked)
  • Logs a warning for manual review

7.3 Error Status

If processing fails completely (e.g., database error, invalid data), the subscription is marked with:

  • reconciliation_status = "ERROR"
  • reconciliation_error = "Error message here"
  • The batch job continues processing other subscriptions (doesn't stop on one error)

8. Configuration Options

The batch job behavior can be customized via configuration:

Configuration Default Value What It Controls
batch.subscription-renewal.enabled true Enable/disable scheduled execution
batch.subscription-renewal.schedule-cron 0 0 */6 * * * Cron expression for schedule (every 6 hours)
batch.subscription-renewal.batch-size 100 Number of subscriptions processed in each batch
batch.subscription-renewal.max-subscriptions 10000 Maximum number of subscriptions to process in one run
batch.subscription-renewal.renewal-days-ahead 7 How many days before renewal to start checking
batch.subscription-renewal.stripe-check-extended-days 30 Extended window for subscriptions with Stripe ID
subscription.renewal.use-database-fallback false Use database dates as fallback when Stripe dates differ (dev/testing only)
Production Warning: The use-database-fallback option should be false in production. Setting it to true means the batch job will use database dates even when Stripe dates differ, which can lead to incorrect renewal processing. Only use true for development/testing environments.

9. What Happens When You Trigger It Manually? (Step 18.4.6)

When you run the command from Section 18.4.6:

Manual Trigger Command
node scripts/test-subscription-renewal/trigger-batch-job.js \
  --tenant-id=tenant_demo_002 \
  --subscription-id=sub_1SeifsK5BrggeAHMBvg2XE93 \
  --batch-size=10 \
  --max-subscriptions=100

9.1 What the Script Does

  1. Authenticates: Gets a JWT token from the backend API using credentials
  2. Sends Request: Makes a POST request to /api/batch-jobs/subscription-renewal
  3. Passes Parameters:
    • tenantId: Which tenant's subscriptions to process
    • stripeSubscriptionId (optional): Process only one specific subscription
    • batchSize: How many subscriptions to process at once
    • maxSubscriptions: Maximum total subscriptions to process
  4. Returns Immediately: The API returns a job execution ID (job runs asynchronously)
  5. Reports Statistics: After job completes, shows how many subscriptions were processed

9.2 What Happens Next

Once triggered, the batch job:

  1. Finds subscriptions matching the criteria (or the specific subscription if ID provided)
  2. Processes each one (checks Stripe, updates database)
  3. Saves all updates to the database
  4. Logs the results

Let's walk through a concrete example:

10.1 Scenario

You have a subscription that renewed yesterday, but the webhook from Stripe failed to update your database.

11. Key Takeaways

What the Batch Job Does:

  • ✅ Finds subscriptions approaching renewal (within 7-30 days)
  • ✅ Checks Stripe for the latest subscription data
  • ✅ Updates database dates to match Stripe (self-healing)
  • ✅ Records when each subscription was last checked
  • ✅ Handles errors gracefully (retries, fallbacks, continues on errors)

What the Batch Job Does NOT Do:

  • ❌ Does NOT charge customers (Stripe handles billing)
  • ❌ Does NOT create new subscriptions
  • ❌ Does NOT cancel subscriptions
  • ❌ Does NOT send emails or notifications
  • ❌ Does NOT process payments

12. Why Is This Important?

This batch job is critical for maintaining data integrity:

  • Webhook Reliability: Webhooks can fail due to network issues, server downtime, or Stripe outages. The batch job ensures your database stays in sync even when webhooks fail.
  • Manual Changes: If someone manually renews or modifies a subscription in Stripe dashboard, the batch job will detect and sync those changes.
  • Data Consistency: Ensures your database always reflects the true state of subscriptions in Stripe.
  • Proactive Monitoring: By running every 6 hours, it catches discrepancies quickly before they cause issues.

13. Monitoring and Troubleshooting

13.1 How to Check If It's Working

  • Check logs for [SUBSCRIPTION-RENEWAL] entries
  • Query database: SELECT * FROM batch_job_execution WHERE job_type = 'SUBSCRIPTION_RENEWAL' ORDER BY created_at DESC
  • Check reconciliation_status field in membership_subscription table
  • Look for last_reconciliation_at timestamps being updated

13.2 Common Issues

Issue What It Means How to Fix
No subscriptions found No subscriptions match the criteria (all are outside renewal window) This is normal - wait for subscriptions to approach renewal
Stripe API errors Cannot connect to Stripe or rate limited Check Stripe API key, network connectivity, or wait for rate limit to reset
Date mismatches Database dates don't match Stripe dates Batch job will auto-fix this (self-healing). Check logs for "UPDATED" status.
Job not running Scheduled job is disabled or cron expression is wrong Check batch.subscription-renewal.enabled and cron expression

14. Summary

The Subscription Renewal Batch Job is a synchronization and reconciliation tool that:

  1. Runs automatically every 6 hours (or on a custom schedule)
  2. Finds subscriptions that are approaching renewal (within 7-30 days)
  3. Checks Stripe to get the latest subscription data
  4. Updates your database to match Stripe (self-healing)
  5. Handles errors gracefully with retries and fallbacks
  6. Can be triggered manually when needed (e.g., after webhook failures)
Bottom Line: This batch job ensures your database stays in sync with Stripe, fixing any discrepancies automatically. It's like having a maintenance worker who checks all your subscriptions regularly and fixes any data inconsistencies they find.

15. Technical Details (For Developers)

15.1 Code Components

  • SubscriptionRenewalJobConfig.java: Defines the Spring Batch job structure
  • SubscriptionRenewalReader.java: Queries database for subscriptions needing renewal
  • SubscriptionRenewalProcessor.java: Checks Stripe and updates subscription data
  • SubscriptionRenewalWriter.java: Saves updated subscriptions to database
  • BatchJobOrchestrationService.java: Coordinates job execution
  • BatchJobScheduler.java: Handles scheduled execution
  • BatchJobController.java: REST API endpoint for manual triggers

15.2 Database Query Logic

Repository Query (MembershipSubscriptionRepository.java)
@Query("SELECT s FROM MembershipSubscription s " +
       "WHERE s.tenantId = :tenantId " +
       "AND s.subscriptionStatus IN ('ACTIVE', 'TRIALING') " +
       "AND s.cancelAtPeriodEnd = false " +
       "AND (" +
       "  s.currentPeriodEnd <= :renewalDateThreshold " +
       "  OR (s.stripeSubscriptionId IS NOT NULL " +
       "      AND s.currentPeriodEnd <= :extendedDateThreshold)" +
       ") " +
       "ORDER BY s.currentPeriodEnd ASC")
List<MembershipSubscription> findSubscriptionsNeedingRenewal(
    @Param("tenantId") String tenantId,
    @Param("renewalDateThreshold") LocalDate renewalDateThreshold,
    @Param("extendedDateThreshold") LocalDate extendedDateThreshold
);

15.3 Processing Logic Flow

Processor Logic (SubscriptionRenewalProcessor.java)
1. Check if subscription has Stripe ID
   ├─ YES → Fetch from Stripe
   │   ├─ Compare Stripe dates with database dates
   │   ├─ If within renewal window → Process
   │   ├─ Update database dates if different (self-healing)
   │   └─ Set reconciliation_status = "SYNCED" or "UPDATED"
   └─ NO → Use database dates only
       ├─ Check if within renewal window
       └─ Set reconciliation_status = "PROCESSED"
