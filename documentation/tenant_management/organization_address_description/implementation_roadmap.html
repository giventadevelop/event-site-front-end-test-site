Organization Address & Description — Implementation Roadmap (v2.0)

Index: generic_enhancement_prd.html  |  Version: 2.0  |  Date: June 17, 2026

Architecture: tenant_organization is the single write target for description, address, and website. tenant_settings identity columns are deprecated (read fallback until v2.1 DROP).

Phased delivery

Phase 1 — Database
  • Run migrations/001_tenant_org_address_description.sql (org columns + settings→org data migration + deprecate comments).
  • Update baseline Latest_Schema_Post__Blob_Claude_12.sql COMMENTs to match v2.0.
  • Verify on dev/staging: org has migrated data; settings columns still present.
  • Estimate: 0.5 day
Phase 2 — Backend API
  • Extend TenantOrganization entity/DTO/mapper/REST for identity fields.
  • Do not add new identity columns to TenantSettings.
  • Reject or ignore PATCH of deprecated settings identity fields (400 preferred).
  • OpenAPI: mark settings identity fields deprecated.
  • Integration tests: org PATCH; settings PATCH without address succeeds.
  • Deploy API before frontend v2.0 refactor.
  • Estimate: 1–2 days
Phase 3 — Frontend refactor
  • Add src/lib/resolveTenantOrganizationIdentity.ts.
  • TenantOrganizationForm: full identity UI (write path).
  • TenantSettingsForm: remove address/description/city; strip deprecated keys on submit.
  • Update org detail view; slim settings detail view.
  • Wire Footer/contact/about to resolved identity (org-first fallback).
  • Fix organizations/new/page.tsx initialData.
  • Estimate: 1–2 days
Phase 4 — QA & UAT
  • Org CRUD with identity fields.
  • Settings CRUD without touching org identity.
  • Public footer: org data; legacy settings-only tenant shows fallback.
  • Regression: homepage toggles, social links, email/phone from settings unchanged.
  • Estimate: 0.5 day
Phase 5 — v2.1 cleanup (optional, after verification window)
  • Run migrations/002_drop_deprecated_tenant_settings_identity_columns.sql.
  • Remove deprecated fields from TenantSettingsDTO and read fallback (org-only reads).
  • Estimate: 0.5 day

Rollout risk

  • Low: Nullable columns; migration 001 copies settings→org without deleting settings data.
  • Low: Read fallback preserves display until admins edit org record.
  • Medium: Admins who only used settings form must use org form for address updates after v2.0 UI ships — mitigate with info callout on settings page.
