Frontend Stripe Key Security Audit

🔴 SECURITY INCIDENT: Live Stripe API key (sk_live_***oP67X) detected as publicly available online.

Executive Summary

This audit examines the frontend Next.js project for vulnerabilities related to Stripe secret key exposure. The backend Java application stores keys encrypted in the database, but this audit focuses on the frontend codebase.

✅ Good News: Frontend Code is Secure

  • No client-side exposure: STRIPE_SECRET_KEY is only used in server-side code (API routes, server actions)
  • No hardcoded keys: No secret keys are hardcoded in source code
  • Environment files protected: .env.local and .env.production are in .gitignore
  • Only publishable keys in client: Only NEXT_PUBLIC_STRIPE_PUBLISHABLE_KEY (safe) is used in client components

⚠️ Potential Risk: next.config.mjs env Section

Location: next.config.mjs line 165
Issue: STRIPE_SECRET_KEY is declared in the env section

While this variable is NOT prefixed with NEXT_PUBLIC_ (so it won't be bundled in client code), the env section makes variables available at build time, which could expose them in:

  • Build logs (AWS Amplify, CI/CD pipelines)
  • CloudWatch logs if build scripts log environment variables
  • Docker build layers if building container images

2. next.config.mjs env Section ⚠️ MEDIUM RISK

Current Configuration: 

// next.config.mjs (line 165)
env: {
  STRIPE_SECRET_KEY: process.env.STRIPE_SECRET_KEY,
  STRIPE_WEBHOOK_SECRET: process.env.STRIPE_WEBHOOK_SECRET,
  // ... other vars
}

Risk:

  • The env section makes variables available at build time
  • If build scripts or Next.js logs these values, they could appear in build logs
  • While not bundled in client code (no NEXT_PUBLIC_ prefix), build-time exposure is still a risk

Recommendation:

Option 1: Remove from env section (Recommended)

Since STRIPE_SECRET_KEY is only used in server-side code (API routes, server actions), it doesn't need to be in the env section. Server-side code can access process.env directly. 

// REMOVE these lines from next.config.mjs env section:
// STRIPE_SECRET_KEY: process.env.STRIPE_SECRET_KEY,
// STRIPE_WEBHOOK_SECRET: process.env.STRIPE_WEBHOOK_SECRET,

// Keep them only in serverRuntimeConfig (line 26-27):
serverRuntimeConfig: {
  STRIPE_SECRET_KEY: process.env.STRIPE_SECRET_KEY || process.env.AMPLIFY_STRIPE_SECRET_KEY,
  STRIPE_WEBHOOK_SECRET: process.env.STRIPE_WEBHOOK_SECRET || process.env.AMPLIFY_STRIPE_WEBHOOK_SECRET,
}

Prevention Measures

1. Update next.config.mjs

// ✅ DO: Keep secret keys ONLY in serverRuntimeConfig
serverRuntimeConfig: {
  STRIPE_SECRET_KEY: process.env.STRIPE_SECRET_KEY || process.env.AMPLIFY_STRIPE_SECRET_KEY,
  STRIPE_WEBHOOK_SECRET: process.env.STRIPE_WEBHOOK_SECRET || process.env.AMPLIFY_STRIPE_WEBHOOK_SECRET,
},

// ❌ DON'T: Include secret keys in env section
env: {
  // Remove STRIPE_SECRET_KEY and STRIPE_WEBHOOK_SECRET from here
  // Only include NEXT_PUBLIC_* variables that are safe for client
  NEXT_PUBLIC_STRIPE_PUBLISHABLE_KEY: process.env.NEXT_PUBLIC_STRIPE_PUBLISHABLE_KEY,
}

2. Use AWS Secrets Manager (Recommended)

Instead of storing keys in AWS Amplify environment variables, use AWS Secrets Manager: 

# Store in Secrets Manager
aws secretsmanager create-secret \
  --name "stripe/live/secret-key" \
  --secret-string "sk_live_..."

# Reference in amplify.yml build script
version: 1
backend:
  phases:
    build:
      commands:
        - |
          export STRIPE_SECRET_KEY=$(aws secretsmanager get-secret-value \
            --secret-id stripe/live/secret-key \
            --query SecretString --output text)

Can Stripe Tell You the Source?

Stripe typically does NOT reveal exact leak sources for security reasons, but you can ask Stripe Support:

  1. "When was the key first detected as exposed?" - Helps narrow down timeline
  2. "Are there any patterns in how the key was used?" - May indicate source type
  3. "Can you provide geographic/IP information about key usage?" - May indicate if it's automated scanning vs. actual use
  4. "What type of unauthorized activity was detected?" - Helps understand scope

This information can help you correlate with your own logs (AWS Amplify builds, deployment dates, etc.) to identify when/where the leak occurred.
