What is a Test Clock?

A Stripe Test Clock is a special testing tool that allows you to control time in Stripe's test mode. It's like a virtual clock that you can set to any time and advance at will, enabling you to test time-based events without waiting for real time to pass.

Key Concept: Test Clocks let you simulate the passage of time in Stripe test mode. When you advance a Test Clock, Stripe automatically processes all time-based events (subscription renewals, invoice generation, webhooks) as if that time had actually passed.

How Test Clocks Work

  1. Create a Test Clock - You create a virtual clock and set its initial time (usually current time)
  2. Attach Resources - You attach customers, subscriptions, or other resources to the Test Clock
  3. Advance Time - You can advance the clock by any amount (days, weeks, months)
  4. Automatic Processing - Stripe automatically processes all time-based events when the clock advances

What Happens When You Advance a Test Clock?

When you advance a Test Clock, Stripe automatically:

  • ✅ Processes subscription renewals
  • ✅ Generates invoices for upcoming billing periods
  • ✅ Sends webhooks (invoice.payment_succeeded, customer.subscription.updated, etc.)
  • ✅ Updates subscription period dates
  • ✅ Handles trial period expirations
  • ✅ Processes any other time-based events

Test Clocks are the official and recommended way to test time-based events in Stripe test mode. Here's why:

Limitations and Important Notes

⚠️ CRITICAL LIMITATION: Test Clocks ONLY work for subscriptions created AFTER the test clock is set up. You cannot retroactively attach an existing subscription to a test clock.

What This Means

  • ❌ Existing subscriptions cannot use Test Clocks - If a subscription was created before the test clock, it cannot be attached
  • ✅ New subscriptions work perfectly - Subscriptions created after attaching the customer to a test clock will use the test clock
  • ⚠️ Order matters - You must create the test clock and attach the customer BEFORE creating the subscription

Why This Limitation Exists

Stripe needs to know about the test clock at the time of subscription creation so it can properly track time-based events. Once a subscription is created with real time, it cannot be retroactively moved to a test clock.

Customer Attachment Limitation

❌ STRIPE RESTRICTION: You CANNOT attach a customer to a test clock if they have existing subscriptions that were created BEFORE the test clock was created.

Error Message: Received unknown parameter: test_clock

This happens when you try to update a customer with test_clock parameter, but Stripe rejects it because the customer has existing subscriptions created before the test clock.

Why This Happens

Stripe prevents attaching customers to test clocks if they have existing subscriptions because:

  • Existing subscriptions cannot be retroactively attached to test clocks
  • Stripe wants to prevent confusion about which subscriptions use the test clock
  • Only new subscriptions created AFTER the test clock can use it

Customer Selection Issue: Multiple Customers with Same Email

⚠️ Critical Issue: If multiple Stripe customers exist with the same email address, the application may select the WRONG customer (one without a test clock) when processing payments.

The Problem

When the application creates a Stripe Checkout Session or processes a payment, it uses:

stripe.customers.list({ email: "user@example.com", limit: 1 })

This returns the FIRST customer found with that email. If multiple customers exist:

  • ❌ The selection is NOT guaranteed to be the newest customer
  • ❌ The selection is NOT guaranteed to be the one attached to a test clock
  • ❌ The application may pick an old customer without a test clock
  • ❌ New subscriptions will NOT use the test clock if the wrong customer is selected

Step-by-Step: Creating Subscriptions with Test Clocks

Follow these steps to create a subscription that can be advanced using Test Clocks:

1 Create a Test Clock

First, create a Test Clock in Stripe test mode. The clock starts at the current time (or you can set a specific time).

const stripe = require('stripe')(process.env.STRIPE_SECRET_KEY); // Create a test clock const testClock = await stripe.testHelpers.testClocks.create({ name: 'Test Clock for Subscription Testing', frozen_time: Math.floor(Date.now() / 1000), // Current time in Unix timestamp }); console.log('Test Clock created:', testClock.id); console.log('Test Clock time:', new Date(testClock.frozen_time * 1000).toISOString());

Parameters:

  • name - A descriptive name for the test clock
  • frozen_time - Unix timestamp for the initial time (usually current time)

2 Create or Retrieve Customer

You need a Stripe customer. You can create a new one or use an existing customer.

// Create a new customer (or retrieve existing) const customer = await stripe.customers.create({ email: 'test@example.com', name: 'Test Customer', metadata: { test_clock_id: testClock.id, // Store test clock ID for reference }, }); console.log('Customer created:', customer.id);

3 Attach Customer to Test Clock

CRITICAL: Attach the customer to the test clock BEFORE creating the subscription. This ensures the subscription will use the test clock.

// Attach customer to test clock const updatedCustomer = await stripe.customers.update(customer.id, { test_clock: testClock.id, }); console.log('Customer attached to test clock'); console.log('Customer test clock:', updatedCustomer.test_clock);
⚠️ IMPORTANT: The customer MUST be attached to the test clock BEFORE creating the subscription. If you create the subscription first, it cannot be attached to the test clock.

4 Create Subscription

Now create the subscription. Since the customer is attached to the test clock, the subscription will automatically use the test clock.

// Create subscription (will automatically use the test clock) const subscription = await stripe.subscriptions.create({ customer: customer.id, items: [{ price: 'price_1234567890', // Your Stripe Price ID }], payment_behavior: 'default_incomplete', // Or 'default_incomplete' if needed metadata: { test_clock_id: testClock.id, membership_plan_id: '4552', tenant_id: 'tenant_demo_002', }, }); console.log('Subscription created:', subscription.id); console.log('Subscription test clock:', subscription.test_clock); console.log('Period start:', new Date(subscription.current_period_start * 1000).toISOString()); console.log('Period end:', new Date(subscription.current_period_end * 1000).toISOString());
✅ Success: The subscription is now attached to the test clock. You can verify this by checking subscription.test_clock - it should match your test clock ID.

5 Complete Subscription (If Needed)

If the subscription was created with payment_behavior: 'default_incomplete', you may need to complete it by paying its invoice first.

// If subscription is incomplete, complete it by paying the invoice if (subscription.status === 'incomplete') { const invoice = subscription.latest_invoice; if (invoice && typeof invoice !== 'string') { const paidInvoice = await stripe.invoices.pay(invoice.id); console.log('Invoice paid:', paidInvoice.id); // Wait for Stripe to process await new Promise(resolve => setTimeout(resolve, 1000)); // Retrieve updated subscription const updatedSubscription = await stripe.subscriptions.retrieve(subscription.id); console.log('Subscription status:', updatedSubscription.status); } }

6 Advance Time Using Test Clock

Now you can advance the test clock to simulate time passing. This will automatically trigger subscription renewals, invoice generation, and webhooks.

// Advance test clock by 30 days const daysToAdvance = 30; const currentTime = testClock.frozen_time; const targetTime = currentTime + (daysToAdvance * 24 * 60 * 60); const advancedClock = await stripe.testHelpers.testClocks.advance(testClock.id, { frozen_time: targetTime, }); console.log('Test clock advanced by', daysToAdvance, 'days'); console.log('New clock time:', new Date(advancedClock.frozen_time * 1000).toISOString()); // Wait for Stripe to process await new Promise(resolve => setTimeout(resolve, 2000)); // Retrieve updated subscription const updatedSubscription = await stripe.subscriptions.retrieve(subscription.id); console.log('Updated period end:', new Date(updatedSubscription.current_period_end * 1000).toISOString());

What Happens:

  • Stripe automatically processes the subscription renewal
  • New invoice is created and paid (if payment method is attached)
  • Subscription period dates are updated
  • Webhooks are sent (invoice.payment_succeeded, customer.subscription.updated, etc.)

Complete Code Example

Here's a complete example that creates a subscription with a test clock and advances it:

// Step 3: Attach Customer to Test Clock (CRITICAL: Before creating subscription) await stripe.customers.update(customer.id, { test_clock: testClock.id, }); console.log('✅ Customer attached to test clock'); ❓ What is a Stripe Customer ID?

A Stripe Customer ID is NOT an email address. It's a unique identifier that Stripe creates for each customer, in the format cus_xxxxx (e.g., cus_Tf535MtgN6otNK).

  1. Creates or finds a test clock by name - If --reuse-test-clock=true, it searches for an existing test clock with the same name. Otherwise, it creates a new one.

Advancing Time with Test Clocks

Once you have a subscription attached to a test clock (using the setup script above), you can use the expedite-stripe-renewal-test-clock.js script to advance time:

Expedite Script
🔧 Customize Parameters:
node scripts/test-subscription-renewal/expedite-stripe-renewal-test-clock.js --subscription-id=sub_1SeifsK5BrggeAHMBvg2XE93 --days-to-advance=30
Windows Single-Line:
node scripts/test-subscription-renewal/expedite-stripe-renewal-test-clock.js --subscription-id=sub_1SeifsK5BrggeAHMBvg2XE93 --days-to-advance=30
⚙️ Script Parameters
Parameter Required Description
--subscription-id Yes Stripe subscription ID
--days-to-advance No Number of days to advance (default: 30)
🔧 What the Script Does
  1. Retrieves current subscription - Gets the subscription details including status, period dates, and test clock ID
  2. Creates or finds existing test clock - If the subscription is not attached to a test clock, it checks if the customer is attached to one. If not, it creates a new test clock (with a warning that it may not work for existing subscriptions).
  3. Calculates time advancement - Determines the target time based on current test clock time plus days to advance
  4. Advances test clock - Advances the test clock by the specified number of days
  5. Stripe automatically processes subscription renewal - Stripe handles renewals, invoices, and webhooks automatically
  6. Verifies final subscription state - Retrieves the updated subscription and reports the actual days advanced
⭐ Why Test Clocks?
  • ✅ Proper Stripe feature - Official way to test time-based events
  • ✅ Automatic processing - Stripe handles renewals, invoices, and webhooks automatically
  • ✅ Works with incomplete subscriptions - Can advance time even if subscription isn't fully active
  • ✅ Realistic testing - Simulates real-world time passing
  • ✅ Webhook triggers - Automatically sends webhooks as if time actually passed
📊 Script Output

The script provides detailed output including:

  • Current subscription status and period dates
  • Test clock creation/retrieval status
  • Time advancement confirmation
  • Updated subscription period dates
  • Actual days advanced (may differ from requested if subscription renewed at natural billing cycle)
  • Warnings if subscription was created before test clock
  • Next steps for verification
⚠️ IMPORTANT: Test Clocks only work for subscriptions created AFTER the test clock is set up. Run setup-test-clock.js first to set up the test clock and attach the customer before creating the subscription. For existing subscriptions, use advance-existing-subscription.js instead (though it has limitations).
💡 Note: If your subscription is in incomplete status, you may need to activate it first before using test clocks. Test clocks work best with active subscriptions.

Troubleshooting

Problem: Subscription Not Advancing

Symptoms: Test clock advances, but subscription period doesn't change

Possible Causes:

  • Subscription was created BEFORE test clock was set up
  • Customer was not attached to test clock before subscription creation
  • Subscription is in an invalid state (cancelled, incomplete_expired, etc.)

Solution: Create a new subscription with the customer attached to the test clock from the start.

Problem: "Cannot attach customer to test clock"

Symptoms: Error when trying to attach customer to test clock

Possible Causes:

  • Customer already has subscriptions created before test clock
  • Test clock is in an invalid state

Solution: Create a new customer specifically for testing, or cancel existing subscriptions first.

Problem: Webhooks Not Being Sent

Symptoms: Test clock advances, but webhooks aren't received

Possible Causes:

  • Webhook endpoint not configured in Stripe dashboard
  • Webhook endpoint not accessible (for local testing, use Stripe CLI)
  • Subscription didn't actually renew (check subscription status)

Solution: Use Stripe CLI to forward webhooks to your local server, or check webhook configuration in Stripe dashboard.

Best Practices

1. Create Test Clocks for Each Test Suite

Create a separate test clock for each test scenario to avoid conflicts and make debugging easier.

2. Always Attach Customer Before Creating Subscription

The order is critical: Test Clock → Customer → Attach Customer to Test Clock → Create Subscription

3. Use Descriptive Names

Give your test clocks descriptive names so you can identify them later:

const testClock = await stripe.testHelpers.testClocks.create({ name: 'Monthly Subscription Renewal Test - ' + new Date().toISOString(), frozen_time: Math.floor(Date.now() / 1000), });

4. Store Test Clock IDs

Store test clock IDs in metadata or a test database so you can reuse them:

await stripe.subscriptions.update(subscription.id, { metadata: { test_clock_id: testClock.id, test_scenario: 'monthly_renewal', }, });

5. Clean Up Test Clocks

Delete test clocks after testing to keep your Stripe test mode clean:

// Delete test clock (optional - for cleanup) await stripe.testHelpers.testClocks.delete(testClock.id); console.log('Test clock deleted');

6. Wait for Processing

Always wait 1-2 seconds after advancing a test clock for Stripe to process events:

await stripe.testHelpers.testClocks.advance(testClock.id, { frozen_time: targetTime, }); // Wait for Stripe to process await new Promise(resolve => setTimeout(resolve, 2000)); // Now retrieve updated subscription const updatedSubscription = await stripe.subscriptions.retrieve(subscription.id);
