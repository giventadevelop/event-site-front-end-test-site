Executive Summary

This document defines a fee-free manual payment flow for the existing domain-agnostic payment system. The flow allows attendees or donors to pay organizers directly using Zelle, Venmo, Cash App, Cash, Check, and future manual methods. Payments are tracked in the platform with a manual status lifecycle (Requested → Received → Voided/Refunded) and produce QR codes only after the organizer confirms receipt. This flow must coexist with Stripe-based checkout while remaining selectable per event.

Key Concept: Manual payments are not processed by Stripe or other gateways. The platform records requests and confirmations while the actual fund transfer happens outside the system.

2. Goals & Non-Goals

Goals

  • Enable fee-free payment options: Zelle, Venmo, Cash App, Cash, Check, and extensible manual methods.
  • Provide a clearly separate flow from Stripe payments, while keeping a consistent UI.
  • Allow per-event configuration: Stripe-only, Manual-only, or Hybrid.
  • Track all manual payments with a status lifecycle and admin verification.
  • Issue ticket QR codes only after payment is marked Received.
  • Provide admin and organizer reporting for requested and received payments.

Non-Goals

  • Automated settlement or webhook confirmation from Zelle/Venmo/Cash App.
  • Processing any funds through Stripe or other gateways for manual payments.
  • Replacing current Stripe checkout for events that require instant confirmation.

3. Scope & Event Configuration

Applicability

  • All events can optionally enable manual payments.
  • Fundraisers and donation-based events default to Hybrid (Stripe + Manual).
  • Ticketed events can be Manual-only for invite-only or organizer-verified workflows.

Configuration Inputs

  • Event-level flag: payment_flow_mode = STRIPE_ONLY, MANUAL_ONLY, HYBRID.
  • Manual method enablement per tenant via payment_provider_config records.
  • Organizer-provided instructions (handles, QR codes, mailing address).

4. User Journeys

Ticket Purchase (Manual)

  1. User selects tickets and chooses a manual payment method.
  2. UI shows amount due, organizer instructions, and required memo format.
  3. User confirms they will pay externally.
  4. System creates a Manual Payment Request (status: Requested).
  5. User sees “Pending manual confirmation” and no QR code yet.

Fundraising / Donation (Manual)

  1. User enters donation amount and selects a manual method.
  2. System records payment request and displays donor instructions.
  3. Organizer marks payment received; donor receipt/thank-you email sent.

Organizer / Admin

  • View manual payment requests by event and method.
  • Mark status as Received or Voided/Refunded.
  • Trigger QR issuance and ticket inventory updates when Received.
  • Export reports for reconciliation and accounting.

5. Manual Payment Status Lifecycle

Status Definition System Behavior
Requested Manual payment created, funds not confirmed. Tickets held in pending state; QR code not issued.
Received Organizer verifies payment receipt. Tickets confirmed, QR code generated, attendee notified.
Voided/Refunded Request cancelled or refunded. Tickets released, payment request archived with reason.

Status changes are manual and audit-logged. The lifecycle is intentionally independent from Stripe payment status codes.

7. Backend Requirements

Service Layer

  • Create ManualPaymentService with CRUD for manual payment requests.
  • Integrate status updates with ticket inventory + QR generation.
  • Audit log each status change with actor and timestamp.
  • Store proof-of-payment uploads in S3 under events/{eventId}/manual-payments/proof-of-payment/.

REST APIs (Example Endpoints)

  • POST /api/manual-payments → Create manual payment request.
  • GET /api/manual-payments?eventId.equals= → List requests for event/admin.
  • PATCH /api/manual-payments/{id} → Update status to Received/Voided.
  • GET /api/manual-payment-methods?tenantId.equals= → List available manual methods.
  • POST /api/manual-payments/{id}/proof → Upload proof-of-payment file.

Integration with Ticketing

  • Manual payment requests should reference event_ticket_transaction or user_payment_transaction.
  • On status Received, call QR issuance workflow and set ticket status to Confirmed.
  • On Voided/Refunded, release inventory and log reason.

Standards

  • Follow cursor_rules/nextjs_api_routes.mdc for proxy routing from the frontend.
  • Match contracts and naming patterns with documentation/Swagger_API_Docs/api-docs.json.
  • Use tenant_id injection and criteria queries (tenantId.equals).

8. Frontend Requirements

Checkout Experience

  • Payment method selector that lists Stripe and Manual options based on event configuration.
  • Manual options include: Zelle, Venmo, Cash App, Cash, Check, Other.
  • Manual selection shows organizer-provided instructions and required memo fields.
  • Submission creates a Manual Payment Request via proxy API.
  • Optional proof-of-payment upload (image/PDF) after request is created.
  • Pending requests never expire automatically; users can cancel a request.

Status & Ticketing UI

  • Pending state banner: “Payment requested. Awaiting organizer confirmation.”
  • QR code panel hidden until status = Received.
  • Show audit timestamp and method in attendee dashboard.

Admin & Organizer Console

  • Manual payments tab with filters (event, method, status).
  • Action buttons: Mark Received, Void/Refund, Add Notes.
  • Follow UI standards in @ui_style_guide.mdc and admin button patterns.

9. Batch Jobs & Reconciliation

  • Summary Aggregation Job: nightly totals grouped by event, status, and payment method.
  • Reporting Table: store daily snapshots for admin dashboard reporting.
  • Reconciliation Export: optional CSV output for finance teams.
  • Stale Cleanup: no auto-void; pending requests remain open until manually cancelled.
  • Batch job implementations live in event-site-manager-batch-jobs.

10. Security & Compliance

  • Manual payment instructions must never store sensitive banking data beyond organizer-provided handles.
  • Audit logs required for all status changes.
  • Role-based access: only admin/organizer can mark Received or Void.
  • Ensure manual payment records respect tenant isolation.

12. Decisions

  • Partial payment tracking: Not supported.
  • Expiration window: Open-ended; requests can be cancelled manually.
  • Proof-of-payment uploads: Supported. Store in S3 under the event path.
  • Admin action webhooks: Not required for v1.
