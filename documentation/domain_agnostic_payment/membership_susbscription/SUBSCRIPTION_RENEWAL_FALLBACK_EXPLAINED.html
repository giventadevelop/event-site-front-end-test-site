Subscription Renewal Fallback Mode Explained

Configuration: subscription.renewal.use-database-fallback
Default Value: false (production-safe)
Purpose: Controls what happens when database dates and Stripe dates don't match

1. What Is "Fallback Mode"?

The batch job checks subscriptions by comparing dates from two sources:

  • Database: Your local database's current_period_end date
  • Stripe: Stripe's current_period_end date (fetched via API)

Fallback mode determines what happens when these two dates don't match.

When Do Dates Mismatch?
  • Webhook from Stripe failed to update your database
  • Someone manually changed dates in Stripe dashboard
  • Testing scripts advanced Stripe dates (expedite renewal)
  • System downtime prevented webhook processing
  • Time zone or synchronization issues

2. The Two Modes Explained

2.1 Production Mode (use-database-fallback = false)

✅ Production Mode: Stripe is Always Right

When use-database-fallback = false (default):

  • Stripe is the source of truth - Stripe's dates are always correct
  • If database date says "renewal in 3 days" but Stripe says "renewal in 20 days" → Skip the subscription
  • If database date says "renewal in 20 days" but Stripe says "renewal in 3 days" → Process the subscription (update database from Stripe)
  • The batch job will update your database to match Stripe (self-healing)

2.2 Testing Mode (use-database-fallback = true)

⚠️ Testing Mode: Database Can Override Stripe

When use-database-fallback = true:

  • Database dates can be used even when Stripe dates differ
  • If database date says "renewal in 3 days" but Stripe says "renewal in 20 days" → Process anyway (use database date)
  • This is useful for testing when you've manually advanced Stripe dates
  • DANGEROUS in production - can cause incorrect renewal processing

3. Real-World Scenarios

Scenario 1: Webhook Failed (Production Mode)

Situation: Subscription renewed in Stripe, but webhook failed
Before Batch Job Runs:
  • Database: current_period_end = 2025-01-15 (expired yesterday!)
  • Stripe: current_period_end = 2025-02-15 (renewed for another month)
  • Difference: 31 days
Batch Job Decision (fallback = false):
  • Stripe date (2025-02-15) is outside 7-day window
  • Database date (2025-01-15) is within 7-day window
  • Result: Skip subscription (Stripe is source of truth)
  • Action: Logs warning, waits for next run or manual sync
Why Skip? Because Stripe says the subscription doesn't need renewal for another 30 days. The batch job trusts Stripe over the database. However, the batch job will still update the database dates from Stripe during processing (self-healing), so on the next run, the dates will match.

Scenario 2: Testing with Expedited Renewal (Testing Mode)

Situation: Testing renewal flow, manually advanced Stripe dates
Before Batch Job Runs:
  • Database: current_period_end = 2025-12-31 (within 7 days)
  • Stripe: current_period_end = 2026-01-25 (manually advanced for testing)
  • Difference: 25 days
  • Fallback Mode: true (testing environment)
Batch Job Decision (fallback = true):
  • Database date (2025-12-31) is within 7-day window
  • Stripe date (2026-01-25) is outside 7-day window
  • Result: Process subscription anyway (use database date)
  • Action: Processes renewal based on database date, updates database from Stripe
Why Process? In testing mode, you want to test the renewal flow even when Stripe dates are artificially advanced. The batch job will process the subscription based on the database date, allowing you to test renewal logic without waiting for real Stripe renewals.

Scenario 3: Normal Operation (Both Modes)

Situation: Dates match (normal case)
Before Batch Job Runs:
  • Database: current_period_end = 2025-01-05 (3 days away)
  • Stripe: current_period_end = 2025-01-05 (matches database)
  • Difference: 0 days (perfect sync!)
Batch Job Decision (any mode):
  • Both dates are within 7-day window
  • Result: Process subscription ✅
  • Action: Updates reconciliation status, syncs dates
Note: When dates match, fallback mode doesn't matter - the batch job processes normally.

4. Decision Logic Flow

Batch Job Decision Tree
1. Fetch subscription from database
2. Check if subscription has Stripe ID
   ├─ NO → Use database dates only (process if within 7 days)
   └─ YES → Fetch from Stripe
       ├─ Compare Stripe date with renewal window (7 days)
       │   ├─ Stripe date WITHIN window → Process ✅
       │   └─ Stripe date OUTSIDE window → Check database date
       │       ├─ Database date WITHIN window → Check fallback mode
       │       │   ├─ fallback = false → Skip ❌ (Stripe is source of truth)
       │       │   └─ fallback = true → Process ✅ (Use database date)
       │       └─ Database date OUTSIDE window → Skip ❌
       └─ Update database dates from Stripe (self-healing)

5. Why Is Fallback Mode Dangerous in Production?

⚠️ Production Risk: Incorrect Renewal Processing

Problem Scenario:

  1. Customer's subscription actually renewed in Stripe (new period: Jan 15 - Feb 15)
  2. Webhook failed, so database still shows old date (Jan 15)
  3. Database says "renewal in 3 days" (based on old date)
  4. Stripe says "renewal in 30 days" (based on actual renewal)
  5. With fallback = true: Batch job processes based on database date
  6. Result: System thinks subscription needs renewal when it's already renewed!

Consequences:

  • ❌ Incorrect renewal notifications sent to customers
  • ❌ System may try to process renewal that already happened
  • ❌ Data inconsistency between your system and Stripe
  • ❌ Customer confusion ("I already paid, why am I getting renewal notices?")
  • ❌ Potential double-billing attempts

6. When Should You Use Fallback Mode?

Environment Fallback Setting Why
Production false ✅ Stripe is the source of truth. Always trust Stripe dates to prevent incorrect processing.
Staging false ✅ Should mirror production behavior. Test with real Stripe dates.
Development true ⚠️ Useful when testing renewal flows with expedited Stripe dates. Allows testing without waiting for real renewals.
Local Testing true ⚠️ When manually advancing Stripe dates for testing, fallback mode allows processing based on database dates.

7. Configuration Examples

7.1 Production Configuration

application-prod.yml
subscription:
  renewal:
    use-database-fallback: false  # ✅ Production: Always trust Stripe

7.2 Development Configuration

application-dev.yml
subscription:
  renewal:
    use-database-fallback: true  # ⚠️ Dev: Allow database fallback for testing

7.3 Environment Variable

Environment Variable
# Set in environment or .env file
USE_DATABASE_FALLBACK=false  # Production
USE_DATABASE_FALLBACK=true   # Development/Testing

8. What Happens in Each Mode (Code Logic)

8.1 Production Mode (fallback = false)

SubscriptionRenewalProcessor.java - Production Logic
if (database date within window BUT Stripe date outside window) {
    log.warn("⚠️ DATE MISMATCH DETECTED (Production Mode)");
    log.warn("Skipping subscription (Stripe is source of truth)");
    return false; // Skip subscription
}

8.2 Testing Mode (fallback = true)

SubscriptionRenewalProcessor.java - Testing Logic
if (database date within window BUT Stripe date outside window) {
    log.warn("⚠️ DATE MISMATCH DETECTED (Testing Mode)");
    log.warn("Using database date for processing");
    return true; // Process subscription using database date
}

9. Self-Healing Behavior

Important: Self-Healing Still Happens

Regardless of fallback mode, the batch job will always update database dates from Stripe during processing. This is the "self-healing" feature:

  • Even if fallback mode processes a subscription based on database dates, it will still update the database to match Stripe
  • On the next batch job run, dates will match and fallback mode won't matter
  • This ensures your database eventually syncs with Stripe

10. Summary

Key Points:

  • Fallback mode controls what happens when database and Stripe dates don't match
  • Production (false): Always trust Stripe dates - skip if Stripe says not ready
  • Testing (true): Can use database dates even when Stripe dates differ
  • Danger: Using fallback mode in production can cause incorrect renewal processing
  • Self-healing: Database dates are always updated from Stripe during processing

⚠️ Remember:

In production, always set use-database-fallback = false. Stripe is the payment processor and source of truth. Trusting database dates over Stripe can lead to incorrect renewal processing, customer confusion, and potential billing issues.
