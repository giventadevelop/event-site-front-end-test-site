Overview

Stripe offers three main products for handling payments and subscriptions. Understanding the differences is crucial for choosing the right solution for your use case.

⚠️ Important: There is NO standalone "Stripe Billing UI Kit" as a single React component library. The term typically refers to using Stripe Checkout + Customer Portal together, which are separate hosted products.
✅ Quick Summary:
  • Stripe Checkout = Hosted payment page (redirect required)
  • Stripe Customer Portal = Hosted subscription management page (redirect required)
  • Stripe Elements = React components for embedded payment forms (no redirect)

1. Stripe Checkout (Hosted Payment Page)

🎯 Stripe Checkout

What it is: A complete, pre-built payment page hosted by Stripe on their servers.

Key Characteristics

  • Hosted by Stripe: Payment page runs on Stripe's servers, not your website
  • Redirect Required: Users are redirected from your site to Stripe's payment page
  • Fully Managed: Stripe handles all UI, payment processing, error handling, and mobile optimization
  • Automatic PRB Support: Apple Pay and Google Pay buttons appear automatically on supported devices
  • Limited Customization: You can customize colors, logo, and some text, but not the layout

User Flow

User clicks "Subscribe" on your website ↓ Your server creates Checkout Session ↓ User is redirected to: checkout.stripe.com/pay/... ↓ User completes payment on Stripe's page ├─ Apple Pay button (if Safari + Apple Wallet) ├─ Google Pay button (if Chrome + Google Pay) └─ Card form (always available) ↓ Stripe redirects back to your success page ↓ Your server retrieves subscription from Checkout Session

Code Example

// Server-side: Create Checkout Session const session = await stripe.checkout.sessions.create({ mode: 'subscription', line_items: [{ price: 'price_1234567890', quantity: 1, }], success_url: 'https://yoursite.com/success?session_id={CHECKOUT_SESSION_ID}', cancel_url: 'https://yoursite.com/cancel', metadata: { userId: 'user_123', planId: 'plan_456', }, }); // Redirect user to Stripe's hosted page return { sessionUrl: session.url }; // Frontend: window.location.href = sessionUrl;

Desktop PRB Buttons Support

✅ YES - Stripe Checkout Shows PRB Buttons on Desktop Automatically:
  • Apple Pay Button: Appears automatically on Safari (macOS) when customer has Apple Pay set up
  • Google Pay Button: Appears automatically on Chrome/Edge/Firefox (Desktop) when customer has Google Pay set up
  • Link Button: May appear if Link payment method is enabled in your Stripe Dashboard
  • Automatic Detection: Stripe detects browser capabilities and customer's saved payment methods
  • No Code Required: All PRB buttons appear automatically - no custom implementation needed

Note: Similar to the buttons shown in your current Stripe Elements implementation, but these appear automatically on Stripe's hosted Checkout page without any custom code.

Pros

  • ✅ Zero frontend code - Stripe handles everything
  • ✅ Automatic mobile optimization - Stripe optimizes for all devices
  • ✅ Automatic PRB buttons - Apple Pay/Google Pay appear automatically (mobile + desktop)
  • ✅ Desktop PRB support - Google Pay on Chrome, Apple Pay on Safari (macOS)
  • ✅ PCI compliance handled - No sensitive data touches your servers
  • ✅ Automatic subscription creation - Stripe creates subscription when payment succeeds
  • ✅ Low maintenance - Stripe updates automatically
  • ✅ 86% code reduction compared to custom implementation

Cons

  • ❌ Redirect required - Users leave your site temporarily
  • ❌ Limited UI control - Can't fully customize layout/design
  • ❌ Less seamless UX - Not as integrated as inline payment

Best For

  • ✅ Standard subscription flows
  • ✅ When you want minimal code and maintenance
  • ✅ When mobile optimization is important
  • ✅ When you want automatic PRB support

2. Stripe Customer Portal (Hosted Subscription Management)

🔐 Stripe Customer Portal

What it is: A hosted page where customers can manage their subscriptions, payment methods, and billing history.

Key Characteristics

  • Hosted by Stripe: Management page runs on Stripe's servers
  • Redirect Required: Users are redirected from your site to Stripe's portal
  • Fully Managed: Stripe handles all UI and functionality
  • Subscription Management: Customers can update payment methods, cancel subscriptions, view invoices
  • No Customization: Stripe controls the entire UI (you can only configure what features are enabled)

User Flow

User clicks "Manage Subscription" on your website ↓ Your server creates Portal Session ↓ User is redirected to: billing.stripe.com/p/session/... ↓ User manages subscription on Stripe's page ├─ Update payment method ├─ View billing history ├─ Cancel subscription └─ Update billing address ↓ Stripe redirects back to your return URL

Code Example

// Server-side: Create Portal Session const session = await stripe.billingPortal.sessions.create({ customer: 'cus_1234567890', return_url: 'https://yoursite.com/account', }); // Redirect user to Stripe's portal return { portalUrl: session.url }; // Frontend: window.location.href = portalUrl;

Features

  • ✅ Update payment methods
  • ✅ View billing history and invoices
  • ✅ Cancel subscriptions
  • ✅ Update billing address
  • ✅ Download invoices

Desktop PRB Buttons Support

✅ YES - Customer Portal Can Show PRB Buttons When Updating Payment Methods:
  • Apple Pay Button: May appear when updating payment method on Safari (macOS)
  • Google Pay Button: May appear when updating payment method on Chrome/Edge/Firefox (Desktop)
  • Conditional: PRB buttons appear based on browser support and customer's saved payment methods
  • Automatic Detection: Stripe handles detection automatically - no code needed

Note: Customer Portal is primarily for subscription management, not initial payments. PRB buttons appear when customers update their payment methods.

Pros

  • ✅ Zero frontend code - Stripe handles everything
  • ✅ Complete subscription management - All features included
  • ✅ PCI compliance handled - No sensitive data touches your servers
  • ✅ Low maintenance - Stripe updates automatically
  • ✅ Secure by default - Stripe handles all security

Cons

  • ❌ Redirect required - Users leave your site
  • ❌ No UI customization - Stripe controls everything
  • ❌ Less integrated UX - Not seamless with your site

Best For

  • ✅ Standard subscription management needs
  • ✅ When you want minimal code and maintenance
  • ✅ When you don't need custom UI for subscription management

3. Stripe Elements (React Components)

⚛️ Stripe Elements

What it is: React/JavaScript components that you embed directly in your website for payment collection.

Key Characteristics

  • Embedded in Your Site: Payment form runs on your website, not Stripe's servers
  • No Redirect: Users stay on your site throughout the payment process
  • Full UI Control: You control the layout, styling, and user experience
  • Custom Implementation Required: You must handle payment processing, error handling, and subscription creation
  • PRB Support Available: Can add Apple Pay/Google Pay buttons using ExpressCheckoutElement
  • More Code Required: Significantly more code than Checkout

User Flow

User visits your subscription page ↓ Stripe Elements form appears on your page ├─ Card input field ├─ Apple Pay button (if ExpressCheckoutElement added) ├─ Google Pay button (if ExpressCheckoutElement added) └─ Submit button ↓ User enters payment details (stays on your page) ↓ Your frontend calls Payment Intent API ↓ Payment processed via Stripe API ↓ Your server creates subscription ↓ User sees success message on your page

Code Example

// Frontend: React component import { PaymentElement, ExpressCheckoutElement } from '@stripe/react-stripe-js'; function PaymentForm() { const stripe = useStripe(); const elements = useElements(); const handleSubmit = async (e) => { e.preventDefault(); // Confirm payment with Stripe const { error, paymentIntent } = await stripe.confirmPayment({ elements, confirmParams: { return_url: 'https://yoursite.com/success', }, }); // Then create subscription on your server await fetch('/api/create-subscription', { method: 'POST', body: JSON.stringify({ paymentIntentId: paymentIntent.id }), }); }; return ( <form onSubmit={handleSubmit}> <ExpressCheckoutElement /> <PaymentElement /> <button type="submit">Subscribe</button> </form> ); }

Pros

  • ✅ Full UI control - Complete customization of layout and design
  • ✅ Seamless UX - Users never leave your site
  • ✅ Brand consistency - Matches your site's design perfectly
  • ✅ Flexible - Can implement custom business logic

Cons

  • ❌ More code required - Significantly more implementation work
  • ❌ Manual PRB setup - Must add ExpressCheckoutElement for Apple Pay/Google Pay
  • ❌ Manual subscription creation - Must create subscription after payment
  • ❌ More error handling - Must handle all edge cases yourself
  • ❌ More maintenance - Must update code when Stripe changes APIs
  • ❌ Mobile optimization - Must handle responsive design yourself

Best For

  • ✅ When you need full UI control and brand consistency
  • ✅ When you want seamless UX (no redirects)
  • ✅ When you have complex custom business logic
  • ✅ When you have design requirements that Checkout can't meet

When to Use Each Product

Use Stripe Checkout When:

  • ✅ You want minimal code and maintenance
  • ✅ You need automatic mobile optimization
  • ✅ You want automatic Apple Pay/Google Pay support
  • ✅ You're okay with redirecting users to Stripe's page
  • ✅ You want automatic subscription creation
  • ✅ Standard subscription flows are sufficient

Use Stripe Customer Portal When:

  • ✅ You want customers to manage their subscriptions
  • ✅ You want minimal code for subscription management
  • ✅ You're okay with redirecting users to Stripe's page
  • ✅ Standard subscription management features are sufficient

Use Stripe Elements When:

  • ✅ You need full UI control and brand consistency
  • ✅ You want seamless UX (no redirects)
  • ✅ You have complex custom business logic
  • ✅ You have specific design requirements
  • ✅ You're willing to write and maintain more code
