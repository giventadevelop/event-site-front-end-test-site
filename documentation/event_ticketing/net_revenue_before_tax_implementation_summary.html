Net Revenue Before Tax Implementation Summary

Document Date: January 11, 2026

Purpose: Analyze whether adding "Net Revenue Before Tax" requires changes across all layers (database, batch job, backend, frontend) and provide implementation summary.

Executive Summary

Question: Do we need to make changes in all layers (DB, batch job, backend, frontend) to display "Net Revenue Before Tax"?

Answer: It depends on the definition and whether you want to store it or calculate on-the-fly.

Recommendation: Calculate on-the-fly in analytics queries (no database changes needed). If performance becomes an issue, add a database field and populate via batch job.

1. Understanding "Net Revenue Before Tax"

1.1 Current Financial Fields

Field Description Example ($1.00 ticket + $0.08 tax)
final_amount Total amount customer paid (including tax if applicable) $1.08
stripe_fee_amount Stripe processing fee $0.33
stripe_amount_tax Tax collected by Stripe (if Stripe Tax enabled) $0.08
net_payout_amount Net payout to bank = final_amount - stripe_fee_amount - stripe_amount_tax $0.67 ($1.08 - $0.33 - $0.08)
platform_fee_amount Platform fee (deducted by your platform, not Stripe) $0.05

1.2 Definition: "Net Revenue Before Tax"

Possible Interpretations:

  1. Net Payout Excluding Tax Component:
    • Formula: final_amount - stripe_fee_amount
    • Meaning: What you'd receive if tax wasn't collected
    • Example: $1.08 - $0.33 = $0.75
  2. Net Payout Plus Tax (Add Tax Back):
    • Formula: net_payout_amount + stripe_amount_tax
    • Meaning: Net payout with tax added back
    • Example: $0.67 + $0.08 = $0.75
  3. Revenue Before All Deductions:
    • Formula: final_amount
    • Meaning: Gross revenue (what customer paid)
    • Example: $1.08

Most Likely Meaning: Interpretation #1 or #2 (both result in the same value: final_amount - stripe_fee_amount)

Financial Flow Example:

Customer Payment: $1.08 (includes $1.00 ticket + $0.08 tax)

Stripe Processing:

  • Stripe Fee: $0.33
  • Stripe Tax: $0.08 (collected from customer, passed through)

Net Payout to Bank: $1.08 - $0.33 - $0.08 = $0.67

Net Revenue Before Tax: $1.08 - $0.33 = $0.75 (or $0.67 + $0.08 = $0.75)

This represents what you'd receive if tax wasn't part of the transaction.

2. Implementation Options

2.1 Option A: Calculate On-the-Fly (Recommended)

Approach: Calculate in Analytics/Reporting Queries

Formula: net_revenue_before_tax = final_amount - stripe_fee_amount

OR: net_revenue_before_tax = net_payout_amount + stripe_amount_tax

Required Changes:

Layer Changes Required Complexity Estimated Time
Database ✅ NO CHANGES - Use existing fields None 0 hours
Batch Job ✅ NO CHANGES - Not needed None 0 hours
Backend API ✅ NO CHANGES - Calculation in service layer None 0 hours
Backend Service 🔧 MINOR CHANGES - Add calculation in analytics service Low 1-2 hours
Frontend 🔧 MINOR CHANGES - Display new metric in analytics page Low 1-2 hours

Implementation Example:

// In Backend Service (Sales Analytics) public SalesMetrics calculateSalesMetrics(List<EventTicketTransaction> transactions) { // ... existing calculations ... // NEW: Calculate net revenue before tax BigDecimal netRevenueBeforeTax = transactions.stream() .filter(t -> t.getStatus().equals("COMPLETED")) .map(t -> { BigDecimal finalAmount = t.getFinalAmount() != null ? t.getFinalAmount() : BigDecimal.ZERO; BigDecimal stripeFee = t.getStripeFeeAmount() != null ? t.getStripeFeeAmount() : BigDecimal.ZERO; // Net revenue before tax = final_amount - stripe_fee_amount return finalAmount.subtract(stripeFee); }) .reduce(BigDecimal.ZERO, BigDecimal::add); // OR using net_payout_amount (if available): BigDecimal netRevenueBeforeTax = transactions.stream() .filter(t -> t.getStatus().equals("COMPLETED")) .map(t -> { BigDecimal netPayout = t.getNetPayoutAmount() != null ? t.getNetPayoutAmount() : BigDecimal.ZERO; BigDecimal stripeTax = t.getStripeAmountTax() != null ? t.getStripeAmountTax() : BigDecimal.ZERO; // Net revenue before tax = net_payout_amount + stripe_amount_tax return netPayout.add(stripeTax); }) .reduce(BigDecimal.ZERO, BigDecimal::add); metrics.setNetRevenueBeforeTax(netRevenueBeforeTax); return metrics; }

Pros:

  • ✅ No database schema changes
  • ✅ No batch job changes
  • ✅ Always accurate (calculated from source data)
  • ✅ No data synchronization issues
  • ✅ Fast to implement

Cons:

  • ⚠️ Must calculate every time (minor performance impact)
  • ⚠️ Calculation logic must be consistent across all queries

2.2 Option B: Store in Database (If Performance Becomes Issue)

Approach: Add Database Field and Populate via Batch Job

Formula: net_revenue_before_tax = final_amount - stripe_fee_amount

Required Changes:

Layer Changes Required Complexity Estimated Time
Database 🔧 SCHEMA CHANGE - Add net_revenue_before_tax numeric(21,2) NULL Low 1 hour (migration script)

Batch Job 🔧 CALCULATE & STORE - Calculate and update field Medium 2-3 hours

Frontend 🔧 DISPLAY METRIC - Display new metric in analytics page Low 1-2 hours

Implementation Steps:

  1. Database Migration:
    ALTER TABLE event_ticket_transaction ADD COLUMN net_revenue_before_tax numeric(21,2) NULL;
  2. Backend Entity Update:
    // In EventTicketTransaction.java @Column(name = "net_revenue_before_tax", precision = 21, scale = 2) private BigDecimal netRevenueBeforeTax;
  3. Batch Job Update:
    // In StripeFeesTaxUpdateService.java BigDecimal finalAmount = transaction.getFinalAmount(); BigDecimal stripeFee = result.getFee(); BigDecimal netRevenueBeforeTax = finalAmount.subtract(stripeFee != null ? stripeFee : BigDecimal.ZERO); // Update payload updatePayload.put("netRevenueBeforeTax", netRevenueBeforeTax);
  4. Backend API Update:
    // In EventTicketTransactionDTO.java private BigDecimal netRevenueBeforeTax; // In PATCH endpoint - already accepts all fields via merge-patch

Pros:

  • ✅ Pre-calculated for fast queries
  • ✅ Consistent value across all reports
  • ✅ Can track historical values

Cons:

  • ⚠️ Requires database schema change
  • ⚠️ Requires batch job update
  • ⚠️ Must ensure batch job populates field correctly
  • ⚠️ More complex implementation

3. Recommended Approach

✅ Recommendation: Option A (Calculate On-the-Fly)

Rationale:

  • ✅ No database changes needed - Use existing fields
  • ✅ No batch job changes needed - Calculation is simple
  • ✅ Fast implementation - Only backend service and frontend changes
  • ✅ Always accurate - Calculated from source data
  • ✅ No data synchronization issues - No stored value to maintain

When to Consider Option B:

  • If analytics queries become slow with large datasets
  • If you need to track historical "before tax" values separately
  • If you need to query/filter by this value frequently
