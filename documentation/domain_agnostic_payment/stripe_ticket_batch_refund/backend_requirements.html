Backend Requirements: Stripe Ticket Batch Refund

This document specifies backend service and API requirements to support batch refund functionality in the Spring Boot service (E:\project_workspace\malayalees-us-site-boot). The implementation follows the pattern established by the existing Stripe Fees and Tax Update batch job endpoint.

1. REST Endpoint

1.1 Endpoint Specification

  • URL: POST /api/cron/stripe-ticket-batch-refund
  • Authentication: Service JWT (Bearer token in Authorization header)
  • Content-Type: application/json
  • Response Code: 202 Accepted (job started asynchronously)
  • Error Codes:
    • 400 Bad Request - Invalid request parameters
    • 401 Unauthorized - Missing or invalid JWT
    • 403 Forbidden - Tenant mismatch or insufficient permissions
    • 404 Not Found - Event not found
    • 500 Internal Server Error - Server error

2. Controller Implementation

2.1 Controller Class

  • Location: com.example.controller.CronController or similar
  • Method: triggerStripeTicketBatchRefund()
  • Annotations:
    • @PostMapping("/api/cron/stripe-ticket-batch-refund")
    • @PreAuthorize("hasRole('ADMIN')") or similar authorization
    • @Valid @RequestBody StripeTicketBatchRefundRequest request

2.2 Controller Logic

  1. Validate request: eventId is required and not null
  2. Extract tenantId from JWT context if not provided in request
  3. Validate event exists and belongs to tenant
  4. Query eligible tickets count (for estimatedRecords)
  5. Trigger batch job service asynchronously
  6. Generate unique job ID (format: stripe-batch-refund-{timestamp}-{eventId})
  7. Return response with job ID and status (HTTP 202 Accepted)

3. Service Layer

3.1 Service Interface

  • Interface: StripeTicketBatchRefundService
  • Method: triggerBatchRefundJob(StripeTicketBatchRefundRequest request)
  • Return: StripeTicketBatchRefundResponse

3.2 Service Implementation

  • Class: StripeTicketBatchRefundServiceImpl
  • Dependencies:
    • EventTicketTransactionRepository (for querying tickets)
    • EventDetailsRepository (for validating event)
    • BatchJobService (for triggering batch job)
    • TenantContextService (for tenant validation)

3.3 Service Methods

  • validateRequest(): Validate eventId, tenantId, event existence
  • countEligibleTickets(): Count tickets matching refund criteria
  • triggerBatchJob(): Submit job to batch job service
  • generateJobId(): Generate unique job identifier
  • estimateCompletionTime(): Calculate estimated completion based on ticket count

4. Database Query

4.1 Eligible Tickets Query

SELECT COUNT(*) FROM event_ticket_transaction
WHERE event_id = :eventId
  AND tenant_id = :tenantId
  AND stripe_payment_intent_id IS NOT NULL
  AND status != 'REFUNDED'
  AND stripe_payment_status IN ('succeeded', 'paid')
  AND (startDate IS NULL OR purchase_date >= :startDate)
  AND (endDate IS NULL OR purchase_date <= :endDate);

4.2 Repository Method

  • Method: countEligibleTicketsForRefund(Long eventId, String tenantId, LocalDateTime startDate, LocalDateTime endDate)
  • Return: Long (count of eligible tickets)
  • Query: Use @Query annotation with JPQL or native query

5. Batch Job Integration

5.1 Job Submission

  • Submit job to batch job service with job parameters
  • Job parameters:
    • eventId - Event ID to process
    • tenantId - Tenant ID for filtering
    • startDate - Optional start date filter
    • endDate - Optional end date filter
    • jobId - Unique job identifier
  • Reference existing batch job submission pattern (Stripe Fees and Tax Update)

5.2 Job Status

  • Return job status as STARTED immediately
  • Batch job service will update status as processing progresses
  • Status values: STARTED, IN_PROGRESS, COMPLETED, FAILED

6. Validation & Error Handling

6.1 Request Validation

  • eventId: Required, must be positive integer
  • tenantId: Optional, validated against JWT context if not provided
  • startDate/endDate: Optional, must be valid ISO 8601 format, startDate <= endDate

6.2 Business Validation

  • Event must exist in database
  • Event must belong to the specified tenant
  • At least one eligible ticket must exist (or return informative message if none)

6.3 Error Responses

Error Condition HTTP Status Response Body
Missing eventId 400 Bad Request {"error": "eventId is required"}
Invalid eventId 400 Bad Request {"error": "Invalid eventId format"}
Event not found 404 Not Found {"error": "Event not found"}
Tenant mismatch 403 Forbidden {"error": "Event does not belong to tenant"}
No eligible tickets 200 OK {"message": "No eligible tickets found for refund"}
