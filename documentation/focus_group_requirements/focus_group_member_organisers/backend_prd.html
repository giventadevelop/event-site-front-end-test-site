Requirement

Support (1) self-service join: an authenticated user can create a focus group membership for themselves (join); (2) leave: a user can remove their own membership; (3) role values: extend or document role to include EXECUTIVE and ORGANISER for executive/organising committee; (4) committee list: allow the frontend to request members filtered by role (e.g. EXECUTIVE, ORGANISER). No new tables or columns; use existing focus_group_members table and role column. Align with the SQL comments in the shared schema file.

Self-Service Join

  • Provide an endpoint or policy so that an authenticated user can create a focus_group_members row for themselves.
  • Example: POST /api/focus-group-members with body containing focusGroupId (and tenantId if not inferred). Backend sets user_profile_id from the authenticated user (Clerk/session). Default role=MEMBER, status=PENDING or ACTIVE per product policy. Enforce unique (tenant_id, focus_group_id, user_profile_id); return 409 or friendly message if already a member.

Leave

  • Allow a user to remove their own membership. Example: DELETE /api/focus-group-members/{id} only when the membership row’s user_profile_id matches the current user’s profile. Return 403/404 if the user does not own the membership.

Role Values

  • Extend or document the role field to include: MEMBER, LEAD, ADMIN, EXECUTIVE, ORGANISER. EXECUTIVE and ORGANISER denote executive/organising committee members. Update enum, validation, or documentation as applicable.

Committee List

  • Support filtering focus group members by role so the frontend can request “committee” members. Options: (a) Add criteria filter e.g. role.in=EXECUTIVE,ORGANISER on existing GET /api/focus-group-members (e.g. focusGroupId.equals=<id> and role.in=EXECUTIVE,ORGANISER); or (b) expose a dedicated endpoint e.g. GET /api/focus-groups/{id}/organisers that returns members with role EXECUTIVE or ORGANISER.

Schema

No migration required. Use existing focus_group_members table and role column. The canonical schema file adds only COMMENTs on role and status; align backend validation/docs with those allowed values. Reference: code_html_template/SQLS/Current_Sqls/Latest_Schema_Post__Blob_Claude_12.sql.

Backward compatibility: Existing MEMBER, LEAD, ADMIN roles remain. Adding EXECUTIVE and ORGANISER is additive. Existing API clients that do not use self-service join or committee filter are unchanged.
