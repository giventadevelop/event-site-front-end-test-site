GiveButter Donation Flow - Database Design Changes
Schema Changes and New Tables

Summary of Changes

  • New Table: donation_transaction - Standalone donation records
  • Enum Update: Add GIVEBUTTER to payment_provider_type enum
  • Existing Tables: No changes required (reuse existing payment infrastructure)
  • Indexes: Add indexes for performance optimization

1. New Table: donation_transaction

1.1 Table Definition

CREATE TABLE donation_transaction ( id BIGINT DEFAULT nextval('public.sequence_generator'::regclass) NOT NULL, tenant_id VARCHAR(255) NOT NULL, event_id BIGINT REFERENCES event_details(id), payment_transaction_id BIGINT REFERENCES payment_transaction(id), transaction_reference VARCHAR(255) UNIQUE NOT NULL, givebutter_donation_id VARCHAR(255), -- GiveButter donation ID amount DECIMAL(10, 2) NOT NULL, email VARCHAR(255) NOT NULL, first_name VARCHAR(255), last_name VARCHAR(255), phone VARCHAR(50), prayer_intention TEXT, -- For Mass offerings is_recurring BOOLEAN DEFAULT FALSE, is_anonymous BOOLEAN DEFAULT FALSE, status VARCHAR(50) NOT NULL, -- VARCHAR, not enum (matches existing pattern) -- PENDING, COMPLETED, FAILED, CANCELLED (VARCHAR, not enum) qr_code_url TEXT, qr_code_image_url TEXT, email_sent BOOLEAN DEFAULT FALSE, metadata TEXT, -- JSON string for additional data created_at TIMESTAMP WITH TIME ZONE DEFAULT NOW(), updated_at TIMESTAMP WITH TIME ZONE DEFAULT NOW(), CONSTRAINT donation_transaction_pkey PRIMARY KEY (id), CONSTRAINT fk_donation_event FOREIGN KEY (event_id) REFERENCES event_details(id), CONSTRAINT fk_donation_payment FOREIGN KEY (payment_transaction_id) REFERENCES payment_transaction(id) );

1.2 Column Descriptions

Column Type Description
id BIGINT Primary key, uses shared sequence_generator (compatible with existing schema pattern)
tenant_id VARCHAR(255) Multi-tenant identifier
event_id BIGINT Foreign key to event_details (nullable for standalone donations)
payment_transaction_id BIGINT Foreign key to payment_transaction (links to payment orchestration)
transaction_reference VARCHAR(255) Unique transaction reference (e.g., "GB-12345")
givebutter_donation_id VARCHAR(255) GiveButter donation ID from API response
amount DECIMAL(10, 2) Donation amount
email VARCHAR(255) Donor email address
first_name VARCHAR(255) Donor first name
last_name VARCHAR(255) Donor last name
phone VARCHAR(50) Donor phone number (optional)
prayer_intention TEXT Prayer intention for Mass offerings (optional)
is_recurring BOOLEAN Whether donation is recurring
is_anonymous BOOLEAN Whether donation is anonymous
status VARCHAR(50) Donation status: PENDING, COMPLETED, FAILED, CANCELLED (VARCHAR, not enum type - matches existing pattern)
qr_code_url TEXT QR code URL for donation confirmation
qr_code_image_url TEXT QR code image URL
email_sent BOOLEAN Whether confirmation email has been sent
metadata TEXT JSON string for additional donation data
created_at TIMESTAMP WITH TIME ZONE Record creation timestamp
updated_at TIMESTAMP WITH TIME ZONE Record last update timestamp

1.3 Indexes

-- Performance indexes CREATE INDEX idx_donation_tenant_event ON donation_transaction(tenant_id, event_id); CREATE INDEX idx_donation_reference ON donation_transaction(transaction_reference); CREATE INDEX idx_donation_givebutter_id ON donation_transaction(givebutter_donation_id); CREATE INDEX idx_donation_status ON donation_transaction(status); CREATE INDEX idx_donation_created ON donation_transaction(created_at); CREATE INDEX idx_donation_email_sent ON donation_transaction(email_sent) WHERE email_sent = FALSE; CREATE INDEX idx_donation_qr_code ON donation_transaction(qr_code_url) WHERE qr_code_url IS NULL;

2. Enum Updates

2.1 Add GIVEBUTTER to payment_provider_type

-- Add GIVEBUTTER to payment_provider_type enum (if not exists) DO $$ BEGIN IF NOT EXISTS ( SELECT 1 FROM pg_enum WHERE enumlabel = 'GIVEBUTTER' AND enumtypid = (SELECT oid FROM pg_type WHERE typname = 'payment_provider_type') ) THEN ALTER TYPE payment_provider_type ADD VALUE 'GIVEBUTTER'; END IF; END $$; -- Verify enum values SELECT unnest(enum_range(NULL::payment_provider_type)) AS provider_type; -- Expected: STRIPE, PAYPAL, ACP, AP2, SQUARE, CRYPTO, GIVEBUTTER

2.2 Verify payment_type Enum

-- Verify payment_type enum includes required values SELECT enumlabel AS payment_type FROM pg_enum WHERE enumtypid = (SELECT oid FROM pg_type WHERE typname = 'payment_type') ORDER BY enumsortorder; -- Required values: -- - TICKET_SALE (for ticketed fundraiser events) -- - DONATION (for donation-based events) -- - DONATION_ZERO_FEE (explicit zero-fee donations) -- - OFFERING (for Mass offerings)

4. Optional: Donation Statistics Table

4.1 Statistics Table (Optional)

Create an optional table for pre-aggregated donation statistics (for admin dashboard performance):

CREATE TABLE donation_statistics ( id BIGINT DEFAULT nextval('public.sequence_generator'::regclass) NOT NULL, tenant_id VARCHAR(255) NOT NULL, event_id BIGINT REFERENCES event_details(id), total_donations INTEGER DEFAULT 0, total_amount DECIMAL(10, 2) DEFAULT 0, average_amount DECIMAL(10, 2) DEFAULT 0, date_range_start DATE, date_range_end DATE, last_updated TIMESTAMP WITH TIME ZONE DEFAULT NOW(), CONSTRAINT donation_statistics_pkey PRIMARY KEY (id), UNIQUE(tenant_id, event_id, date_range_start, date_range_end) ); CREATE INDEX idx_donation_stats_tenant_event ON donation_statistics(tenant_id, event_id); CREATE INDEX idx_donation_stats_date_range ON donation_statistics(date_range_start, date_range_end);
Note: This table is optional. Statistics can also be calculated on-the-fly from donation_transaction table. Use this table only if performance optimization is needed for admin dashboard.

6. Data Relationships

6.1 Entity Relationship Diagram

event_details (1) ──< (N) donation_transaction payment_transaction (1) ──< (N) donation_transaction event_details (1) ──< (N) event_ticket_transaction (for ticketed fundraisers) payment_transaction (1) ──< (N) event_ticket_transaction

6.2 Relationship Notes

  • donation_transaction.event_id: Links to event (nullable for standalone donations)
  • donation_transaction.payment_transaction_id: Links to payment orchestration record
  • For ticketed fundraisers: Use event_ticket_transaction instead of donation_transaction

7. Query Examples

7.1 Get Donations for Event

SELECT id, transaction_reference, amount, email, first_name, last_name, status, created_at FROM donation_transaction WHERE event_id = 4202 AND tenant_id = 'tenant_demo_002' AND status = 'COMPLETED' ORDER BY created_at DESC;

7.2 Get Donation Statistics

SELECT event_id, COUNT(*) AS total_donations, SUM(amount) AS total_amount, AVG(amount) AS average_amount, MIN(amount) AS min_amount, MAX(amount) AS max_amount FROM donation_transaction WHERE event_id = 4202 AND tenant_id = 'tenant_demo_002' AND status = 'COMPLETED' GROUP BY event_id;

7.3 Get Pending Email Donations

SELECT id, transaction_reference, email, amount, event_id FROM donation_transaction WHERE status = 'COMPLETED' AND email IS NOT NULL AND email_sent = FALSE ORDER BY created_at ASC LIMIT 100;

10. Architecture Note: Frontend Polling Pattern

No Webhook-Based Transaction Creation

The database schema supports donation transactions that are created via API endpoints (called from frontend), not via webhook handlers. The frontend polls donation status and calls backend APIs to create transactions, generate QR codes, and send emails.

This matches the existing checkout flow pattern where the frontend handles webhook-like behavior through polling.
