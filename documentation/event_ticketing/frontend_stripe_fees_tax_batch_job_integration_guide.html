Frontend Integration Guide: Stripe Fees and Tax Batch Job API

Document Date: January 11, 2026

Purpose: Guide for frontend developers to integrate the Stripe Fees and Tax Update Batch Job API.

Backend API Base URL: Configured in your frontend environment (typically http://localhost:8080/api for development)

Executive Summary

This document provides a comprehensive guide for frontend developers to integrate the Stripe Fees and Tax Update Batch Job API. The batch job service retrieves missing Stripe fee and tax data from Stripe's API and updates transaction records in the database. The API returns immediately with a job ID and status (HTTP 202 Accepted), while processing happens asynchronously in the background.

1. Overview

The Stripe Fees and Tax Update Batch Job API allows you to:

  • Trigger batch processing of Stripe fee and tax data retrieval
  • Update transaction records in the database
  • Process transactions for specific tenants or all tenants
  • Filter transactions by date range
  • Force update already populated transactions (if needed)

Key Characteristics

  • Asynchronous Processing: The API returns immediately (HTTP 202 Accepted) while processing happens in the background
  • Non-blocking: Don't wait for job completion - the job runs independently
  • Idempotent: Safe to call multiple times with the same parameters
  • Multi-tenant Support: Can process for specific tenant or all tenants

2. API Endpoint Details

POST /api/cron/stripe-fees-tax-update

Description: Triggers the Stripe fees and tax update batch job.

Response: Returns immediately with job ID and status (HTTP 202 Accepted)

Authentication: Requires JWT token in Authorization header

Processing: Runs asynchronously in the background

2.1 Request Headers

Header Value Required Description
Content-Type application/json Yes Request body content type
Authorization Bearer {jwt_token} Yes JWT authentication token
X-Tenant-Id {tenant_id} No Optional tenant ID override (if not provided in request body)

2.2 Request Body

All fields in the request body are optional. An empty JSON object {} will process all tenants with no date filters.

Field Type Description Required Example
tenantId String Filter by specific tenant ID. If not provided, processes all tenants sequentially. No "tenant_demo_002"
startDate String (ISO 8601) Process transactions created on or after this date. Format: yyyy-MM-dd'T'HH:mm:ss.SSSXXX No "2025-01-01T00:00:00.000Z"
endDate String (ISO 8601) Process transactions created on or before this date. Format: yyyy-MM-dd'T'HH:mm:ss.SSSXXX No "2025-01-31T23:59:59.999Z"
forceUpdate Boolean If true, update even if stripe_fee_amount is already populated. Default: false No false

2.3 Response Format

HTTP Status: 202 Accepted (when job starts successfully)

{ "jobId": "stripe-fees-update-20250111-140530-tenant_demo_002", "status": "STARTED", "tenantId": "tenant_demo_002", "startDate": "2025-01-01T00:00:00.000Z", "endDate": "2025-01-31T23:59:59.999Z", "forceUpdate": false, "estimatedRecords": 150, "estimatedCompletionTime": "2025-01-11T14:15:30.000Z", "message": "Batch job started successfully" }
Field Type Description
jobId String Unique job identifier for tracking (can be used for future job status queries)
status String Job status: STARTED, IN_PROGRESS, COMPLETED, FAILED
tenantId String | null Tenant ID being processed (null if processing all tenants)
startDate String | null Start date filter (null if not provided)
endDate String | null End date filter (null if not provided)
forceUpdate Boolean Force update flag
estimatedRecords Long | null Estimated number of transactions to process
estimatedCompletionTime String | null Estimated completion time (ISO 8601 format)
message String Human-readable status message

2.4 Error Responses

HTTP Status Description Response Body Frontend Handling
400 Bad Request Invalid request parameters (e.g., startDate after endDate) {"status": "FAILED", "message": "startDate must be before or equal to endDate"} Show validation error to user, allow them to fix the date range
401 Unauthorized Missing or invalid authentication token {"error": "Unauthorized"} Redirect to login or refresh token
500 Internal Server Error Job failed to start or service error {"status": "FAILED", "message": "Failed to trigger job: [error details]"} Show error message, suggest retry or contact support

// services/stripeFeesTaxBatchJobService.ts import axios from 'axios'; const API_BASE_URL = process.env.NEXT_PUBLIC_API_BASE_URL || 'http://localhost:8080/api'; export const triggerStripeFeesTaxUpdate = async ( request: StripeFeesTaxUpdateRequest, authToken: string, tenantId?: string ): Promise<StripeFeesTaxUpdateResponse> => { try { const headers: Record<string, string> = { 'Content-Type': 'application/json', 'Authorization': `Bearer ${authToken}`, }; // Add tenant ID to header if provided if (tenantId) { headers['X-Tenant-Id'] = tenantId; } const response = await axios.post( `${API_BASE_URL}/cron/stripe-fees-tax-update`, request, { headers } ); return response.data; } catch (error: any) { if (error.response) { // Server responded with error status throw new Error(error.response.data?.message || 'Failed to trigger batch job'); } else if (error.request) { // Request made but no response received throw new Error('Network error: Unable to connect to server'); } else { // Something else happened throw new Error(error.message || 'An unexpected error occurred'); } } };

5. UI/UX Considerations

5.1 User Experience Recommendations

  • Loading States: Show a loading spinner or disabled button while the API call is in progress
  • Success Feedback: Display a success message with the job ID and estimated records count
  • Error Handling: Show clear, user-friendly error messages (e.g., "Start date must be before end date")
  • Confirmation: Consider showing a confirmation dialog before triggering the job, especially for large operations
  • Job ID Display: Display the job ID so users can reference it if needed (e.g., for support tickets)
  • Estimated Time: If available, show the estimated completion time to set user expectations

5.2 Form Validation

  • Validate that startDate is before or equal to endDate if both are provided
  • Validate date format (ISO 8601: yyyy-MM-dd'T'HH:mm:ss.SSSXXX)
  • Show validation errors inline before submitting the form

6. Error Handling Best Practices

6.1 Error Response Handling

function handleBatchJobError(error: any): string { if (error.response) { // Server responded with error status const status = error.response.status; const message = error.response.data?.message || 'Unknown error'; switch (status) { case 400: return `Validation Error: ${message}`; case 401: return 'Authentication required. Please log in again.'; case 403: return 'You do not have permission to perform this action.'; case 500: return `Server Error: ${message}. Please try again later or contact support.`; default: return `Error (${status}): ${message}`; } } else if (error.request) { // Request made but no response received return 'Network Error: Unable to connect to server. Please check your connection.'; } else { // Something else happened return `Error: ${error.message || 'An unexpected error occurred'}`; } }

6.2 Retry Logic

Note: The batch job API is idempotent, so it's safe to retry. However, avoid retrying too frequently as it may create multiple jobs for the same request.
