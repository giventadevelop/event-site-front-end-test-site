No schema migration is required for v1. Tenant IDs continue to live in tenant_organization.tenant_id (varchar(255) NOT NULL UNIQUE). Sequence is derived at runtime from existing values.

2. Sequence discovery (application SQL)

-- PostgreSQL: global max numeric suffix
SELECT MAX(
  (regexp_match(tenant_id, '_([0-9]+)$'))[1]::bigint
) AS max_seq
FROM tenant_organization
WHERE tenant_id ~ '_([0-9]+)$';

Next sequence = COALESCE(max_seq, 0) + 1, or 1 when no rows match.

3. Legacy tenant IDs

IDs without a trailing _digits pattern (e.g. tenant_demo_001) still work. They do not participate in max-sequence calculation unless they match *_NNNNN. New IDs use 5+ digit suffixes. Mixed conventions can coexist.

4. Optional future: sequence table

Only if backend needs atomic allocation under heavy concurrency:

CREATE TABLE tenant_id_sequence (
  id              SMALLINT PRIMARY KEY DEFAULT 1 CHECK (id = 1),
  last_value      BIGINT NOT NULL DEFAULT 0,
  updated_at      TIMESTAMPTZ NOT NULL DEFAULT NOW()

-- On each allocate: UPDATE ... SET last_value = last_value + 1 RETURNING last_value;

This is not required for initial rollout.
