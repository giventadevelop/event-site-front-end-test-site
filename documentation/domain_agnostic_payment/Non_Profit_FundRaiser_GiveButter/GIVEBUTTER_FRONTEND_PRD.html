1. Overview

This PRD details the frontend implementation of Givebutter integration for zero-fee fundraising and Mass offerings. The implementation follows the existing Stripe payment component patterns, ensuring consistent UX and maintainability.

✅ Architecture Pattern: Domain-agnostic frontend. All payment processing happens in backend. Frontend only calls proxy endpoints, never directly calls Givebutter API. No API keys exposed to client.
⚠️ Critical Rule: Client components must NEVER make direct API calls. All API calls must be made from server components or server actions. See shared-cursor-rules/nextjs_api_routes.mdc for details.

3. API Integration

3.1 Payment Initialization Flow

// Frontend: Mass Offering Widget
// src/components/MassOfferingWidget.tsx
'use client';

import { initializePayment } from '@/lib/payments/paymentApi';
import type { PaymentInitializeRequest, PaymentUseCase } from '@/types';

export function MassOfferingWidget({ eventId, massDate }) {
    const handleSubmit = async () => {
        // Build payment initialization request
        const request: PaymentInitializeRequest = {
            paymentUseCase: PaymentUseCase.OFFERING,
            amount: offeringAmount,
            currency: 'USD',
            items: [{
                itemType: 'OFFERING',
                description: 'Mass Offering',
                quantity: 1,
                unitPrice: offeringAmount
            }],
            customerEmail: email,
            customerName: name,
            eventId: eventId,
            metadata: {
                prayerIntention: prayerIntention,
                isAnonymous: isAnonymous,
                isRecurring: isRecurring,
                recurringFrequency: recurringFrequency, // 'weekly' or 'monthly'
                massDate: massDate,
                offeringType: 'MASS_COLLECTION'
            }
        };

        // Call backend proxy endpoint (NOT Givebutter directly)
        const session = await initializePayment(request);

        // Backend returns checkout URL - redirect user
        if (session.sessionUrl) {
            window.location.href = session.sessionUrl;
        }
    };
}

6. Page Integration

6.1 Event Page Integration

Location: src/app/events/[id]/page.tsx

Pattern: Server component fetches event data, passes to client component

// Server Component
export default async function EventPage({ params }: { params: { id: string } }) {
    const event = await fetchEventServer(params.id);

    // Parse metadata JSON string (backend stores as TEXT, not JSONB)
    // Backend should parse and return as object, or frontend parses if string
    const metadata = typeof event.metadata === 'string'
        ? JSON.parse(event.metadata || '{}')
        : event.metadata || {};

    // Check if event is configured for zero-fee fundraising
    const isZeroFeeFundraiser = metadata?.donationConfig?.useZeroFeeProvider === true
        && metadata?.donationConfig?.zeroFeeProvider === 'GIVEBUTTER';

    // Check if event is marked as fundraiser/charity event
    const isFundraiserEvent = metadata?.isFundraiserEvent === true
        || metadata?.isCharityEvent === true;

    return (
        <>
            <EventDetails event={event} />
            {/* Show donation widget for donation-based fundraisers */}
            {isZeroFeeFundraiser && event.admissionType === 'DONATION_BASED' && (
                <ZeroFeeFundraiserWidget eventId={event.id} />
            )}
            {/* For ticketed fundraiser events, UniversalPaymentCheckout will route to Givebutter */}
            {/* The backend PaymentOrchestrationService handles the routing based on event metadata */}
        </>
    );
}
✅ Ticketed Fundraiser Events: For events marked as fundraisers (isFundraiserEvent: true or isCharityEvent: true) with donationConfig.useZeroFeeProvider: true and zeroFeeProvider: "GIVEBUTTER", the existing UniversalPaymentCheckout component will automatically route ticket sales to Givebutter. No frontend changes needed - the backend handles routing based on event metadata.

12. Ticketed Fundraiser Events

12.1 Event Configuration

Events can be marked as fundraiser/charity events in the admin panel or via API. The metadata column is stored as TEXT (not JSONB) and contains JSON as a string.

// Event metadata structure for fundraiser events (stored as JSON string)
{
    "isFundraiserEvent": true,  // or "isCharityEvent": true
    "donationConfig": {
        "useZeroFeeProvider": true,
        "zeroFeeProvider": "GIVEBUTTER",
        "givebutterCampaignId": "campaign_fundraiser_123"
    }
}

// Backend parses JSON string using Jackson ObjectMapper
// Frontend receives parsed object or parses string if needed:
const metadata = typeof event.metadata === 'string'
    ? JSON.parse(event.metadata || '{}')
    : event.metadata || {};

12.2 Payment Flow for Ticketed Fundraisers

  1. User selects tickets on event page
  2. User clicks "Buy Tickets" button
  3. UniversalPaymentCheckout component calls POST /api/proxy/payments/initialize with paymentUseCase: TICKET_SALE
  4. Backend PaymentOrchestrationService checks event metadata
  5. If event is marked as fundraiser with Givebutter configured, backend routes to GivebutterPaymentAdapter
  6. Backend returns Givebutter checkout URL
  7. Frontend redirects user to Givebutter checkout page
  8. User completes payment on Givebutter
  9. Givebutter sends webhook to backend
  10. Backend updates payment transaction status
✅ No Frontend Changes Required: The existing UniversalPaymentCheckout component works for ticketed fundraiser events. The backend automatically routes to Givebutter based on event metadata. Frontend only needs to ensure event metadata is properly set when creating/editing fundraiser events.

12.3 Admin UI Requirements

Admin interface should allow marking events as fundraisers:

  • Checkbox: "This is a fundraiser/charity event"
  • Checkbox: "Use zero-fee payment provider"
  • Dropdown: "Zero-fee provider" (Givebutter, Zeffy, etc.)
  • Input: "Givebutter Campaign ID" (if Givebutter selected)
