2. Proposed Database Design

2.1 New Table: focus_group_member_event_registration

Purpose:

Track which focus group members have registered for events that belong to their focus group(s). This creates a direct link between focus group membership and event registrations.

Use Cases:

  • Track focus group member participation in focus group events
  • Generate reports on member engagement
  • Send targeted communications to members who haven't registered
  • Analyze event registration patterns by focus group
  • Automatically create records when members register for focus group events

Table Structure:

CREATE TABLE public.focus_group_member_event_registration (
    id bigint DEFAULT nextval('public.sequence_generator'::regclass) NOT NULL,
    tenant_id character varying(255) NOT NULL,

    -- Foreign Keys
    focus_group_id bigint NOT NULL,
    focus_group_member_id bigint NOT NULL,
    user_profile_id bigint NOT NULL,
    event_id bigint NOT NULL,
    event_ticket_transaction_id bigint,

    -- Registration Details
    registration_date timestamp without time zone DEFAULT now() NOT NULL,
    registration_status character varying(50) NOT NULL DEFAULT 'REGISTERED',
    registration_source character varying(50) DEFAULT 'MANUAL',

    -- Optional Metadata
    notes text,
    attended boolean DEFAULT false,
    attendance_date timestamp without time zone,

    -- Audit Fields
    created_at timestamp without time zone DEFAULT now() NOT NULL,
    updated_at timestamp without time zone DEFAULT now() NOT NULL,
    created_by_id bigint,

    -- Constraints
    CONSTRAINT focus_group_member_event_registration_pkey PRIMARY KEY (id),
    CONSTRAINT ux_focus_group_member_event_reg__tenant_group_member_event
        UNIQUE (tenant_id, focus_group_id, focus_group_member_id, event_id),

    -- Foreign Key Constraints
    CONSTRAINT fk_fg_member_event_reg__focus_group
        FOREIGN KEY (focus_group_id)
        REFERENCES public.focus_group(id) ON DELETE CASCADE,

    CONSTRAINT fk_fg_member_event_reg__focus_group_member
        FOREIGN KEY (focus_group_member_id)
        REFERENCES public.focus_group_members(id) ON DELETE CASCADE,

    CONSTRAINT fk_fg_member_event_reg__user_profile
        FOREIGN KEY (user_profile_id)
        REFERENCES public.user_profile(id) ON DELETE CASCADE,

    CONSTRAINT fk_fg_member_event_reg__event
        FOREIGN KEY (event_id)
        REFERENCES public.event_details(id) ON DELETE CASCADE,

    CONSTRAINT fk_fg_member_event_reg__transaction
        FOREIGN KEY (event_ticket_transaction_id)
        REFERENCES public.event_ticket_transaction(id) ON DELETE SET NULL,

    CONSTRAINT fk_fg_member_event_reg__created_by
        FOREIGN KEY (created_by_id)
        REFERENCES public.user_profile(id) ON DELETE SET NULL,

    -- Check Constraints
    CONSTRAINT check_registration_status
        CHECK (registration_status IN ('REGISTERED', 'CANCELLED', 'WAITLISTED', 'ATTENDED', 'NO_SHOW')),

    CONSTRAINT check_registration_source
        CHECK (registration_source IN ('MANUAL', 'AUTO_FROM_TRANSACTION', 'IMPORT', 'API'))
);

3. Constraints and Relationships

3.1 Unique Constraint

CONSTRAINT ux_focus_group_member_event_reg__tenant_group_member_event
    UNIQUE (tenant_id, focus_group_id, focus_group_member_id, event_id)

Purpose: Prevents duplicate registrations for the same focus group member, event, and focus group combination.

Rationale: A member can register for the same event through multiple focus groups (if event belongs to multiple groups), but only once per focus group.

3.2 Foreign Key Relationships

Cascade Delete Rules:

  • focus_group_id: ON DELETE CASCADE - If focus group is deleted, remove all registrations
  • focus_group_member_id: ON DELETE CASCADE - If membership is removed, remove all registrations
  • user_profile_id: ON DELETE CASCADE - If user is deleted, remove all registrations
  • event_id: ON DELETE CASCADE - If event is deleted, remove all registrations
  • event_ticket_transaction_id: ON DELETE SET NULL - If transaction is deleted, keep registration but clear link
  • created_by_id: ON DELETE SET NULL - If creator is deleted, keep registration but clear creator

3.3 Check Constraints

Registration Status:

6. Use Cases and Query Examples

6.1 Common Queries

Query 1: Get all events a focus group member has registered for

SELECT
    e.id,
    e.title,
    e.start_date,
    e.start_time,
    fgmer.registration_date,
    fgmer.registration_status,
    fgmer.attended
FROM focus_group_member_event_registration fgmer
INNER JOIN event_details e ON fgmer.event_id = e.id
INNER JOIN focus_group_members fgm ON fgmer.focus_group_member_id = fgm.id
WHERE fgm.user_profile_id = :userId
  AND fgm.focus_group_id = :focusGroupId
  AND fgmer.tenant_id = :tenantId
ORDER BY e.start_date DESC;

Query 2: Get all members who registered for a specific event

SELECT
    up.id,
    up.first_name,
    up.last_name,
    up.email,
    fg.name AS focus_group_name,
    fgmer.registration_date,
    fgmer.registration_status,
    fgmer.attended
FROM focus_group_member_event_registration fgmer
INNER JOIN user_profile up ON fgmer.user_profile_id = up.id
INNER JOIN focus_group fg ON fgmer.focus_group_id = fg.id
WHERE fgmer.event_id = :eventId
  AND fgmer.tenant_id = :tenantId
  AND fgmer.registration_status = 'REGISTERED'
ORDER BY fgmer.registration_date DESC;

Query 3: Get registration statistics for a focus group

SELECT
    COUNT(*) AS total_registrations,
    COUNT(DISTINCT fgmer.user_profile_id) AS unique_members,
    COUNT(DISTINCT fgmer.event_id) AS unique_events,
    SUM(CASE WHEN fgmer.attended = true THEN 1 ELSE 0 END) AS total_attended
FROM focus_group_member_event_registration fgmer
WHERE fgmer.focus_group_id = :focusGroupId
  AND fgmer.tenant_id = :tenantId
  AND fgmer.registration_status = 'REGISTERED';

Query 4: Auto-create registration when transaction is created

-- This would be done via trigger or application logic
INSERT INTO focus_group_member_event_registration (
    tenant_id,
    focus_group_id,
    focus_group_member_id,
    user_profile_id,
    event_id,
    event_ticket_transaction_id,
    registration_date,
    registration_status,
    registration_source
)
SELECT
    ett.tenant_id,
    efg.focus_group_id,
    fgm.id AS focus_group_member_id,
    ett.user_id AS user_profile_id,
    ett.event_id,
    ett.id AS event_ticket_transaction_id,
    ett.purchase_date,
    'REGISTERED',
    'AUTO_FROM_TRANSACTION'
FROM event_ticket_transaction ett
INNER JOIN event_focus_groups efg ON ett.event_id = efg.event_id
INNER JOIN focus_group_members fgm
    ON efg.focus_group_id = fgm.focus_group_id
    AND ett.user_id = fgm.user_profile_id
WHERE ett.id = :transactionId
  AND ett.status = 'CONFIRMED'
  AND fgm.status = 'ACTIVE'
ON CONFLICT (tenant_id, focus_group_id, focus_group_member_id, event_id)
DO NOTHING;

7. Implementation Considerations

📝 Important Notes

  • Data Integrity: The focus_group_member_id foreign key ensures that only actual focus group members can have registrations
  • Automatic Creation: Consider creating triggers or application logic to auto-create registrations when:
    • User purchases ticket for focus group event
    • User is added to focus group and has existing registrations for focus group events
  • Validation: Before inserting, verify:
    • Event belongs to the focus group (via event_focus_groups)
    • User is a member of the focus group (via focus_group_members)
    • Focus group member status is ACTIVE
  • Performance: Indexes are critical for query performance, especially composite indexes
  • Tenant Isolation: Always filter by tenant_id in queries

⚠️ Migration Considerations

  • Existing Data: If there are existing event registrations, consider creating a migration script to populate this table
  • Backward Compatibility: This table is additive - existing functionality will continue to work
  • Gradual Rollout: Can be implemented incrementally without breaking existing features
  • Data Validation: Run validation queries to ensure data integrity after migration
